# 05-接口与联调规则

> 文档类型：规则正文
> 状态：draft
> 更新时间：2026-04-25
> 说明：本文件用于统一正式接口、provider 口径、联调入口和接口文档集中规则；当前项目级后端契约入口已迁入 `规则体系/后端契约/`，本文件负责固定它与前端实现契约之间的关系。

## 1. 目标

本文件解决四个问题：

1. 什么才算“真的给后端留出了 API”。
2. 正式接口文档现在以什么为准，后续集中到哪里。
3. `mock / current-http / backend-http` provider 分别承担什么职责，什么不能拿来当联调完成证据。
4. 后端、前端、规则文档之间如何保持一个长期可维护、可追溯的统一入口。

## 2. 总原则

接口与联调规则的首要目标固定为“后端和前端都能低成本理解、接入和验证”，不是“前端内部已经留了抽象层”。

因此必须满足：

1. 正式接口文档有统一真值源。
2. 后端阅读入口明确，不需要在多个目录里自己拼信息。
3. 前端 provider 选择与正式 contract 分层，不互相替代。
4. 契约、字段字典、联调说明和运行时规则不再长期混写。
5. 任何“已留出 API”的结论，都必须能被仓内文档和联调入口证明。
6. 请求链和 UI 展示链必须分层，不能靠组件直接消费请求语义来维持接口接入。

## 4. “已留出 API” 的判定标准

只有同时满足下面 6 项，才允许宣称“已经给后端留出了 API”：

1. 正式接口路径、方法、请求参数、响应骨架已经写入正式契约文档。
2. 字段归属已经明确：哪些是正式后端字段，哪些仍是前端上下文，哪些仍是 UI preset。
3. 前端 `http` adapter 已存在且路径、参数、DTO 对齐正式 contract。
4. 仓内存在明确的联调入口说明，后端知道该连什么、怎么验。
5. 仓内存在可切换到 `http` 的运行方式，且不是只能停留在 `mock`。
6. 至少有一条正式验证链可以证明“不是只有文档和壳”。

只满足以下情况时，不允许宣称已经留出 API：

1. 只有契约文档，没有可用的联调入口。
2. 只有前端 adapter，没有明确的后端阅读入口。
3. 默认 provider 仍是 `mock`，且仓内没有 `http` 联调说明。
4. 只有前端内部台账或计划说“后端以后按这个接”，没有正式 contract 回写。

1. 统一响应包：

### 6.2 当前仍属于前端本地上下文

### 6.3 当前仍是 UI preset

1. `owner`
2. `contract`
3. `chain`
4. `tokenStandard`
5. `traits`
6. `provenance`

规则：

1. 这些字段只有单独立项后，才允许进入正式 contract。
2. 不允许把前端当前占位字段误写成“后端已承诺字段”。

## 7. Provider 规则

### 7.1 Provider 分工

当前 `P11.12` 过渡期 provider 固定分三类：

1. `mock`
   - 用于前端开发、页面联调、结构自测
2. `current-http`
   - 用于当前 `/api/content/*` 抽象接口联调与现行内容域契约验证
3. `backend-http`
   - 用于项目级后端契约 wrapper 联调与新版后端接驳验证

### 7.2 Provider 边界

`mock` 可以证明：

1. 前端页面可运行
2. adapter 链路可消费稳定模型
3. 页面状态和交互链可验证

`mock` 不能证明：

1. 后端接口真实存在
2. 后端接口已可直接接入
3. 联调入口已经完整
4. 字段 contract 已被后端兑现

`current-http` 模式要承担的职责是：

1. 直连正式后端 base URL
2. 走当前 `/api/content/*` 抽象接口路径
3. 验证当前前端内容域实现契约是否能真实联调

`backend-http` 模式要承担的职责是：

1. 按项目级后端契约 wrapper 接入新版后端
2. 验证 `operation-usage-guide.md`、`field-usage-index.md` 和 adapter 映射是否成立
3. 给 `P11.12` 分线迁移提供真实后端 readiness 证据

### 7.3 Provider 声明规则

1. 任何文档声称“接口已留出”时，必须同时注明当前是否已有 `current-http` 或 `backend-http` 联调入口。
2. 不允许再用“已有 `mock` + 已有 adapter”替代“后端可直接接入”。
3. 运行时 provider 规则属于现行规则，但不属于后端 contract 本身。
4. 不允许把 `current-http` 的通过结果写成新版后端 wrapper 已接通；新版后端 readiness 必须由 `backend-http` 或明确的后端确认支撑。

## 7.4 请求链与 UI 链解耦规则

当前接口消费链固定执行以下解耦口径：

1. 页面和组件不得直接依赖 `http` implementation、`mock` implementation 或 provider 选择逻辑。
2. `contract / port / implementation` 只承接正式接口与 transport 边界，不直接承接页面展示字段、动画字段和交互展示态。
3. 页面专属 runtime 可以组装请求 DTO，但 DTO 必须在 runtime/service/adapters 边界内消化，不得直接透传为 UI props。
4. 若页面需要稳定展示模型，必须先经过 adapter 或等价规范化层，再进入页面展示链。
5. `content.service.ts` 及其上位消费链是当前允许感知 provider seam 的最外层；组件层和纯展示层不得感知 provider。
6. 若仓内保留 `src/adapters/` 这类接入层目录，它就必须承接真实的“请求模型转前端稳定模型”职责；空壳目录不算已完成的解耦层。

## 8. 契约入口与 handoff 规范

### 8.1 当前前端实现契约职责

[content-api.contract.ts](/H:/工作/天异/uniapp+vue新架构/src/contracts/content-api.contract.ts) 必须统一承载当前前端内容域已经实现的契约：

1. 正式接口、路径、方法、请求参数与响应 envelope
2. DTO、枚举、字段归属与约束
3. 正式资源类型、场景范围与分页口径
4. 当前明确不属于后端字段的内容

### 8.2 handoff 文档职责

`backend-handoff/` 只承载：

1. 后端最小阅读顺序
2. 联调环境与启动方式
3. 验证入口与交付清单

### 8.3 参考快照职责

`references/` 只承载：

1. 最小参考索引
2. 前端消费方式说明

`references/` 不再承载：

1. contract 代码副本
2. port 代码副本
3. http adapter 代码副本
4. service 代码副本

约束：

1. `backend-handoff/` 和 `references/` 都不得再次变成正式 API 正文承载层。
2. 已退场的根 API 正文、拆分式接口目录和冻结模板统一送入 `垃圾桶/`。

## 9. 接口文档维护规则

补充规则：

1. `规则体系/后端契约/` 是当前项目所使用、所对接的后端契约入口。
2. 根目录 `01-正式契约/` 自本轮起只保留历史快照属性，不再作为现行入口。
3. 任何二线、联调线或适配线，都必须优先回链 `规则体系/后端契约/README.md`。
4. 项目级后端契约签收、待确认标识和迁移入口变更，必须优先回写 `规则体系/后端契约/README.md` 及其下属契约正文。
5. 当前前端内容域实现变更，必须优先回写 [content-api.contract.ts](/H:/工作/天异/uniapp+vue新架构/src/contracts/content-api.contract.ts)。
6. handoff 阅读顺序、环境或验证方式变更时，必须同步回写 `backend-handoff/`。
7. 不允许只改代码或只改台账，不改项目级后端契约入口、当前前端实现契约或 handoff 入口。
8. 不允许把前端页面级运行时细节继续混进正式后端阅读正文。
9. 不允许重新创建根目录旧正文、拆分式接口目录或映射表充当当前 API 入口。

## 10. 联调最小交付清单

当对外宣称“后端现在可以直接接入”时，仓内至少要同时具备：

1. [content-api.contract.ts](/H:/工作/天异/uniapp+vue新架构/src/contracts/content-api.contract.ts)
2. `backend-handoff/` 中的联调环境与交付清单
3. `http` adapter
4. 可切换到 `http` 的运行方式
5. 至少一条正式验证链

如果缺少其中任一项，只能说：

1. “契约已预留”
2. 或“adapter 已预留”
3. 不能说“已可直接接入”

## 11. 禁止事项

1. 不允许继续把接口口径散落到台账、专项档案和历史目录。
2. 不允许用 `mock` 运行结果证明后端 readiness。
3. 不允许把前端本地上下文误写成正式后端字段。
4. 不允许先改接口实现、后补正式契约。
5. 不允许在没有统一阅读入口的情况下对后端宣称“API 已经留好”。

## 12. 与后续阶段的关系

本文件只定义“接口与联调应该怎么管理”。

后续分工固定为：

1. `G0` 负责把接口规则和目录规范写清楚。
2. `P1` 负责把“后端可直接接入”补成事实。
3. `P2` 负责逐条审核正式接口 readiness。
4. 后续如需再次发明新的正式 API 文档结构，必须先取得用户许可并先改规则，再实施物理迁移。
