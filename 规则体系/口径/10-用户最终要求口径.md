# 10-用户最终要求口径

> 文档类型：现行口径
> 状态：active
> 更新时间：2026-04-25
> 说明：本文件承载用户已经明确拍板的最终执行口径，用于约束后续前端改造、字段取舍和接口接线判断。它不是规则正文，不承载真实 API path、method、DTO 或联调说明。

## 1. 职责边界

本文件负责回答：

1. 用户已经最终裁定的前端行为应该是什么。
2. 后续前端改造应优先按什么口径落地。
3. 哪些问题不再要求后端补充，哪些问题只在未来扩展时再进入后端确认。

本文件不负责：

1. 不替代 [05-接口与联调规则.md](/H:/工作/天异/uniapp+vue新架构/规则体系/05-接口与联调规则.md)。
2. 不替代 [content-api.contract.ts](/H:/工作/天异/uniapp+vue新架构/src/contracts/content-api.contract.ts)。
3. 不替代 [06-项目后端契约现行使用口径.md](/H:/工作/天异/uniapp+vue新架构/规则体系/口径/06-项目后端契约现行使用口径.md)。
4. 不把当前用户裁决写成后端已经支持。
5. 不把前端自洽能力写成跨页全市场后端排序能力。

## 2. 优先级

当本文件与其他现行口径都涉及同一用户裁决事项时，按以下关系理解：

1. 用户已经明确拍板的前端最终要求，以本文件为当前执行口径。
2. 真实 API 新增、变更和联调入口，仍按接口与联调规则落地。
3. 后端是否已经支持，仍以项目级后端契约和真实联调证据为准。
4. 若用户后续把某事项改回待定，对应事项必须重新进入后端补充口径，不得沿用旧的前端自洽裁决。

## 3. 市场加载与排序待定口径

当前市场加载与排序口径已经调整为待定。

待定项固定为：

1. 市场列表首屏加载：待确认后端真实返回全量、当前页还是推荐窗口。
2. 市场列表刷新/筛选重载：待确认下拉刷新、系列筛选、类型切换或排序变化时是否重置分页，以及返回顺序如何保证。
3. 市场列表触底分页加载：待确认普通藏品和盲盒是否都支持稳定分页追加，以及末页、无新增、`total/hasMore` 语义。
4. 市场列表排序：待确认最新、价格、流通量等排序是否由后端统一处理，还是只允许前端对已加载集合做临时排序。

约束：

1. 不得继续把旧“前端自洽排序”写成当前最终口径。
2. 不得把旧前端普通藏品市场的一次性赋值行为推导成后端全量返回承诺。
3. 不得把盲盒分页追加行为推导成普通藏品市场也具备同等分页能力。
4. 不得在真实后端确认前宣称具备全市场跨页全局排序。
5. 页面模板仍不得直接消费后端 DTO、分页协议或排序参数。

## 4. 市场首页可用数据口径

旧前端市场首页卡片已确认可消费的数据包括：

| 旧字段            | 当前用途                         | 新前端稳定语义建议                 |
| ----------------- | -------------------------------- | ---------------------------------- |
| `id`              | 市场卡标识                       | `id` / `itemId`                    |
| `goods_type`      | 区分数字藏品和盲盒               | `resourceType` 或页面内部类型      |
| `product.id`      | 跳转到藏品市场详情列表           | `target.targetId` 或 `itemId`      |
| `product.name`    | 卡片标题                         | `title` / `name`                   |
| `product.listimg` | 卡片主图                         | `asset` / `imageUrl`               |
| `min_price`       | 数字藏品最低价                   | `priceInCent` / `price`            |
| `price`           | 盲盒价格或兜底价格               | `priceInCent` / `price`            |
| `flux`            | 旧首页流通量排序和旧卡片角标来源 | 仅作为待确认统计字段或候选排序指标 |
| `total_num`       | 限量/发行总量                    | `issueCount`                       |
| `is_out`          | 下架或售罄状态                   | `isOut` / `status`                 |
| `new_time`        | 最新上架排序                     | `listedAt`                         |

约束：

1. `total_num` 只能承接“总量/限量”，不得拿来假装“流通量”。
2. `flux` 如后续继续用于排序，必须先确认后端真实字段语义；未确认前不得与“总量/限量”混写。
3. 价格排序仍需等待后端确认字段单位、空值和分页一致性；旧值为 `--` 时只能按缺失值处理，不得直接参与字符串排序。

## 5. 当前待确认的候选排序项

当前市场首页排序项全部降级为候选，等待后端真实加载和排序口径确认：

1. 最新上架
   - 旧字段：`new_time`
   - 新字段：`listedAt`
   - 状态：待确认
2. 价格升序
   - 旧字段：数字藏品优先 `min_price`，盲盒可用 `price`
   - 新字段：`priceInCent` 或已归一的 `price`
   - 状态：待确认
3. 价格降序
   - 字段同价格升序
   - 状态：待确认
4. 流通量升序
   - 旧字段：`flux`
   - 新字段：仅当 adapter 明确保留 `flux` 或等价流通量统计时启用
   - 状态：待确认
5. 流通量降序
   - 字段同流通量升序
   - 状态：待确认

补充口径：

1. `issueCount` / `total_num` 可以用于展示“总量/限量”，当前不默认作为用户排序项。
2. `is_out` 当前不作为用户排序项；如需要，可作为所有排序后的置底规则，但必须单独确认。
3. 名称排序当前不进入默认要求。

## 6. 加载与分页边界

后端真实加载逻辑确认前，加载与分页边界固定为：

1. 首屏加载、刷新/筛选重载、触底分页加载均保持待定。
2. 前端不得宣称旧后端普通藏品市场是全量返回，也不得宣称当前新版后端已经稳定分页。
3. 如临时前端排序存在，只能限定在“当前已加载集合”内部，并必须标记为待确认 fallback。
4. 当前不得宣称已经具备“全市场跨页全局排序”。
5. 后端确认真实加载逻辑后，再决定排序是在后端、adapter、runtime 还是 service 层稳定落地。

## 7. 后续前端改造要求

后续前端改造按以下口径执行：

1. 市场加载与排序逻辑暂不进入最终稳定口径，先等待后端真实逻辑确认。
2. 页面模板只消费已经排序后的 view-model 列表。
3. 旧后端 `market.getMarketList` 接入时，不得擅自新增排序请求参数；是否需要新增参数等待后端确认。
4. `backend-http` 或 legacy adapter 如需临时排序，只能作为 fallback 限定在已加载集合内，不得写成最终排序能力。
5. 若当前模型没有可靠 `flux` 或等价字段，先隐藏或降级流通量排序，不得用 `issueCount` 冒充。
6. 卡片展示继续遵守 [08-P11.12前端改造口径.md](/H:/工作/天异/uniapp+vue新架构/规则体系/口径/08-P11.12前端改造口径.md)：市场卡左下角展示“总量/限量”，不展示成交量、交易量或流通量。

## 8. 与后端补充口径的关系

当前用户要求已经调整：

1. “市场首页排序”重新进入当前必须后端确认范围。
2. 首屏加载、刷新/筛选重载、触底分页加载和排序统一写入 [09-P11.12需后端补充口径.md](/H:/工作/天异/uniapp+vue新架构/规则体系/口径/09-P11.12需后端补充口径.md)。
3. 真实 API 新增、排序参数、分页字段或响应结构变化，仍必须按接口与联调规则落到 contract、port、implementation、service、handoff 和验证链。

## 9. 个人中心区块链地址最终口径

当前最终口径固定为：

1. 个人中心顶部地址展示真实接旧前端接口时，优先使用旧用户信息接口返回的 `ntf_url`。
2. `ntf_url` 来源于旧前端用户信息返回的 `userInfo`，旧个人中心、个人信息页、订单确认页和兑换页均按该字段展示“区块链地址”。
3. 新前端不得继续把 `userId`、`accountId`、手机号或邀请标识当成个人中心区块链地址兜底。
4. 藏品详情、寄售详情中的 `user_hash` / `user_hash_text` 属于藏品或合约地址展示链，不等同于个人中心顶部用户区块链地址。
5. 当 `ntf_url` 缺失时，前端可以显示“拥有藏品后自动生成”类占位文案，但不得把占位文案写回为真实地址，也不得作为复制内容或用户身份参数。
6. 主线后续接入个人中心区块链地址时，以本节作为现行口径入口。

后续前端改造要求：

1. `backend-http` 或 legacy adapter 适配旧用户信息接口时，个人中心 `profile_summary.address` 优先映射 `ntf_url`。
2. 如果未来新版后端改用 `wallet_address`、`chain_address` 或 `blockchain_address` 等标准字段，可以在 adapter 内作为兼容字段处理，但优先级不得高于旧前端已确认的 `ntf_url`。
3. `content` 用户态请求仍不得新增 `userId`、`accountId` 或 `walletAddress` 作为身份参数；用户身份继续来自登录态、token 或 session。
4. 主线接入时不得把当前新前端临时展示的用户 id、mock address 或短地址文案继续作为真实接线依据。

## 10. 个人中心资格证分类最终口径

当前最终口径固定为：

1. 个人中心“资格证”不得继续作为后端或内容域契约的一级资产分类。
2. 一级资产分类只保留“藏品/资产”和“盲盒”两类；在当前内容域契约中分别对应 `collections` 和 `blindBoxes`。
3. “资格证”归入 `collections` 下的二级分类或系列；旧前端证据中该层级使用 `series_id` / `series_name` 表达。
4. 后续主线或二线如果需要展示“资格证”，只能把它作为藏品下的二级筛选项、系列项或标签展示，不得把它接成独立后端大类。
5. 旧数据、缓存、路由或 mock 中仍出现 `certificates` 时，接入层应迁移或归一到 `categoryId=collections`，并通过 `subCategory` / `seriesName` / `seriesId` 表达系列。

后续前端改造要求：

1. [content-api.contract.ts](/H:/工作/天异/uniapp+vue新架构/src/contracts/content-api.contract.ts) 中 `ContentProfileCategoryId` 不再包含 `certificates`。
2. `profile_asset` 目标、列表、详情和个人中心资产块中，资格证类资产的 `categoryId` 必须是 `collections`。
3. `subCategory` 作为二级分类展示文本，优先承接旧前端 `series_name`。
4. `seriesId` 作为可选系列 ID，优先承接旧前端 `series_id`；没有稳定 ID 时可以只传 `subCategory`，但不得用 `categoryId=certificates` 兜底。
5. 二线视觉改造如保留“资格证”入口，应在“资产/藏品”分组下呈现为二级筛选，不得在模板或页面状态中重新发明资格证一级 tab。

## 11. 首页 banner 接口最终口径

当前最终口径固定为：

1. 首页 banner 接线走项目级后端契约中的 `banner.getBannerList`，也就是旧前端业务口径里说的 `getBanner` / `getBannerList`。
2. 旧前端首页里的 `<w-swiper>` 当前是注释状态，不再作为“旧首页正在展示 banner”的证据；但旧前端存在正式 wrapper、通用 `w-swiper` 调用和已注册页面消费证据，因此该接口按“旧系统真实存在、可作为新前端首页 banner 接线来源”处理。
3. 新前端首页继续消费稳定的 `home_banner` / `banner_carousel` view-model，不允许页面模板或二线 UI 直接消费旧 wrapper 响应结构。
4. 当前测试环境该接口可能返回空列表；空列表只能视为“当前无 banner 数据”，不得据此改回用首页藏品轮播、盲盒列表或 mock 图替代正式 banner 来源。
5. 接旧后端时必须兼容旧前端成功包口径；旧请求层以 `code=1` 为成功，不能因为当前内容域 envelope 默认以 `code=0` 为成功就把旧 `getBanner` 误判为不可用。
6. 当前默认请求语义沿用旧前端 `w-swiper` 的默认 banner 类型；如后续要扩展多个 banner 位、多个 `type` 或运营位枚举，必须另行进入后端确认和正式契约更新，不得由二线或页面层自行发明。

后续前端改造要求：

1. 主线接真实后端首页 banner 时，优先走 `banner.getBannerList` 对应的 backend-http adapter 映射，不新增第二套首页 banner wrapper。
2. adapter 负责把旧响应归一为 `banner_carousel.items`；页面、组件和二线视觉层只读 `homeBannerItems` 一类稳定展示模型。
3. 当后端返回空 banner 时，前端按空状态、占位或隐藏 banner 位处理；不得把空返回写成接口废弃，也不得把其他首页资源偷接成 banner。
4. 若后端后续确认 `getBanner` 已正式废弃，必须先回写项目级后端契约、用户最终要求口径和通讯，再改前端接线。
