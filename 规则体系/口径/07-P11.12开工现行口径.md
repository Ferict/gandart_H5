## 2. 下一步正式入口

当前下一步只允许进入：

1. `P11.12.1`：新版后端契约签收与冻结批。

`P11.12.1` 开始前必须确认：

1. `规则体系/后端契约/` 当前内容是本轮要使用的项目级后端契约原件或整理件。
2. `confirmed / exported-unused / unresolved / inferred` 四类状态已经能被读取并引用。
3. `src/contracts/content-api.contract.ts` 仍是当前前端已实现内容域契约入口，不能被直接作废。
4. 新版契约和当前实现契约在 `P11.12` 完成前允许并存。
5. 差异必须由 adapter、runtime、service 或后续正式计划吸收，不得让 UI 直接吃未签收字段。

## 3. 主线后续工作清单

主线 Codex 后续按以下顺序推进：

1. 读取 `规则体系/后端契约/README.md`、`api-inventory.md`、`unresolved.md`、`contracts/index.ts` 和相关 `contracts/*.contract.ts`。
2. 把后端契约项目按 `confirmed / exported-unused / unresolved / inferred` 分类成签收表。
3. 标出本轮能作为稳定依据的接口、字段、枚举和示例。
4. 标出本轮不能直接接 UI 的候选项、未签收项和推断项。
5. 对照 `src/contracts/content-api.contract.ts`、`src/ports/content.port.ts`、`src/implementations/content.mock.ts`、`src/implementations/content.http.ts`、`src/services/content/content.service.ts` 和 `src/adapters/content/**` 做新旧差异矩阵。
6. 设计 `contract -> port -> implementation -> adapter -> runtime/service -> view-model -> component props` 的目标落点。
7. 明确 mock、http、provider 的过渡方式。
8. 再按 `home/home`、`home/activity`、`home/profile`、`profile-asset-detail`、`updating` 分线迁移。

## 4. 主线当前禁止事项

当前主线禁止：

1. 禁止为了适配后端契约改视觉。
2. 禁止页面模板直接消费后端 DTO。
3. 禁止把 `exported-unused`、`unresolved`、`inferred` 或 Level C 字段写成稳定 UI 依赖。
4. 禁止在未完成签收批前直接替换 `src/contracts/content-api.contract.ts`。
5. 禁止在普通 adapter 迁移批里改 provider seam。
6. 禁止把二线 page-local 产物当作主线契约依据。
7. 禁止把视觉批、后端适配批、测试批混成一个不可回滚提交。

## 5. 当前需要从二线了解的信息

主线开工前需要二线明确以下信息；如果二线没有相关改动，也必须回传“无”：

1. 当前是否存在未提交或未回传的页面级 UI 改动。
2. 当前正在处理的页面、panel、section 或外来 UI 目标。
3. 实际写入文件清单。
4. 只读未改文件清单。
5. 本批 Git 分类：`docs-only`、`evidence-only`、`page-local visual`、`page-local structure`、`shared/global visual`、`backend-adaptation` 或 `tests-only`。
6. 是否触碰或需要触碰 `src/contracts/**`、`src/ports/**`、`src/implementations/**`、`src/services/content/**`、`src/components/**`、`src/pages.json`、`src/manifest.json`、全局样式或正式规则。
7. 是否存在字段缺口；字段缺口必须写清字段用途、所在页面、期望展示位置、当前证据来源和后端契约状态。
8. 是否使用了 `exported-unused`、`unresolved`、`inferred` 或 Level C 项。
9. 是否需要主线新增 adapter、runtime、service 或 view-model 字段。
10. 是否需要 375 / 390 / 430 宽度视觉证据包。
11. 是否存在会影响 P11.12 差异矩阵的页面消费变化。
12. 是否有阻塞需要主线接手。

## 7. P11.12 开工门槛

满足以下条件后，主线可以进入 `P11.12.1`：

1. 当前 Git 脏链已分类清楚。
2. 二线通讯中没有未说明的源码写入。
3. `规则体系/后端契约/` 能作为当前项目级后端契约入口读取。
4. 已明确 `P11.12.1` 是签收与冻结批，不是直接改代码批。
5. 已确认本批不改视觉、不改二线产物、不做物理删除。

如果二线新增未回传页面改动，先读档、分类、记录，再决定是否推迟 `P11.12.1`。

## 8. P11.12 停止条件

命中以下任意情况必须停止：

1. 新版后端契约来源不清。
2. 后端契约仍在频繁变化，无法冻结本轮基线。
3. 需要直接改视觉才能继续。
4. 需要直接改交互语义才能继续。
5. 需要 UI 直接消费未签收字段才能继续。
6. 需要二线同时修改主线 ownership 文件。
7. Git 边界开始混乱，无法区分 docs-only、backend-adaptation、page-local visual 或 tests-only。
8. 需要执行删除、清退、物理清理或 `git rm`。
