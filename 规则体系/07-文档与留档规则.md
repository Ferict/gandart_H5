# 07-文档与留档规则

> 文档类型：规则正文  
> 状态：draft  
> 更新时间：2026-04-21
> 说明：本文件用于统一正式计划、正式审查、交付记录、即时思路和历史记录边界；目标固定为让关键行为可追溯、可反查。

## 1. 目标

本文件解决四个问题：

1. 什么是现行规则正文，什么是留档，什么是历史参考。
2. 新计划、新审查、新交付、新即时思路分别落到哪里。
3. 每类正式记录最少要写什么，才能可追溯、可反查。
4. 历史记录体系后续如何重建，而不再依赖旧台账群。

## 2. 总原则

文档与留档体系的首要目标固定为“现行规则清晰、历史记录可追溯、人工查找成本低”。

因此必须满足：

1. 规则正文与留档分层。
2. 留档与垃圾桶分层。
3. 新记录有明确落点。
4. 一切关键行为都能回链到规则、审查、计划、交付或历史记录。
5. 当现有留档结构无法承接新类型档案时，只能在用户明确允许后新建目录。

### 3.1 现行规则正文

现行规则正文固定落在：

1. 根 [AGENTS.md](/H:/工作/天异/uniapp+vue新架构/AGENTS.md)

### 3.2 现行口径

现行口径固定落在：

1. [规则体系/口径](/H:/工作/天异/uniapp+vue新架构/规则体系/口径/README.md)

规则：

1. 现行口径用于承载当前实现基线、当前仍有效的执行口径和当前明确不扩展范围。
2. 现行口径必须回链对应上位规则。
3. 现行口径不是留档，也不是长期规则正文。

### 3.3 正式留档

正式留档固定落在：

1. [留档](/H:/工作/天异/uniapp+vue新架构/留档/README.md)

### 3.4 垃圾与废弃内容

已废弃内容固定落在：

1. `垃圾桶/`

规则：

1. 垃圾桶不是现行入口。
2. 现行规则、现行计划、现行审查和现行交付不得回链垃圾桶。

### 3.5 删除禁止与留档转移规则

正式文档、历史材料和阶段性误落内容都不得直接删除。

严格规则：

1. 不得执行任何物理删除操作。
2. 不得使用 `Remove-Item`、`rm`、`del`、`rmdir`、`git rm`、`git clean` 或等价删除命令清理文档、留档、历史材料、临时误落文件或未跟踪内容。
3. 任何“删除、清理、清退、移除、裁掉、丢弃”诉求，都必须先转化为“转移到 `垃圾桶/` 冻结区”的处置方案。
4. 任何留档、规则、计划、审查、交付或历史材料转移前，必须先写明源路径、目标垃圾桶路径、转移原因、现行入口替代关系和后续索引。
5. 即使是本轮误建、误落或临时生成的文件，也不得自行删除；必须停止并向用户说明误建路径、影响范围和建议处理方式，等待用户确认。
6. 如 `垃圾桶/` 当前结构无法承接待转移对象，不得自行新建长期分类目录；必须停止并请求用户确认落点。

## 4. 当前正式落点规则

### 4.1 规则正文

以下内容必须写入 `规则体系/` 或根 `AGENTS.md`：

1. 稳定行为规则
2. 目录规范
3. 接口规则
4. 工程工具规则
5. 安全边界规则
6. 实现边界规则
7. 子代理规则

### 4.2 现行口径

以下内容必须写入 `规则体系/口径/`：

1. 当前实现基线
2. 当前仍有效的执行口径
3. 当前明确不扩展范围
4. 当前专题的现实默认值、现实入口或现实阶段边界
5. 当前尚未完全专题化、但必须有统一说法的过渡口径

规则：

1. 口径文必须回链对应上位规则。
2. 规则正文不得再混写“当前默认值”“当前只属于”“当前已知问题口径”这类现状型内容。
3. 留档不得代替口径层承接现行执行口径。

### 4.3 计划记录

以下内容进入 `留档/计划记录/`：

1. 正式计划
2. 主线计划
3. 分批计划
4. 治理计划

规则：

1. 正式计划不再写入 `留档/交付记录/`。
2. 计划开始执行后，必须在原计划文内回填执行状态与执行索引。
3. 新顶层阶段或新正式计划开工前，计划文必须写明“已检查规则”。
4. 计划文必须写明本批的计划提交边界；若暂不提交，必须写明延后条件。
5. 若计划将进入自动化、`codex exec` 或其他非交互执行模式，计划文必须额外写明：
   - 停止边界
   - 失败退出条件
   - 完成后汇报要求
   - 已规划的尽头

### 4.3.1 新类型留档目录审批规则

当现有 `留档/` 分层无法承接新的长期档案类型时，执行以下规则：

1. 先判断是否可被现有 `计划记录/`、`审查记录/`、`交付记录/`、`即时思路记录/`、`历史记录/` 吸收。
2. 若现有层级无法承接，必须先取得用户明确允许，才可新建新的留档目录。
3. 新建后必须同步补齐：
   - 目录级 `README.md`
   - 与现有五类记录的边界关系
   - 在 [留档/README.md](/H:/工作/天异/uniapp+vue新架构/留档/README.md) 中登记入口
   - 在相关规则正文或 `AGENTS.md` 中补充高层提示
4. 未获批准前，不得把临时中转目录写成长期正式档案层。

### 4.3.2 唯一总规划挂载规则

当某个专题已经完成“唯一总规划入口”挂载时，必须同步满足：

1. [留档/README.md](/H:/工作/天异/uniapp+vue新架构/留档/README.md) 必须把该总规划提升为当前关键入口，不再把旧计划链并列列为同级现行入口。
2. [规则体系/README.md](/H:/工作/天异/uniapp+vue新架构/规则体系/README.md) 与相关 [规则体系/口径/README.md](/H:/工作/天异/uniapp+vue新架构/规则体系/口径/README.md) 必须回指该总规划和对应现行口径。
3. 同专题的旧计划如仍保留，只能标注为 `已完成`、`已搁置`、`准备批`、`历史参考` 或其他非现行状态，不得继续以“当前主入口”口吻扩写。
4. 现行执行基线仍写在 `规则体系/口径/`，总规划仍写在 `留档/计划记录/`；两者必须一一对应，不得分别长出第二套现行说明。

### 4.3.3 未来 Skill 派生规则

当某个专题后续准备沉淀为 skill 时，必须同步满足：

1. skill 只能派生自已经落盘的 `规则体系/`、`规则体系/口径/` 和 `留档/计划记录/`，不得先写 skill 再反向倒逼现行规则改口。
2. 当前执行口径一旦变化，必须先更新正式规则、正式口径或正式计划，再同步映射到 skill；不得只改 skill 不改现行真值源。
3. skill 只作为执行入口、复用封装或操作引导，不得成为并列现行真值源。
4. 若 skill 与现行规则冲突，以 `规则体系/`、`规则体系/口径/`、`留档/` 中的正式落盘内容为准；skill 必须回链这些正式入口。
5. 在用户未单独批准前，不得为了 skill 化新建顶层 `.codex/`、顶层 `docs/` 或任何平行长期目录承载“当前说明”。

### 4.4 交付记录

以下内容进入 `留档/交付记录/`：

1. 正式方案
2. 正式执行记录
3. 批次结论
4. 主线完成记录

### 4.5 审查记录

以下内容进入 `留档/审查记录/`：

1. 只读审查
2. 风险判断
3. 结构审查
4. 契约审查

### 4.6 即时思路记录

以下内容进入 `留档/即时思路记录/`：

1. 当前轮已经形成、但尚未进正式规则或正式方案的判断
2. 临时方向选择
3. 待拍板问题

### 4.7 历史记录

后续统一历史记录体系的长期目标目录为：

1. `留档/历史记录/`

它负责承载：

1. 关键行为流水
2. 关键批次追溯
3. 可回查的历史事件链

当前阶段旧历史目录已开始物理并入新历史体系；后续不得再新增长期保留在仓根的旧治理入口。

建议长期子目录为：

1. `留档/历史记录/对话/`
2. `留档/历史记录/批次档案/`
3. `留档/历史记录/索引与总账/`

## 5. 旧路径过渡规则

在新历史记录体系完全重建前：

1. `审核与修改计划/`
2. `对话台账/`
3. `项目台账.md`
4. `项目索引/`

当前只保留历史参考属性，不再作为现行规则真值源。

规则：

1. 新规则不再写回这些旧路径。
2. 新的正式计划、正式审查、正式交付和新历史结构，优先进入 `留档/`。
3. 旧历史目录的物理迁移留到后续阶段，不在本规则中提前执行。

### 5.1 旧历史迁移顺序

旧历史路径后续迁移顺序固定建议为：

1. `项目索引/`
2. `对话台账/`
3. `审核与修改计划/`
4. `项目台账.md`

顺序原则：

1. 先迁移低耦合、易分流的目录。
2. 再迁移单维时间流水目录。
3. 再迁移体量最大、结构最重的历史档案目录。
4. 最后迁移全局总账，避免在迁移过程中失去全局回链。

### 5.2 旧历史文件分类规则

旧历史文件后续分为两类：

#### A 类：迁入新历史体系

满足以下任一条件的历史文件，应迁入 `留档/历史记录/`：

1. 对关键行为有事实记录价值
2. 对关键批次有追溯价值
3. 对后续审查、复盘或责任回链有证据价值
4. 能明确回答“何时、谁、做了什么、为什么做”

#### B 类：不迁入，只保留冻结快照或最终进垃圾桶

满足以下任一条件的文件，不应迁入新历史体系：

1. 纯重复镜像
2. 只剩失效入口作用
3. 大量引用已死路径且无独立历史价值
4. 纯模板壳、空壳文件、过程噪音
5. 迁入后只会增加历史目录噪音

规则：

1. 新历史体系不是旧台账群的原样搬家。
2. 迁入前必须先分类，而不是整目录直接复制。
3. B 类内容最终可保留冻结快照后再进垃圾桶。

## 6. 正式文档头部规则

所有手工维护的正式 `.md` 文件至少应包含：

1. 文档类型
2. 状态
3. 更新时间
4. 文件职责说明

若文件属于正式执行记录或正式审查，建议同时写明：

1. 当前批次或阶段
2. 适用对象

## 7. 各类记录的最小字段

### 7.1 正式计划

正式计划至少写明：

1. 目标
2. 范围
3. 不做什么
4. 顺序
5. 验收标准
6. 待拍板项
7. 已检查规则
   - 至少列出本次开工前重新检查的规则入口，如 `AGENTS.md`、相关规则正文和相关口径入口
8. 计划提交边界
   - 写明本批预期按什么边界提交；若暂缓提交，必须写明暂缓原因和触发提交的条件
9. 执行状态
   - 至少使用 `未开始`、`进行中`、`已完成`、`已中止` 或 `已替代`
10. 执行索引

- 一旦进入执行，必须补回对应审查记录、交付记录或阶段复核的索引
- 若尚未开始，可写 `暂无`

### 7.1.1 非交互执行计划附加字段

当计划将交给自动化、`codex exec` 或其他非交互模式执行时，计划文必须额外包含：

1. `停止边界`
2. `失败退出条件`
3. `完成后汇报要求`
4. `已规划的尽头`

规则：

1. 自动化等同于 `codex exec` 非交互执行模式。
2. 自动化不是无限续做授权，也不是只做计划后停下。
3. 自动化必须先补齐或确认计划，然后执行到该计划的验收标准、完成条件、固定停止点或失败退出条件。
4. 未写清停止边界、失败退出条件、完成后汇报要求和已规划的尽头的计划，不得直接进入自动化执行。
5. 自动化执行计划的细化要求以 [10-自动化执行边界专题细则.md](/H:/工作/天异/uniapp+vue新架构/规则体系/10-自动化执行边界专题细则.md) 为准。

### 7.2 正式审查

正式审查至少写明：

1. 审查目标
2. 审查对象
3. 证据来源
4. 已确认问题
5. 影响范围
6. 风险与遗留项
7. 后续状态
   - 至少使用 `未修复`、`部分修复`、`已修复`、`已转规划` 或 `已作废`
8. 修复或处置索引
   - 一旦问题进入修复、转规划或作废裁决，必须补回对应交付记录、计划记录或替代审查记录的索引
   - 若尚未进入后续动作，可写 `暂无`

### 7.3 正式执行/交付记录

正式执行或交付记录至少写明：

1. 做了什么
2. 没做什么
3. 写入了哪些规则或结构
4. 验证情况
5. 下一步
6. 问题状态
   - 修复类记录必须明确写 `未修复`、`部分修复` 或 `已修复`
   - 非修复类记录可写 `不适用`
7. 修复方式
   - 修复类记录必须写明采用了什么修复方式
   - 若未修复或仅部分修复，必须写明当前未落地的部分
8. 审查回链
   - 若修复的是此前审查已确认的问题，必须回链对应审查记录
   - 应明确说明本次交付对应修复了审查中的哪一条问题
9. 提交状态
   - 至少使用 `未提交`、`已提交`、`部分提交` 或 `不适用`
10. 提交边界或提交索引

- 若已提交，应写明对应提交边界、提交说明或 commit 索引
- 若未提交或仅部分提交，应写明当前暂缓原因和预期提交边界

若对应一次非交互执行，还必须写明：

1. `实际停止位置`
2. `剩余未做事项`
3. `下一轮建议入口`
4. `是否到达已规划的尽头`

### 7.4 即时思路记录

即时思路记录至少写明：

1. 思路来源
2. 当前判断
3. 关键假设
4. 待拍板点
5. 后续应回写到哪里

### 7.5 历史记录

后续历史记录至少应能回答：

1. 何时
2. 谁
3. 做了什么
4. 为什么做
5. 依据是什么
6. 结果是什么
7. 回链到哪里

迁入新历史体系的历史文件，建议至少补齐以下迁移元信息：

1. 原始路径
2. 原始类别
3. 对应批次或日期范围
4. 迁移批次
5. 迁移原因
6. 关联正式记录

## 8. 回链规则

1. 正式计划应回链到对应方案或上层总计划。
2. 正式执行记录应回链到对应计划。
3. 正式审查应回链到被审对象或对应批次。
4. 即时思路进入正式方案后，应保留回链，不再单独充当真值源。
5. 历史记录应回链到正式规则、正式计划、正式审查或正式执行记录，而不是只回链聊天内容。
6. 修复类正式交付记录如对应此前审查问题，必须同时回链对应审查记录，并在正文中标明问题状态与修复方式。
7. 正式计划一旦进入执行，原计划文必须回填“执行状态”和“执行索引”，不能只在交付记录里宣布已做。
8. 正式审查一旦有后续修复、转规划或作废裁决，原审查文必须回填“后续状态”和“修复或处置索引”，不能只在修复记录里单向回链。
9. 正式计划在开工前必须回填“已检查规则”和“计划提交边界”，不能只在聊天里口头说明。
10. 正式交付记录必须写明“提交状态”和“提交边界或提交索引”，不能只在 Git 历史中隐含存在。

## 9. 命名与组织规则

1. 正式记录文件名优先采用 `YYYY-MM-DD-主题.md`。
2. 同一批次如需分阶段，可采用 `YYYY-MM-DD-G0.1-主题.md` 这类命名。
3. 目录命名优先表达职责，不用含糊缩写。
4. 同一目录内不混放语义完全不同的正式文件。

## 10. 子代理留痕规则

当启用有实质工作的子代理时，正式记录中至少要能追溯：

1. 启用时间
2. 任务目标
3. 模型/能力等级
4. 文件所有权

## 11. 禁止事项

2. 不允许把留档当成现行规则真值源替代品。
3. 不允许继续向旧台账群写新的现行规则。

5. 不允许把垃圾桶当成历史记录体系的一部分。

7. 不允许把现行口径写进 `留档/` 代替 `规则体系/口径/`。
