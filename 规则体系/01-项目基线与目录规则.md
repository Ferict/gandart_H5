# 01-项目基线与目录规则

> 文档类型：规则正文  
> 状态：draft  
> 更新时间：2026-04-21
> 说明：本文件先行定义仓级目录规范、`src/` 目标结构蓝图和接口契约目录目标。当前只定规则，不开始物理整理。

## 1. 目标

本文件解决三个问题：

1. 仓库长期应如何分层，才利于人工维护。
2. `src/` 长期应该长成什么样，才不再显得“非常 AI、非常碎”。
3. 后端接口文档应该集中到哪里，才便于后端和前端共同维护。

## 2. 总原则

目录规范的首要目标固定为“人工可维护”，不是“架构术语看起来完整”。

因此必须满足：

1. 人一眼能看懂目录职责。
2. 新文件有明确落点。
3. 现行内容、历史内容、垃圾内容不混用。
4. 每个长期目录都有自己的说明文件。
5. 业务语义优先于纯技术抽象。
6. 新类型目录只有在现有结构无法承接且用户明确允许时才可建立。

### 3.8 删除禁止与垃圾桶转移规则

当前仓的目录治理固定采用“冻结转移”模型，不采用直接删除模型。

严格规则：

1. 不得执行任何物理删除操作。
2. 不得使用 `Remove-Item`、`rm`、`del`、`rmdir`、`git rm`、`git clean` 或等价删除命令清理文件、目录、索引或未跟踪内容。
3. 任意“删除、清理、清退、移除、裁掉、丢弃”诉求，都必须先转化为“转移到 `垃圾桶/` 冻结区”的处置方案。
4. 转移前必须明确源路径、目标垃圾桶路径、转移原因、是否影响现行引用、验证方式和回滚方式。
5. 涉及 `src/`、`tests/`、`scripts/`、`.github/`、`规则体系/`、`留档/`、`backend-handoff/`、`references/` 或根工程配置时，必须先有审查记录或用户明确确认。
6. 即使是本轮误建、误落或临时生成的文件，也不得自行删除；必须停止并向用户说明误建路径、影响范围和建议处理方式，等待用户确认。
7. 如 `垃圾桶/` 当前结构无法承接待转移对象，不得自行新建长期分类目录；必须停止并请求用户确认落点。

执行口径：

1. 需要退出现行路径的内容，优先通过移动到 `垃圾桶/` 保留冻结证据，而不是从磁盘或 Git 历史中抹除。
2. 转移到 `垃圾桶/` 后，该内容不再作为现行依据；不得从现行规则、接口、说明或源码回链垃圾桶内容。
3. 若用户明确要求“不保留、直接删除”，仍必须先复述风险并等待二次确认；未二次确认前不得执行。
4. 自动模式等同于 `codex exec` 非交互执行模式；自动模式、`codex exec`、子代理任务和当前交互会话都适用本规则。

## 4. 长期目录说明规则

从新规则体系生效起，所有长期维护目录都必须具备自己的 `README.md`。

至少包括：

1. 顶层长期目录
2. `规则体系/`
3. `留档/`
4. `src/` 下长期维护的大模块目录

目录级 `README.md` 至少要说明：

1. 目录职责
2. 内容边界
3. 常见文件落点
4. 不应放入的内容

## 4.1 新类型目录审批规则

当出现现有结构无法承接的新类型档案、文件类别或长期目录需求时，执行以下规则：

1. 先判断是否真的是“新类型”，而不是现有目录缺少子目录或命名不清。
2. 未经用户明确允许，不得自行新建新的长期顶层目录，也不得自行新建新的长期分类目录。
3. 用户明确允许后，方可新建目录承接该类内容。
4. 新目录一旦建立，必须同步补齐：
   - `README.md`
   - 目录职责与内容边界
   - 新文件落点说明
   - 对应的规则入口或留档入口回链
5. 若只是阶段性中转，不得伪装成长期正式目录。

## 5. 文件头职责注释规则

### 5.1 文档文件

所有手工维护的正式 `.md` 文件都应包含头部元信息：

1. 文档类型
2. 状态
3. 更新时间
4. 文件职责说明

### 5.2 源码文件

所有非平凡 `.ts`、`.js`、`.vue` 文件都应包含职责注释或等价说明。

最少回答：

1. 这个文件负责什么
2. 这个文件不负责什么

### 5.3 豁免范围

以下文件可豁免：

1. 生成物
2. 静态资源
3. lockfile
4. 极短的纯导出聚合文件
5. 框架强约束的简短配置文件

## 6. `src/` 目标结构蓝图

### 6.1 当前问题

当前 `src/` 存在以下结构性问题：

1. 顶层技术目录过多，业务分层不足。
2. `home` 相关实现被拆成大量平铺微文件。
3. page-local、service、mock、contract、implementation 的边界风格不统一。
4. 缺少目录级说明和文件职责头，维护依赖记忆。

### 6.2 长期目标

`src/` 的长期结构应转为“业务优先，技术从属”。

目标蓝图建议为：

1. `src/app/`
   - 启动、provider、全局接线、运行时总入口
2. `src/pages/`
   - 只保留 uni-app 路由壳和页面注册耦合内容
3. `src/features/`
   - 各业务域实现
4. `src/shared/`
   - 真正跨业务共享的组件、工具、基础模型
5. `src/static/`
   - 静态资源

### 6.3 页面壳规则

后续 `src/pages/*` 的职责应收敛为：

1. 路由壳
2. 页面入口
3. 少量页面级 orchestration

不再长期承载：

1. 过多业务细节
2. 大量共享逻辑
3. 跨页面复用实现

### 6.4 共享与业务归属规则

后续应按以下边界落点：

1. 单业务域专属实现进入 `features/<domain>/`
2. 跨业务共享内容进入 `shared/`
3. 启动和 provider 进入 `app/`
4. 页面私有壳留在 `pages/`

### 6.5 过渡期写入规则

在真正重构 `src/` 之前，先执行以下过渡规则，防止继续变乱：

1. 不再新增新的 `src/` 顶层技术目录。
2. 新的页面专属逻辑优先放在对应页面或未来业务域附近，不再默认丢进根 `services/` 或 `utils/`。
3. `src/services/` 只允许新增“真正跨业务、跨页面”的运行时服务。
4. `src/utils/` 只允许新增“真正无业务归属”的工具。
5. `src/models/` 只允许新增“跨业务共享模型”。
6. `src/mocks/` 与 `src/implementations/content.mock/` 的长期并存视为问题，不得继续扩散同类分裂。
