## 2. 这套契约是什么

这套契约是从旧前端源码抽取出来的接口资料，不是后端重新提供的 OpenAPI。

它能说明：

1. 旧前端导出了哪些接口 wrapper。
2. 哪些 wrapper 有真实页面或组件调用。
3. 旧前端真实消费了哪些请求字段和响应字段。
4. 哪些接口或字段仍有未确认风险。

它不能直接说明：

1. 新版后端已经全部支持。
2. 字段业务语义已经由后端最终签收。
3. 页面可以直接消费旧 wrapper 响应。

## 4. 维护口径

1. 旧前端 wrapper 成功包常见为 `code=1`，新版内容域 envelope 默认成功为 `code=0`；接入时必须归一，不得误判失败。
2. `confirmed` 可以作为接入基线，但字段语义、分页、排序、空值和错误包仍需要真实后端样例确认。
3. `exported-unused`、`contracts/inferred/*` 和 `unresolved.md` 不能直接作为新版前端 UI 稳定依赖。
4. 新版前端页面不得直接读取旧 wrapper 响应结构，必须经过 backend implementation、adapter、runtime/service 和 view-model。

## 5. 当前用户已裁定事项

1. 首页 banner 接线走 `banner.getBannerList` / 旧 `getBanner`。
2. 发售日历 `collection.getCalenderList` 通过 `goods_type=2` 显示藏品，通过 `goods_type=1` 显示盲盒。
3. 个人中心区块链地址优先使用旧用户信息接口返回的 `ntf_url`。
4. 个人中心“资格证”不作为一级资产分类，归入藏品/资产下的系列或二级分类。

## 6. 后端需要重点确认

1. 旧 wrapper 是否继续统一返回 `code=1` 成功包。
2. 列表分页字段和末页判断规则。
3. 市场排序是否由后端支持，以及排序参数和空值规则。
4. 价格字段单位和类型。
5. `goods_type`、`series_id`、`series_name`、`ntf_url` 等关键字段是否稳定保留。
6. `exported-unused` 接口是否保留、废弃或等待后续恢复。
