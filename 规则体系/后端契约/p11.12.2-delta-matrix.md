# P11.12.2 新旧契约差异矩阵

> 文档类型：后端契约差异矩阵  
> 状态：active  
> 更新时间：2026-04-25  
> 说明：本文件对照当前前端内容域契约 `src/contracts/content-api.contract.ts` 与项目级后端契约冻结基线 `p11.12.1-freeze.md`；它是后续 adapter/runtime/service 迁移输入，不是代码改造结果。

## 1. 矩阵结论

当前差异结论固定为：

1. 当前前端内容域契约是 5 个抽象聚合接口：
   - `GET /api/content/scene`
   - `GET /api/content/resource`
   - `GET /api/content/list`
   - `POST /api/content/action/notice-read`
   - `POST /api/content/action/service-reminder-consume`
2. 项目级后端契约冻结基线是旧 H5 业务域 wrapper：
   - 190 个 wrapper
   - 20 个 domain
   - 122 个 `confirmed`
   - 68 个 `exported-unused`
3. 二者不是同构关系，不能直接替换。
4. 当前正确迁移方向是：保留 `ContentPort` 和页面稳定 view-model，在 implementation / adapter / runtime-service 层把后端具体 wrapper 组合成当前 UI 需要的稳定模型。
5. `content-api.contract.ts` 不在本批作废，仍是当前前端内容域实现契约。

## 2. 顶层差异矩阵

| 当前内容域接口 | 当前语义 | 后端冻结基线形态 | 差异类型 | 责任层 | 结论 |
| --- | --- | --- | --- | --- | --- |
| `scene` | 按 `sceneId` 返回页面首屏聚合块 | 后端无等价 scene 聚合接口，只有 banner、notice、market、user、collection 等具体 wrapper | 抽象聚合 vs 具体业务接口 | implementation + adapter + runtime/service | 保留 `scene` 对 UI 的语义，底层用多 wrapper 组合 |
| `resource` | 按 `resourceType/resourceId` 返回统一详情资源 | 后端按 domain 拆分详情，例如 market、collection、notice、lottery | 统一资源模型 vs 多详情接口 | adapter | 按 resourceType 分派到具体 wrapper，再归一化为 `ContentResourceDto` |
| `list` | 按 `resourceType` 返回 market、notice、profile_asset 三类分页列表 | 后端按具体列表 wrapper 拆分，且大多为 POST | 统一分页模型 vs 多列表接口 | implementation + adapter | 保留当前 `ContentListDto`，底层按 resourceType 映射 wrapper |
| `notice-read` | 公告已读写回 | 后端有 message read 相关 wrapper，但多为 `exported-unused`，且与公告/消息语义可能不完全一致 | action 语义不完全确认 | P11.12 待确认 | 不直接接；需后端确认是否存在正式公告已读接口 |
| `service-reminder-consume` | 服务入口提醒消费 | 冻结基线无明确等价 confirmed wrapper | 缺少等价接口 | P11.12 待确认 | 保持当前 mock/page-state，等待后端确认 |

## 3. scene 差异矩阵

| 当前 `sceneId` | 当前 UI 消费 | 候选后端 domain / wrapper | 状态 | 差异与阻塞 | 责任层 |
| --- | --- | --- | --- | --- | --- |
| `home` | 公告条、banner、首发、市场概览 | `banner.getBannerList`、`notice.getNoticeList`、`notice.getNoticeDetails`、`market.getMarketList`、`market.getMarketBoxList`、`collection.getCalenderList`、`collection.getCollectionDetails` | 多数 confirmed | 无单一聚合接口；需要多请求组合、排序与字段归一化 | implementation + adapter + home-rail service |
| `activity` | 活动入口、公告 feed | `active.category`、`news.getActivityList`、`notice.getNoticeList`、`notice.getNewsList`、`lottery.*`、`draw.*` | 多数 confirmed，部分 exported-unused | 活动、公告、抽签语义混杂；需要先定活动页首屏真实来源 | implementation + adapter + activity service |
| `profile` | 用户摘要、资产分类、个人藏品列表首屏 | `user.getUserInfo`、`collection.antMycollection`、`collection.getUserCollectionDetails`、`collection.user_collection_Detail`、`collection.getSeriesList`、`collection.getSelfSeries` | 多数 confirmed，部分候选 | 当前 profile summary 字段与旧 H5 用户/藏品接口不是一一对应 | implementation + adapter + profile service |
| `settings` | 统一建设中或设置承载 | 当前冻结基线有 user/address/wallet 等具体接口 | confirmed + exported-unused | 当前现行 UI 未作为重点迁移线；不得抢先扩展设置页 | 暂缓 |
| `service_hub` | 服务入口提醒 | user/order/wallet/address 等具体接口 | confirmed + exported-unused | 缺少统一提醒消费接口；提醒模型需要后端确认 | 暂缓 / 待确认 |

## 7. UI 迁移线矩阵

| 迁移线 | 当前 UI 消费 | 后端候选入口 | 第一落点 | 阻塞项 |
| --- | --- | --- | --- | --- |
| `L1 home/home` | notice bar、banner、featured drop、market list | banner、notice、market、collection/box | 新建或改造 content implementation 的 concrete backend adapter，再复用 `homeRailHomeContent.adapter.ts` | 首发 drop 来源需确认 |
| `L2 home/activity` | 活动入口、公告 feed、notice read | active、news、notice、lottery/draw | activity adapter + action seam | 公告 read action 只有候选，不能硬接 |
| `L3 home/profile` | profile summary、asset list | user、collection | profile adapter + list adapter | 主资产列表 wrapper 需确认首选 |
| `L4 profile-asset-detail` | hero、value、provenance、traits、description | collection details | detail adapter | 五卡片字段可能超过旧后端 confirmed 响应，需要字段缺口表 |
| `L5 updating` | 抽签承载页 | lottery、draw | page-local seam 或正式 content seam 待裁决 | 是否并入正式 content 主链待确认 |

## 8. 当前责任层裁决

| 差异类型 | 责任层 |
| --- | --- |
| path / method 从 `/api/content/* GET` 变成具体业务 `POST /xxx` | implementation |
| wrapper 多请求组合成一个 scene | implementation + runtime/service |
| 后端字段名、价格单位、图片结构、状态枚举归一化 | adapter |
| UI 卡片字段、展示文案、占位状态 | runtime/service 或 view-model，不进 contract |
| 未确认字段、缺响应样例、`FrontendUnknown` | 待确认清单，不接 UI |
| 页面 loading、分页状态、缓存策略 | page/runtime，不进后端契约 |
| provider 切换 | 后续 `P11.12.4`，不在本矩阵批改代码 |

## 9. 当前不能做的事

1. 不能把 190 个旧 wrapper 直接暴露给页面。
2. 不能用旧后端 wrapper 替代当前 `ContentPort`。
3. 不能让 Gemini UI 二线按 `api-inventory.md` 自行接字段。
4. 不能把 `exported-unused` 当 confirmed。
5. 不能跳过 `P11.12.3` 目标分层，直接改 implementation。

## 10. 下一步输入

下一步进入 `P11.12.3` 时，必须基于本矩阵确定：

1. 是否保留 `ContentPort` 作为 UI 稳定边界。
2. 新后端 concrete implementation 的命名与落点。
3. 每条迁移线的 wrapper 组合方式。
4. `FrontendUnknown` 响应和五卡片字段缺口如何标记。
5. mock/http/provider 过渡策略是否需要先开 `P11.12.4` 再动代码。
