# 02-工程与工具规则

> 文档类型：规则正文  
> 状态：draft  
> 更新时间：2026-04-21
> 说明：本文件用于统一工程命令、质量门禁、环境变量边界和工具角色；先定规则，不引入新的工程工具栈。

## 1. 目标

本文件解决四个问题：

1. 项目应如何启动、构建和验证。
2. 哪些命令是正式入口，哪些只是临时排查。
3. 各质量工具分别负责什么，不互相替代什么。
4. Windows 本地环境应如何搭建，才能稳定复现仓内检查。

## 2. 总原则

工程与工具规则的首要目标固定为“任何维护者都能低成本复现同一套命令和结果”。

因此必须满足：

1. `package.json` 中的 `scripts` 是正式命令真值源。
2. 常用运行、构建、验证命令要可直接复制执行。
3. 质量门禁要分层，避免混淆“日常开发检查”和“全量质量检查”。
4. 平台相关要求要明确写出，尤其是 Windows 本地工具链。

## 3. 命令真值源规则

当前正式命令入口固定为：

1. [package.json](/H:/工作/天异/uniapp+vue新架构/package.json)

规则：

1. 可重复使用的工程命令必须优先沉淀到 `package.json` scripts。
2. 不允许长期依赖口头命令或散落在台账里的命令片段。
3. 对外说明项目如何启动或验证时，优先引用 `npm run ...` 入口，而不是直接写底层工具长命令。

## 4. 本地运行与构建规则

### 4.1 安装依赖

标准安装命令：

```bash
npm install
```

说明：

1. `prepare` 会随依赖安装触发 `lefthook install`。
2. 不允许把“本地先手动执行一堆隐式准备动作”当成默认前提。

### 4.2 本地开发

当前正式本地开发命令：

1. `npm run dev:h5`
2. `npm run dev:h5:5180`

规则：

1. 默认本地联调优先使用 H5 开发入口。
2. 需要稳定端口时，优先使用 `dev:h5:5180`。
3. 除非任务明确要求 App 包构建，否则不把 App 构建链作为默认日常开发入口。

### 4.3 构建

当前正式构建命令：

1. `npm run build:h5`
2. `npm run build:app`
3. `npm run build:app-harmony`

规则：

1. 构建命令只说明产物构建，不替代功能验证。
2. 未明确要求发版或包体验证时，不默认执行所有构建命令。

## 5. 验证分层规则

### 5.1 基础开发验证

普通代码改动的基础验证默认优先级为：

1. `npm run type-check`
2. `npm run lint`
3. `npm run test:unit`

### 5.2 交互与页面链路验证

当改动涉及以下内容时，应追加：

1. 页面交互
2. 路由
3. 刷新协议
4. 缓存链
5. 列表/详情联动
6. 首屏稳定性

对应追加命令：

1. `npm run test:e2e`

### 5.3 全量质量验证

需要做较完整质量回归时，使用：

1. `npm run quality:check`

它当前包含：

1. `type-check`
2. `lint`
3. `test:unit`
4. `knip`
5. `lint:docs`

### 5.4 扩展质量验证

需要补安全或文档风格门禁时，使用：

1. `npm run lint:semgrep`
2. `npm run lint:vale`
3. `npm run lint:external-resources`
4. `npm run audit:deps`

规则：

1. `semgrep` 属于静态安全/规则扫描，不替代类型检查和测试。
2. `vale` 属于文档措辞门禁，不替代正式规则正文审阅。
3. `audit:deps` 负责暴露依赖风险，不替代功能回归。

### 5.5 Git 提交边界与规则重查

当前工程执行基线固定为：

1. 每个可独立验证且边界清楚的批次，一旦完成本批所需检查，就应尽快形成 Git 提交。
2. 文档-only 批次至少完成 `git diff --check` 后再提交。
3. 代码批次至少完成本批所需的类型检查、单测或其他对应验证后再提交，不允许把“还在同一主线里继续做”当成长期不提交的理由。
4. 若因边界未稳定而需要暂缓提交，必须在计划记录或交付记录中写明暂缓原因与预期提交边界。
5. 每次进入新的顶层阶段（`G*` / `P*`）或新正式计划前，必须重新检查 `AGENTS.md` 与相关规则正文，而不是只依赖上一轮记忆。

## 6. 工具角色规则

当前仓内工具角色固定为：

1. `vue-tsc`
   - 类型检查真值源
2. `eslint`
   - 脚本与 Vue 代码风格、静态规则
3. `stylelint`
   - 样式规则
4. `prettier`
   - 格式一致性
5. `vitest`
   - 单元测试、纯逻辑测试、service/composable 测试
6. `playwright`
   - H5 烟测与交互链验证
7. `knip`
   - 未使用文件、导出、依赖检查
8. `textlint`
   - 文档类基础 lint
9. `semgrep`
   - 安全与仓规类静态扫描
10. `vale`

- 文档措辞和术语检查

11. `lefthook`

- 提交前/推送前门禁接线

12. `commitlint`

- 提交信息格式约束

13. `promptfoo`

- AI 规则评测，不作为工程功能验证替代

14. `Continue CLI`

- AI review 辅助工具，不作为正式质量结论替代

规则：

1. 任何工具都不能越权替代别的验证层。
2. AI 工具的输出只能作为辅助证据，不能替代类型检查、测试和正式代码审查。

## 7. Windows 本地工具链规则

当前 Windows 首次质量工具安装入口固定为：

1. `npm run setup:quality:windows`

该命令当前负责：

1. 安装 `Semgrep`
2. 安装 `Vale`
3. 下载 `Reviewdog`
4. 安装 `Playwright Chromium`
5. 安装 `Lefthook hooks`

规则：

1. Windows 是当前仓的正式本地维护环境之一。
2. 若本地缺少系统级工具，优先走仓内安装脚本，不自己发明新安装说明。

## 8. 环境变量边界

当前工程与工具层只约束环境变量边界，不承接后端联调 contract 本身。

规则：

1. provider、base URL、联调说明的正式口径以 [05-接口与联调规则.md](/H:/工作/天异/uniapp+vue新架构/规则体系/05-接口与联调规则.md) 为准。
2. 工具类环境变量只在对应工具文档或脚本入口说明。
3. 不允许把敏感环境变量直接写入正式规则正文示例值。

当前已知工具类环境变量示例：

1. `PROMPTFOO_PROVIDER`
2. `OPENAI_API_KEY`

## 9. 质量门禁回写规则

涉及工程或工具链的长期规则变更时，至少同步回写以下位置之一：

1. 本文件
2. [README.md](/H:/工作/天异/uniapp+vue新架构/README.md)
3. [质量门禁工具接入说明.md](/H:/工作/天异/uniapp+vue新架构/质量门禁工具接入说明.md)

规则：

1. `README.md` 负责给外部维护者最小入口。
2. 本文件负责现行规则真值源。
3. `质量门禁工具接入说明.md` 当前仍可保留为工具专项说明，但后续规则应以本文件为准，不再反向让专项说明成为规则源头。
4. 提交边界、提交时机与阶段开始前的规则重查，属于工程执行规则，必须同步回写到本文件或 `AGENTS.md`，不得只留在聊天结论中。

## 10. 禁止事项

1. 不允许把临时排查命令长期当成正式工程入口。
2. 不允许把某一项工具结果当成全部质量结论。
3. 不允许只在台账记录“应该跑哪些检查”，却不更新正式规则。
4. 不允许让 Windows 首次搭建继续依赖口头说明。
5. 不允许让 AI 工具输出替代正式验证链。
6. 不允许在自动化或 `codex exec` 这类非交互模式下，未先定义停止边界就直接连续扩题执行。
7. 不允许把非交互执行做成没有结束条件的长链条自动施工。

## 11. 与后续阶段的关系

本文件只定义“项目应该如何运行和验证”。

后续分工固定为：

1. `G0` 负责把工程与工具规则写清楚。
2. `P1` 与 `P2` 只在需要时补联调相关工具入口，不重写本文件的基础分层。
3. 后续若工具链调整，先改规则，再改脚本和说明。

## 12. `codex exec` 使用规则

`codex exec` 在当前仓只允许作为非交互执行壳使用，不改变本仓计划、留档、规则和 Git 的基本约束。

自动化、自动化修复、自动模式、云端任务和用户明确要求“直接做完”“继续做到规划尽头”的当前会话执行，均等同于 `codex exec` 非交互执行模式。

详细执行边界、停止条件和细则调取要求，以 [10-自动化执行边界专题细则.md](/H:/工作/天异/uniapp+vue新架构/规则体系/10-自动化执行边界专题细则.md) 为准。

执行含义：

1. 自动化必须先有计划，或先补齐当前批次的计划边界。
2. 自动化默认不是停在计划文，而是执行到当前计划的验收标准、完成条件、固定停止点或失败退出条件。
3. 自动化不得越过计划边界扩展到新顶层阶段、新长期目录、新业务目标或需要用户拍板的决策。
4. 自动化不得把每个小步骤完成都当成停止点；只在命中计划终点或停止边界时停下汇报。

固定要求：

1. 开工前必须已有正式计划，或在现有计划中明确本轮的执行目标、停止边界和汇报要求。
2. 每轮 `codex exec` 或自动化默认只处理一个当前批次，不跨多个顶层阶段。
3. 执行中如遇新的结构决策、目录许可、规则冲突或提交边界失真，必须停止并回到交互审查。
4. 执行完成后必须回写正式留档，并明确：
   - 已完成内容
   - 停止位置
   - 下轮建议
   - 提交状态
