# 08-交互与适配专题细则

> 文档类型：规则正文  
> 状态：draft  
> 更新时间：2026-04-13  
> 说明：本文件用于承接 UI 细颗粒专题规则，统一顶部底部结构、视口与键盘、热区与交互状态、滚动容器、小字号文字、毛玻璃降级与旧 UI 迁移重写口径。

## 1. 目标

本文件解决七类细颗粒问题：

1. 顶部、底部和内容起终点结构如何统一。
2. 视口、`safe-area`、键盘联动如何统一。
3. 热区、交互状态和小视觉控件双层结构如何统一。
4. 主滚动区、局部滚动区和滚动穿透如何统一。
5. 小字号和文字容器如何统一。
6. 毛玻璃与视觉降级如何统一。
7. 旧项目 UI 迁移时，什么可以参考，什么必须重写。

## 3. 顶部、底部与结构边界

### 3.1 命中场景

涉及以下任一内容时，必须命中本节：

1. 顶部栏、导航区、标题区
2. 底栏、吸底操作区
3. 内容起点、内容终点
4. 页面壳层的结构占位

### 3.2 强制规则

1. 顶部区必须明确：
   - 安全区延伸层
   - 标题区
   - 内容起点
2. 底部区必须明确：
   - 主标签栏
   - 吸底操作区
   - 内容滚动终点
3. 顶部和底部占位必须由共享壳层或统一结构处理，不得每页各自补丁修正。
4. 同一产品语义下的页面应复用同一套顶部和底部结构口径。
5. 页面切换、键盘弹起和内容滚动时，不得出现结构脱节或遮挡关键操作。

### 3.3 禁止事项

1. 禁止每页各自修一套顶部和底部补丁。
2. 禁止把底部吸附问题丢给绝对定位硬顶。
3. 禁止只修视觉层，不修结构占位层。

## 4. 视口、`safe-area` 与键盘

### 4.1 命中场景

涉及以下任一内容时，必须命中本节：

1. 视口高度
2. `safe-area`
3. 键盘弹起
4. 焦点输入区、发送区和关键操作区可见性

### 4.2 强制规则

1. H5 视口口径必须统一，不得一页一个算法。
2. 优先使用 `dvh` 与 `VisualViewport` 能力探测，并保留可靠降级路径。
3. `safe-area` 只做环境补偿，不承载页面设计职责。
4. 安全区补偿应集中在共享结构或明确边界层，不得散写到多个局部容器。
5. 键盘弹起时，优先保证焦点输入区、发送区和关键操作区可见。
6. 键盘联动必须与主滚动区协同，不得靠临时空白或硬编码位移顶过去。
7. 任何只在单一 H5 浏览器成立的方案，都只能记为过渡降级，不得当作最终结论。

### 4.3 禁止事项

1. 禁止每个页面自己造视口算法。
2. 禁止把 `safe-area` 当设计布局主工具。
3. 禁止靠魔法常量、空白堆叠或一次性补丁掩盖键盘顶起问题。

## 5. 热区、交互状态与小视觉控件

### 5.1 命中场景

涉及以下任一内容时，必须命中本节：

1. 按钮、图标按钮、列表项、卡片点击区
2. `hover / active / focus-visible / selected / disabled`
3. 小视觉控件

### 5.2 强制规则

1. 所有可点击元素必须接入统一共享交互基元。
2. 共享交互基元的最小物理热区不得低于 `44px x 44px`。
3. 共享交互基元必须统一管理：
   - `active`
   - `focus-visible`
   - `disabled`
   - `hover` 仅按输入设备能力补充
4. 图标按钮、角标按钮、轻量 chip 等小视觉控件必须采用“视觉层 + 独立交互层”结构。
5. 独立交互层必须是局部覆盖，不得扩展为全屏透明层。
6. 可视激活提示不得脱离真实交互层边界。
7. 导航区、浮层区、密集卡片区如需例外，必须进入正式例外清单并可回查。

### 5.3 禁止事项

1. 禁止在页面散写私有热区算法，绕开共享交互基元。
2. 禁止小视觉控件只有视觉层、没有独立交互层。
3. 禁止使用全屏透明层充当通用点击热区。
4. 禁止未登记例外就擅自放宽热区规则。

## 6. 滚动容器专题

### 6.1 命中场景

涉及以下任一内容时，必须命中本节：

1. 页面主滚动区
2. 局部滚动区
3. 弹层滚动
4. 横滑区
5. 滚动穿透

### 6.2 强制规则

1. 一个页面只能有一个明确主滚动区。
2. 主滚动区必须与顶部、底部和键盘联动策略协同定义。
3. 局部滚动容器只在确有必要时出现。
4. 局部滚动区必须说明其存在原因、边界和与主滚动区的关系。
5. 滚动穿透、滚动截断和 overscroll 行为必须被明确处理。
6. 结构性滚动问题优先通过容器层级解决，不得靠空白堆叠和脚本强顶替代。

### 6.3 禁止事项

1. 禁止页面同时存在多个语义不清的主滚动区。
2. 禁止通过绝对定位和空白垫高掩盖滚动结构问题。
3. 禁止在没有必要时叠加局部滚动容器。

## 7. 小字号与文字容器

### 7.1 命中场景

涉及以下任一内容时，必须命中本节：

1. 正文、标签、角标、按钮文案、说明文字
2. 视觉小字号
3. 缩放策略
4. 断行和容器稳定性

### 7.2 强制规则

1. 视觉上低于 `12px` 的文本必须走“`12px + scale`”策略。
2. 装饰性文案视觉最小限制为 `8px`，且必须通过“`12px + scale`”实现。
3. 可读性文案最小物理字号为 `12px`。
4. 标签、角标、按钮和说明文字必须先有稳定容器，再考虑视觉压缩。
5. 文字适配先服从可读性和可维护性，再追求极小视觉密度。
6. 当文字过长时，应优先调整容器、层级或文案策略，而不是继续压小字号。

### 7.3 禁止事项

1. 禁止直接写物理极小字号。
2. 禁止只压字体不处理容器。
3. 禁止为了塞下文案牺牲可读性。

## 8. 毛玻璃与视觉降级

### 8.1 命中场景

涉及以下任一内容时，必须命中本节：

1. 毛玻璃
2. 背景模糊
3. 半透明表面
4. 视觉能力降级

### 8.2 强制规则

1. 毛玻璃、透明层和模糊只作为辅助效果，不得成为可读性的前提。
2. 效果必须建立在稳定结构、文字可读和交互可用之上。
3. 必须明确普通实体表面、纯色表面或弱阴影表面的降级路径。
4. 降级只能降低效果复杂度，不得破坏结构语义和交互层级。
5. 当效果影响性能或可读性时，应优先降级，而不是强撑视觉方案。

### 8.3 禁止事项

1. 禁止把毛玻璃当作信息可读性的前提条件。
2. 禁止没有降级路径就把高成本效果写入主链。
3. 禁止用重模糊掩盖结构和层级问题。

## 9. 旧项目 UI 迁移重写

### 9.1 命中场景

涉及以下任一内容时，必须命中本节：

1. 参考旧项目样式并在新架构重写
2. 占位 UI 替换
3. 页面迁移分批
4. 迁移质量验收

### 9.2 强制规则

1. 旧项目只可作为视觉语义、信息结构和交互目标的参考来源。
2. 迁移必须在当前仓架构内 clean-room 重写，不得复制旧项目源码、旧依赖耦合和旧样式堆叠写法。
3. 占位 UI 只保留壳层框架职责，不得作为正式 UI 的尺寸、间距、层级和动效基线。
4. 迁移批次必须写明：
   - 旧来源路径
   - 新落点路径
   - 允许修改范围
   - 明确不改范围
   - 保留语义点
   - 剔除问题点
5. 旧视觉目标与现行规则冲突时，优先按现行规则执行。
6. 一个迁移批次只处理一个主主题，不得混入无关页面或无关域改动。

### 9.3 验收要求

1. 迁移后必须复查关键交互态与热区一致性。
2. 必须复查多宽度模式下的结构稳定性。
3. 不得把旧项目 bug、临时补丁和历史技术债原样带入新实现。

## 10. 验收要求

满足以下条件，才算本专题通过：

1. 顶部、底部和主滚动区边界能被稳定描述和复查。
2. 焦点区域在键盘弹起后仍可见，不遮挡关键操作。
3. 可点击元素热区、状态和例外清单可回查。
4. 页面主滚动区唯一且稳定。
5. 小字号区域可读、容器稳定。
6. 毛玻璃在支持和不支持的环境中都可读、可点、可滚。
7. 旧 UI 迁移批次能追溯“旧来源 -> 新落点 -> 保留项 -> 剔除项 -> 验证结果”。

## 11. 禁止事项

1. 禁止把本专题重新拆回页面补丁。
2. 禁止只保留视觉效果，不保留结构和交互边界。
3. 禁止把旧专题法则原文整体复制进新体系。
