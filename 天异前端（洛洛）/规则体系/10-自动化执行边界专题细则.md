# 10-自动化执行边界专题细则

> 文档类型：规则正文
> 状态：active
> 更新时间：2026-04-21
> 说明：本文件用于细化自动化、自动模式、云端任务与 `codex exec` 非交互执行边界；目标是保证自动化能做到已规划尽头，同时不越界扩题。

## 1. 目标

本细则解决四个问题：

1. 什么情况下进入自动化执行。
2. 自动化应该做到哪里。
3. 自动化必须在什么情况下停止。
4. 自动化执行时应如何主动拉取相关规则细则。

## 2. 定义

以下说法在当前仓内等同：

1. 自动化
2. 自动化修复
3. 自动模式
4. 云端任务
5. `codex exec`
6. 非交互执行
7. 用户明确要求“直接做完”“继续做到规划尽头”“遇到问题再停止”

统一定义：

1. 自动化等同于 `codex exec` 非交互执行模式。
2. 自动化不是无限续做授权。
3. 自动化也不是只写计划后停下。
4. 自动化的默认动作是：先确认或补齐计划，然后执行到已规划的尽头。

## 3. 已规划的尽头

“已规划的尽头”只能来自正式计划或本轮补齐的执行边界。

它包括：

1. 当前计划写明的验收标准。
2. 当前计划写明的完成条件。
3. 当前计划写明的固定停止点。
4. 当前计划写明的失败退出条件。
5. 当前批次完成验证、留档回写和 Git 提交后的交付点。

它不包括：

1. 顺手进入新顶层阶段。
2. 顺手新建长期目录。
3. 顺手引入新业务目标。
4. 顺手处理与当前计划无关的 review finding。
5. 顺手恢复或读取 `垃圾桶/` 作为现行依据。

## 4. 自动化开工前置条件

进入自动化前必须满足：

1. 已读根 [AGENTS.md](/H:/工作/天异/uniapp+vue新架构/AGENTS.md)。
2. 已读 [规则体系/README.md](/H:/工作/天异/uniapp+vue新架构/规则体系/README.md)。
3. 已读本细则。
4. 已按“可能关联即拉取”的原则读取相关专题细则。
5. 已有正式计划，或先补齐当前批次计划。
6. 计划内必须写清执行目标、范围、不做什么、停止边界、失败退出条件、完成后汇报要求和已规划的尽头。
7. 已检查 Git 工作树；若存在非本轮 dirty 内容，必须判断是否影响本批，影响则停止。

## 5. 细则调取规则

自动化执行时，规则读取采用“可能关联即拉取”原则。

要求：

1. 不等到确定相关才读细则。
2. 只要任务可能碰到某主题，就必须先拉出对应细则。
3. 执行中范围变化、对象变化或判断依据变化时，应重新拉取对应细则。
4. 如果多个细则可能相关，应同时读取最小必要集合，而不是只读一个最像的文件。
5. 如果细则之间冲突，以更具体、更晚更新、且与当前对象更直接相关的规则为优先；仍无法判断时停止并汇报。

1. 涉及目录、垃圾桶、删除、迁移、长期目录、`src/` 落点：读取 [01-项目基线与目录规则.md](/H:/工作/天异/uniapp+vue新架构/规则体系/01-项目基线与目录规则.md)。

## 6. 自动化允许连续执行的内容

只要在当前计划范围内，自动化应连续执行：

1. 审查与证据收集。
2. 计划补齐。
3. 实现或文档修改。
4. 非破坏性目录转移准备。
5. 验证与失败定位。
6. 本批范围内的修复。
7. 正式留档回写。
8. Git 提交。
9. 提交索引回填。
10. 最终状态复查。

不应因为以下事项机械停下：

1. 单个子步骤完成。
2. 一个文件已经改完。
3. 一个检查首次失败但原因明显属于本批且可修。
4. 需要补交付记录或计划索引。
5. 需要重跑计划内检查。
6. 需要提交已完成且边界清楚的批次。

## 7. 自动化必须停止的条件

命中任一条件必须停止：

1. 已达到当前计划的已规划尽头。
2. 触碰新的顶层阶段或新正式计划。
3. 需要新增长期目录、长期分类或新的垃圾桶落点。
4. 需要用户决定产品方向、接口范围、页面去留、目录归属或提交边界。
5. 发现当前计划遗漏关键对象，继续执行会扩大范围。
6. 发现与现行规则或现行口径冲突。
7. 发现 Git 工作树存在非本轮 dirty 内容且会影响当前批次判断。
8. 发现必须执行物理删除，或用户要求直接删除但未二次确认。
9. 发现正式 API contract 需要新增或改变，但当前计划未纳入接口变更。
10. 发现需要真实后端、真实设备、secret、账号权限或外部服务人工配置才能继续。
11. 发现质量检查失败且无法判断是否由本批引入。
12. 发现修复需要跨越当前批次边界。
13. 发现提交边界已经不清楚。
14. 发现需要读取 `垃圾桶/` 作为现行依据。

## 8. 自动化失败退出条件

自动化失败退出时不得回滚用户改动，不得使用破坏性命令。

失败退出必须保留：

1. 已完成内容。
2. 失败命令。
3. 失败摘要。
4. 当前判断。
5. 未解决阻塞项。
6. 下一步建议。
7. 是否已经形成 Git 提交。

## 9. 自动化完成要求

自动化完成时必须确认：

1. 是否到达已规划的尽头。
2. 是否还有未解决阻塞项。
3. 是否已回写计划、审查或交付记录。
4. 是否已运行计划要求的检查。
5. 是否已形成 Git 提交。
6. 若已提交，是否已回填提交索引。
7. 下一轮建议入口是什么。

## 10. 禁止事项

1. 不允许把自动化理解成只做计划不执行。
2. 不允许把自动化理解成无限续做。
3. 不允许不读相关细则就开始执行。
4. 不允许只读一个总入口就替代专题细则。
5. 不允许在计划未写清停止边界时进入自动化。
6. 不允许在命中停止条件后继续扩题。
7. 不允许用“自动化”为理由跳过留档、验证或提交边界。
