# 03-安全与边界规则

> 文档类型：规则正文  
> 状态：draft  
> 更新时间：2026-04-13  
> 说明：本文件用于统一依赖、外链、敏感信息、缓存、文件和网络边界；先定边界，不提前扩容能力或接入新第三方。

## 1. 目标

本文件解决四个问题：

1. 什么内容不能进仓、不能进日志、不能进正式文档。
2. 哪些输入、外链、第三方回调和后端下发内容默认不可信。
3. 缓存、本地文件、用户域数据应该如何隔离。
4. 第三方依赖、外链资源和壳层能力如何接入才不越界。

## 2. 总原则

安全与边界规则的首要目标固定为“先划清边界，再谈实现方便”。

因此必须满足：

1. 敏感信息不进仓。
2. 不可信输入不直接进页面。
3. 用户态缓存与公共缓存不混用。
4. 第三方接入先审查、后接入、再留档。

## 3. 敏感信息与配置边界

### 3.1 禁止入仓内容

以下内容不得进入仓库：

1. 真实密钥
2. 长期 token
3. 证书原文
4. 生产账号
5. 生产敏感配置
6. 未脱敏的接口返回样本中敏感字段

### 3.2 配置与环境变量规则

1. 配置必须按职责分层，不得散落在页面层。
2. 不允许用“临时环境变量”长期占位正式配置。
3. 正式规则正文只能写变量名和用途，不写真实值。
4. 示例值必须使用脱敏或占位值。

### 3.3 前端边界

1. 前端不得长期保存敏感密钥。
2. 前端不得在页面层散写鉴权逻辑。
3. 任何真正依赖壳层、网关或后端的安全能力，不得伪装成纯前端能力。

## 4. 日志、截图与对外输出边界

1. 日志、错误上报、审查结论、截图和留档内容都必须先脱敏。
2. 任何可能包含地址、账号、token、文件路径、请求头、设备标识的内容，都不得原样外发。
3. 正式文档中的接口示例、调试结果、错误样本必须只保留必要字段。
4. 若仓内已有安全日志包装层，应优先走统一包装层，不继续散写高风险日志调用。

## 5. 不可信输入规则

以下内容一律视为不可信输入：

1. URL
2. 查询参数
3. 外链
4. 富文本
5. 后端下发 HTML
6. 第三方回调
7. 外部文件内容

规则：

1. 不可信输入只能先经过 `service / adapter / port / 明确安全边界` 处理，再进入页面。
2. 页面层不得直接对不可信输入做真值判断。
3. 文档与计划中不得把“不可信输入未处理”描述成可接受默认状态。

## 6. 缓存与本地存储边界

### 6.1 基本原则

1. 本地缓存必须能说明存储内容、保存时机、恢复时机和清理时机。
2. 页面层不得到处直接读写本地缓存。
3. 用户态缓存与公共缓存必须分层。

### 6.2 用户域缓存规则

1. 任何用户态缓存都必须显式绑定用户域。
2. 缺失用户域时，不得隐式回落到 `public`。
3. 用户身份边界变化时，旧用户域缓存必须明确失效或切换。
4. 不允许把“当前调用时的当前用户域”误当成“请求发起时的用户域”。

### 6.3 App 与 H5 边界

1. App 允许有运行时本地持久缓存与本地文件落盘。
2. H5 主要依赖浏览器缓存与 HTTP 缓存。
3. 不允许把 App 缓存语义和 H5 缓存语义混写成同一个概念。

### 6.4 安装包、运行时缓存、线上加载三层分离

必须明确区分：

1. 安装包内置
2. 运行时缓存
3. 线上加载

规则：

1. 不得把“可缓存”误写成“必须预打包进安装包”。
2. 不得把“本地持久缓存”弱化成“只在内存里暂存”。
3. 文档、计划、审批和联调说明中不得混写这三层。

## 7. 本地文件与下载资源边界

1. 本地文件写入、图片落盘、文件映射和清理策略必须有统一入口。
2. 下载资源的本地路径、索引元数据和用户域归属必须可解释、可追溯。
3. 不允许页面层自行分叉新的本地文件读写逻辑。
4. 文件清理优先走显式策略，不做无说明的粗暴全量删除。

## 8. 第三方依赖与外链资源边界

### 8.1 新依赖接入规则

新增第三方依赖前必须先审查：

1. 来源
2. 维护状态
3. 许可证
4. 权限范围
5. 体积与兼容性
6. 替换成本

不允许：

1. 先装包后补审查
2. 先接入后补留档

### 8.2 外链资源规则

当前 H5 入口未接入外部脚本或样式 CDN 资源。

若未来新增第三方脚本、样式、`preload` 或 `modulepreload` 等会参与执行链的外链资源，必须同时满足：

1. 命中白名单
2. 提供 `integrity`
3. 提供 `crossorigin=\"anonymous\"`

普通内容图片链接不在本条外链执行资源治理范围内。

### 8.3 第三方接入边界

1. 任何需要壳层协作的能力，不得只在前端侧偷偷接入。
2. 需要新增权限、宿主能力或调试暴露面的接入，必须先审查再计划。
3. 第三方 SDK、uni_modules、壳能力包装层必须有明确归属，不得散落在页面中直接消费。

## 9. 发布前最小安全检查

提交或发布前至少检查：

1. 敏感信息
2. 调试开关
3. 外链资源
4. 依赖风险
5. 权限边界
6. 留档脱敏状态

相关工具入口以 [02-工程与工具规则.md](/H:/工作/天异/uniapp+vue新架构/规则体系/02-工程与工具规则.md) 为准。

## 10. 禁止事项

1. 禁止把真实密钥、长期 token、证书原文写进仓库。
2. 禁止把未脱敏日志、截图和返回样本直接外发或留档。
3. 禁止页面层到处直接读写本地缓存。
4. 禁止把用户态缓存回落到公共域。
5. 禁止先装包、先接外链、先接能力，再补安全审查。
6. 禁止把不可信输入直接塞进页面真值链。

## 11. 与后续阶段的关系

本文件只定义“哪些边界不能踩”。

后续分工固定为：

1. `G0` 负责把安全与边界规则写清楚。
2. `P1-P4` 任何命中缓存、接口、身份、文件或网络的实施，都必须回指本文件。
3. 具体工具命令、扫描入口和安装方式，以 [02-工程与工具规则.md](/H:/工作/天异/uniapp+vue新架构/规则体系/02-工程与工具规则.md) 为准。
