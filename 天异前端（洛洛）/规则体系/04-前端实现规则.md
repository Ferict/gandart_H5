# 04-前端实现规则

> 文档类型：规则正文  
> 状态：draft  
> 更新时间：2026-04-24
> 说明：本文件用于统一页面、组件、状态、service、adapter 和实现边界；先定边界，不提前做 `src/` 物理重构。

## 1. 目标

本文件解决四个问题：

1. 页面壳与业务实现如何分层，才不再让 `src/` 显得“非常 AI、非常乱”。
2. `feature`、`shared`、`service`、`adapter`、`provider` 各自负责什么。
3. 页面状态、业务状态、缓存状态、接口消费状态如何分层。
4. 什么情况下应该局部收敛，什么情况下才允许抽象复用。

## 2. 总原则

前端实现规则的首要目标固定为“人工可读、人工可维护、职责稳定”。

因此必须满足：

1. 业务语义优先于技术术语。
2. 页面壳尽量薄，业务实现尽量靠近业务域。
3. 共享只共享真正共享的内容，不为抽象而抽象。
4. 一段逻辑的归属，优先由业务范围决定，不由个人偏好决定。
5. UI 展示链与请求链必须彻底解耦，不让组件直接理解 transport 语义。

## 3. 分层模型

长期分层模型固定为：

1. `app`
   - 启动、provider 接线、全局运行时
2. `pages`
   - 路由壳、页面入口、少量页面级 orchestration
3. `features`
   - 单业务域实现
4. `shared`
   - 跨业务共享的组件、工具、基础模型

规则：

1. `pages` 不是长期业务实现承载层。
2. `shared` 不是“找不到地方放就先放这里”的兜底目录。
3. `app` 不是业务逻辑目录。

## 4. 页面壳规则

### 4.1 页面文件负责什么

页面壳文件负责：

1. 路由入口
2. 页面级装配
3. 少量页面级事件转发
4. 页面生命周期 orchestration

### 4.2 页面文件不负责什么

页面壳文件不应长期负责：

1. 大块业务计算
2. 大块缓存逻辑
3. 大块 provider 选择逻辑
4. 大块 DTO 转译逻辑
5. 大块跨页面复用实现

### 4.3 页面私有实现规则

如果一段逻辑只服务于单个页面，优先留在该页面邻近位置：

1. page-local composable
2. page-local helper
3. page-local component

但要求：

1. 这些局部实现必须围绕页面目录收拢。
2. 不允许只因为“技术上能复用”就过早拉到全局。

## 5. Feature 与 Shared 边界

### 5.1 Feature 归属规则

满足以下任一条件，应优先归属到业务域：

1. 只服务于某一个产品域
2. 只服务于某一组页面
3. 依赖该业务域的状态、契约或展示口径

### 5.2 Shared 归属规则

只有同时满足以下条件，才允许进入 `shared`：

1. 被多个业务域稳定复用
2. 不依赖某一个业务域的专属语义
3. 抽出后不会让理解成本变高

### 5.3 不允许的伪共享

以下情况不应进入 `shared`：

1. 只被一个页面或一个业务域使用
2. 只是为了显得“架构更抽象”
3. 抽走后需要不断解释它原本属于哪个业务

## 6. 组件边界规则

### 6.1 展示组件

展示组件应优先满足：

1. 输入稳定
2. 输出明确
3. 少状态或无状态
4. 不直接持有复杂业务运行时

### 6.2 页面装配组件

如果组件承担的是页面装配职责，则应靠近页面或业务域，不应误标成全局共享组件。

### 6.3 组件职责注释

非平凡组件文件应在文件头或等价位置说明：

1. 负责什么
2. 不负责什么
3. 主要依赖哪些输入

## 7. 状态与运行时分层规则

### 7.1 本地 UI 状态

仅影响当前组件或当前页面表现的状态，留在组件或页面附近：

1. 动画状态
2. 展开收起
3. loading 占位展示态
4. 输入框临时值

### 7.2 页面级运行时状态

只影响当前页面链路的运行时状态，优先放在 page-local composable：

1. 页面刷新状态
2. 页面缓存 hydrate 状态
3. 页面请求版本号
4. 页面路由签名

### 7.3 跨页面业务状态

只有跨多个页面稳定复用的状态，才允许进入 feature 层或 store 层。

### 7.4 持久缓存状态

App 级或用户域级缓存状态必须：

1. 明确用户域
2. 明确持久化边界
3. 明确由谁写入
4. 不与临时 UI 状态混用

## 8. Service / Adapter / Provider 边界

### 8.1 Service

`service` 负责：

1. 业务流程协调
2. 缓存与持久化编排
3. 稳定运行时行为

`service` 不负责：

1. 直接渲染页面
2. 承载大量临时 UI 状态
3. 混写后端原始 DTO 兼容细节和页面显示细节

### 8.2 Adapter

`adapter` 负责：

1. 后端 DTO 到前端稳定模型的转译
2. 字段兼容和规范化

`adapter` 不负责：

1. 页面刷新协议
2. 路由逻辑
3. 交互动画状态

### 8.3 Provider

`provider` 负责：

1. `mock / http` 源选择
2. 端口调用接线
3. 运行时 provider 切换

`provider` 不负责：

1. 业务页面逻辑
2. 展示结构
3. 把后端 contract 重新定义一遍

### 8.4 Root 技术目录过渡规则

在 `src/` 真正重构前，继续执行以下边界：

1. `src/services/` 只允许新增真正跨业务、跨页面的运行时服务
2. `src/utils/` 只允许新增真正与业务域无关的工具
3. `src/contracts/`
   - 只承接正式 contract 与 DTO 结构
4. `src/ports/`
   - 只承接跨 provider 的能力边界
5. `src/implementations/`
   - 只承接 provider 实现

不允许：

1. 把页面专属逻辑塞进 root `services/`
2. 把页面专属 helper 塞进 root `utils/`
3. 把业务规则散落到 `contracts / ports / implementations / services` 多层之间而没有明确业务归属

### 8.5 UI 与请求彻底解耦规则

前端当前强制执行以下解耦规则：

1. UI 组件、section 组件、展示壳组件不得直接 import `src/contracts/`、`src/ports/`、`src/implementations/`，也不得直接调用 `uni.request`。
2. UI 层只能接收稳定的 view model、展示状态和事件回调；不得把 `ContentEnvelope`、HTTP 状态码、`requestId`、`etag`、`page/pageSize`、provider 标识直接当成组件 props。
3. 请求参数组装、provider 选择、超时、重试、304/etag、请求版本隔离、失败收口都属于 page runtime、service 或 adapter，不属于模板、展示组件或纯 UI composable。
4. 正式链路固定为：`contract DTO -> adapter 规范化模型 -> runtime / service 业务模型 -> UI view model -> component props`；未经适配的后端 DTO 不得直接下放到组件层。
5. 页面需要 `refresh / retry / load-more / search / filter` 时，UI 只负责表达用户意图；是否发请求、发哪条请求、如何并发控制、如何收口，统一由上层运行时决定。
6. 页面壳可以协调请求触发，但不得在模板层或展示组件层夹带 transport-only 字段。
7. 任何以技术层命名的长期目录如果只剩 `README.md`、`.gitkeep` 之类空壳，不得长期以“名义分层”存在；必须在正式计划里明确是填实、并入现有层，还是转移到 `垃圾桶/`。

### 8.6 AI UI 转化接轨规则

凡属当前仓 UI 转化、外部设计落地、AI UI Refresh、页面壳换接、组件映射等任务，固定执行以下规则：

1. 必须同时回链当前专题唯一总规划和对应现行口径；就 AI UI Refresh 而言，默认回链 [2026-04-24-P11.10-AI-UI-refresh终极版总规划-视觉冻结版.md](/H:/工作/天异/uniapp+vue新架构/留档/计划记录/2026-04-24-P11.10-AI-UI-refresh终极版总规划-视觉冻结版.md) 与 [规则体系/口径/04-AI-UI-refresh专项治理现行口径.md](/H:/工作/天异/uniapp+vue新架构/规则体系/口径/04-AI-UI-refresh专项治理现行口径.md)。
2. 在现行总规划未明确开放真实视觉刷新前，UI 转化默认按“零视觉差结构治理”理解，不得把“转化 UI”自动扩大解释为自由改视觉。
3. 外来设计稿、截图、提示词、第三方 pipeline 只能映射为当前仓对象模型：`page-shell / panel / section / shared component / global component`；不得把旧项目页面树、老组件库命名和旧接口组织方式直接带入当前仓。
4. 不得为 UI 转化另起第二套现行载体，如顶层 `docs/`、平行 skill 草案目录或独立“当前规则说明”；现行规则留在 `规则体系/`，现行口径留在 `规则体系/口径/`，计划留在 `留档/计划记录/`。
5. 如外来 pipeline 与当前仓规则冲突，以当前仓正式请求链、当前专题唯一总规划和现行口径为准；旧准备批、旧计划链和历史说明只可作为迁移输入，不得回升为并列口径。

### 8.7 UI 转化可复用执行骨架

当前仓后续 UI 转化默认按以下骨架执行，后续若沉淀为 skill，也只能从这套骨架派生：

1. 先确认本轮目标对象、来源素材、允许变化边界，明确本轮到底是“零视觉差结构治理”还是未来单独批准的真实视觉刷新。
2. 先把目标归类到 `page-shell / panel / section / shared component / global component`，再决定写入位置；不允许跳过对象归类直接开始铺文件。
3. 先划定写入集合与锁定集合，页面私有实现优先 page-local；共享壳、规则、留档和敏感路径默认不顺手改。
4. 先搭展示结构、交互壳和稳定输入输出；UI 只消费前端稳定模型，不消费 transport-only 字段。
5. 当正式接口契约或新版后端契约尚未到位时，只允许使用 page-local placeholder、稳定 view model 或受控 mock 承接展示；不得把临时占位结构升格为正式 contract。
6. 所有未接上的后端缺口，统一停在 `adapter / runtime / service` 输入侧；不得为了少写一层把 DTO、HTTP 包壳或 provider 细节直接打进 UI。
7. 每轮 UI 转化完成后，都必须补最小验证、回链现行口径和留档；若未来要做成 skill，skill 只能复用这套顺序，不得再发明第二套执行流程。

## 9. 抽象与复用规则

### 9.1 优先收敛，后做复用

默认顺序固定为：

1. 先在局部把职责收清
2. 再观察是否真的跨域复用
3. 最后再抽到共享层

### 9.2 允许抽象的前提

只有满足以下条件，才建议抽象：

1. 至少有两个稳定使用点
2. 语义一致
3. 抽象后总复杂度下降

### 9.3 不允许的抽象

1. 仅为减少文件数量做抽象
2. 仅为看起来更“架构化”做抽象
3. 抽象后仍需大量上下文解释才能读懂

## 10. 命名与组织规则

1. 名称优先表达业务职责，不优先表达技术炫技词。
2. 文件名应让人一眼知道：
   - 这是页面壳
   - 这是页面私有 composable
   - 这是业务域 service
   - 这是共享组件
3. 同一目录下避免同时混入多种不相干语义。
4. 目录结构要服务于人工查找，不服务于“理论分层完整”。

## 12. 禁止事项

1. 不允许继续把页面巨石文件视为常态。
2. 不允许为了抽象而抽象。
3. 不允许把业务专属内容伪装成共享内容。
4. 不允许让 root 技术目录继续无限长大。
5. 不允许在没有规则的情况下先做大规模 `src/` 搬运。

## 13. 与后续阶段的关系

本文件只定义“前端实现应该如何分层和收口”。

后续分工固定为：

1. `G0` 负责把实现边界写清楚。
2. 后续单独批次负责按规则治理 `src/` 物理结构。
3. 任何 `src/` 目录重构都应先回指本文件，而不是边改边发明新口径。
