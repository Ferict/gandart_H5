# 09-内容域运行时与缓存专题细则

> 文档类型：规则正文

4. App 安装包、运行时落盘缓存、H5 线上加载三层资源语义如何统一。
5. `profile` 用户态内容的缓存写入、用户域切换与 request scope 绑定如何统一。

## 3. 契约、Provider 与前端接线边界

### 3.1 命中场景

涉及以下任一内容时，必须命中本节：

1. `ContentPort`
2. `VITE_CONTENT_PROVIDER`
3. `VITE_CONTENT_API_BASE_URL`
4. `mock / http` provider 切换
5. 前端 query 快照键与 adapter 参数编码

### 3.2 强制规则

1. `ContentPort` 是内容域唯一 provider seam。
2. provider 的选择、配置与运行时切换必须明确，不得隐式回落或隐式替换。
3. `VITE_CONTENT_API_BASE_URL` 只在 `http` provider 下使用。
4. 前端 query 对象允许作为仓内运行时结构存在，但不属于后端正式契约正文。
5. adapter 负责把前端稳定 query 编码成正式接口参数，不得反向让后端契约适配页面私有字段。
6. query 快照键必须按完整查询条件构造，不得省略会影响 `etag / notModified / request-signature` 的字段。
7. provider、刷新协议、页面等待链、用户域与 cache key 只属于前端运行时规则，不得继续写回正式接口正文。

### 3.3 禁止事项

1. 禁止用 `mock` provider 证明“后端可直接接入”。
2. 禁止把页面运行时字段补写成正式后端字段。
3. 禁止在多个目录散写新的 API 正文或运行时 contract 说明。

## 4. 刷新协议与更新生效分层

### 4.1 命中场景

涉及以下任一内容时，必须命中本节：

1. `refreshContent(options?)`
2. `waitForRefreshPresentation()`
3. 前台恢复
4. 静默轮询
5. `pull-refresh`
6. `page-open`

### 4.2 强制规则

1. 页面对统一壳层只允许暴露：
   - `refreshContent(options?)`
   - `waitForRefreshPresentation(): Promise<void>`
2. 壳层刷新链固定为：先调用页面刷新，再等待页面 presentation settled，最后结束壳层刷新反馈。
3. 更新生效必须拆成三层：
   - 内容数据更新
   - 前端页面代码更新
   - App 壳与原生能力更新
4. 内容数据更新的触发源和更新等级必须按页面类型显式声明，不得统一宣称“全页面实时更新”。
5. 更新是否真正生效，必须结合 `version / updatedAt / signature` 与 `etag / notModified` 等运行时信号判断。
6. 旧请求晚到时，不得覆盖当前页面内容，也不得写回当前缓存域。
7. 详情页结果应用必须至少受以下条件共同约束：
   - 请求版本仍有效
   - 路由签名仍一致
   - 页面仍 active
   - request scope 仍可被正确持久化

### 4.3 禁止事项

1. 禁止页面自行定义一套脱离壳层的刷新完成标准。
2. 禁止把前端代码热更新误写成“内容数据更新已生效”。
3. 禁止把某一页面当前阶段的触发源误写成长期不变规则。

## 5. 内容卡片列表加载、分页与图片反馈

### 5.1 命中场景

涉及以下任一内容时，必须命中本节：

1. 首屏加载
2. 分页加载
3. 图片加载与失败反馈
4. footer / endline
5. overpull / 回弹
6. 窗口化挂载与页面运行时分页缓存

### 5.2 强制规则

1. 列表至少拆分三类状态：
   - 首屏加载
   - 分页加载
   - 图片加载
2. 首屏失败、首屏空态、分页失败、无更多必须分层，不得共用同一出口。
3. 同一列表键下的分页请求必须采用单飞行模型。
4. 分页触发必须保留节流冷却；搜索与条件切换必须保留独立 debounce / throttle 策略。
5. 查询条件变化时，必须生成新的列表键，并重置活动请求代次、窗口和分页状态。
6. 旧列表键或旧请求代次返回的结果，不得污染当前结果集。
7. 图片请求随卡片挂载自然开始，不再把隐藏预热当主加载链。
8. 图片加载占位与分页 loading、首屏 loading 必须分开。
9. 图片失败只处理图片层，不得把整张卡误判为列表失败。
10. footer 必须常驻，不得在“有更多 / 分页失败 / 无更多”之间切换为完全无反馈。
11. `hasMore = true` 与 `hasMore = false` 的反馈语义必须清晰区分，不得把无更多伪装成分页失败或首屏空态。
12. 后端正式分页契约与本地窗口、缓冲、显示批量必须分层描述，不得混写。

### 5.3 禁止事项

1. 禁止把内容卡片列表重新做回“全量常驻挂载 + 全量图片一起请求”。
2. 禁止把分页失败、图片失败和空态混成一个状态出口。
3. 禁止因 overpull 或连续滑动重复触发新分页请求。

## 6. 资源交付、启动链与缓存层级

### 6.1 命中场景

涉及以下任一内容时，必须命中本节：

1. App 安装包内置资源
2. App 运行时落盘缓存
3. H5 线上加载
4. 页面启动顺序
5. 图片本地文件缓存
6. 缓存清理与失效

### 6.2 强制规则

1. 安装包内置资源只允许承载稳定静态资源与低频默认数据，不包含真实运营内容图与真实动态列表数据。
2. App 首版起必须具备运行时持久缓存，不得只依赖内存缓存。
3. App 页面启动优先级固定为：
   - 安装包骨架
   - 本地持久缓存
   - 网络增量更新
4. 运行时持久缓存至少覆盖：
   - 列表分页数据
   - 详情基础快照
   - 增量复用元数据
   - 已成功获取的图片资源
5. H5 不按 App 安装包口径描述；H5 真实内容仍以线上资源和浏览器缓存为主。
6. 缓存长期保留不等于内存常驻；页面挂载窗口、运行时内存和持久缓存必须分层。
7. 缓存清理必须可解释、可追溯，不得把存储压力处理成全量粗暴删除。
8. 资源交付说明必须明确区分：
   - 安装包内置
   - 运行时缓存
   - 线上加载

### 6.3 禁止事项

1. 禁止把“App 有本地缓存”表述成“所有真实内容都预打包进安装包”。
2. 禁止把 H5 线上加载链按 App 安装包路径描述。
3. 禁止混写安装包资源、运行时缓存和线上真实来源。

## 7. Profile 用户域、request scope 与持久化边界

### 7.1 命中场景

涉及以下任一内容时，必须命中本节：

1. `profile scene`
2. `profile asset list`
3. `profile asset detail`
4. 用户域切换
5. 登录态变化
6. `requestUserScope`
7. `persistent scope`

### 7.2 强制规则

1. `profile` 用户态缓存只允许通过统一 wrapper 读写，不得绕过 wrapper 直接命中底层通用缓存服务。
2. `profile` 相关持久化 wrapper 必须显式接收 `userScope`，不得在内部重新读取“当前 scope”代替请求 scope。
3. `requestUserScope` 为空时，`profile` 场景的持久化必须 `no-op`，不得回落到 `public`。
4. `profile scene / asset list / detail snapshot` 一旦带着 `requestUserScope` 发出请求，最终持久化必须写入该 `requestUserScope`。
5. 用户域切换时，必须先处理旧用户域的 profile 缓存清理，再切入新域。
6. 详情页结果应用顺序固定为：
   - 守卫判定通过
   - 写页面内容
   - 同步页面持久态 scope
   - 按 `requestUserScope` 显式写 detail snapshot
7. 个人页列表与 `coordinator` 预取链同样必须按请求发起时捕获的 `requestUserScope` 写缓存。
8. 旧请求晚到时，即使当前 scope 已变化，也不得把旧结果写进新域。
9. `profile` 图片缓存必须采用显式 `required-user` 策略；缺失用户域时直接跳过，不得命中公共域。

### 7.3 禁止事项

1. 禁止把 guard 判定建立在带副作用的 scope 读取函数上。
2. 禁止在 `profile` 用户态链路上隐式回落 `public`。
3. 禁止用“持久化发生时的当前 scope”替代“请求发起时的 request scope”。
