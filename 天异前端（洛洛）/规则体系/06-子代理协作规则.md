# 06-子代理协作规则

> 文档类型：规则正文  
> 状态：draft  
> 更新时间：2026-04-13  
> 说明：本文件用于统一子代理启用条件、ownership、回收与留痕要求；子代理是常用能力，但按需使用，不机械默认开启。

## 1. 目标

本文件解决五个问题：

1. 何时应该启用子代理，何时不该启用。
2. 子代理如何分工，才不会把主线切碎。
3. 模型如何按任务复杂度选择。
4. 如何防止同轮多代理争用同一文件。
5. 如何做到子代理行为可追溯、可回收、可复核。

## 2. 总原则

当前子代理协作原则固定为：

1. 子代理是常用提效能力，但按需使用。
2. 子代理更多时候是即用即弃，不常驻。
3. 主控永远保留关键路径判断、文件所有权划分和最终收口责任。
4. 不为并行而并行，不为省事而外包关键判断。

## 3. 角色模型

### 3.1 主控

每轮有且只有一个主控。

主控负责：

1. 判断是否启用子代理
2. 划分任务
3. 划分文件所有权
4. 处理关键阻塞判断
5. 复核子代理结果
6. 统一对外回复与最终收口

### 3.2 任务子代理

任务子代理的默认形态是：

1. 即用即弃
2. 任务单一
3. 范围明确
4. 完成后关闭

### 3.3 治理/审查子代理

当任务命中下列内容时，可启用专用治理或审查子代理：

1. 规则整理
2. 留档收口
3. 只读复查
4. 风险核对

默认要求：

1. 治理/审查子代理优先只读。
2. 治理/审查子代理不得代替任务实现子代理修改业务逻辑，除非被主控明确授权。

## 4. 启用条件

满足以下任一条件时，建议考虑启用子代理：

1. 任务可以清晰切成多个互不阻塞的子问题
2. 存在明显可并行的只读探索工作
3. 需要在主线推进时并行做验证或审查
4. 存在边界清晰、写入文件集合互不重叠的实现切片

## 5. 不启用条件

满足以下任一条件时，不建议启用子代理：

1. 任务很小，主控本地完成更快
2. 规格不清，连主控自己都还没定边界
3. 下一步动作立即依赖该结果，主控本地做更稳
4. 文件写入范围高度重叠
5. 并行不会带来明确收益

## 6. 模型选择规则

模型选择按“能力等级”而不是按固定名字死写。

当前可用模型若包含类似：

1. `gpt-5.4`
2. `gpt-5.4-mini`
3. `gpt-5.3-codex`
4. `gpt-5.3-codex-spark`

则优先按以下口径选型：

1. 复杂主线、跨链路实现、高风险规则治理、关键结构收口
   - 选最强主力模型
2. 边界清晰的代码实现子任务
   - 选强 coding 模型
3. 只读探索、快速检索、低风险整理
   - 选更快的 mini / explorer 类模型

规则：

1. 模型能力不够时，优先升级模型，不优先继续切碎任务。
2. 不允许让明显偏弱的模型静默接管复杂主线。

## 7. 任务派发规则包

主控下发给子代理的规则包至少要包含：

1. 任务目标
2. 当前背景
3. 必须遵守的现行规则
4. 文件所有权
5. 禁止范围
6. 验收标准
7. 回传格式要求

若任务涉及代码写入，还必须写清：

1. 允许修改哪些文件
2. 不允许改哪些文件
3. 是否需要补测试

## 8. 文件所有权与冲突避免

1. 同一文件同一轮只允许一个写入代理。
2. 审查代理、安全代理默认只读。
3. 主控不得把同一文件同时派给多个写入代理。
4. 如果多个子任务天然共享同一写入集合，应改为主控本地处理或重新切片。

## 9. 执行与等待规则

1. 派出子代理后，主控应继续做不重叠的本地工作。
2. 只有在下一步明确被阻塞时，才等待子代理结果。
3. 不允许机械频繁轮询等待。
4. 子代理失败或超时后，主控要么重派，要么本地接手，不得悬空。

## 10. 回传与复核规则

子代理回传至少应包含：

1. 完成项
2. 改动文件
3. 发现的问题
4. 风险与待确认项
5. 已做验证或未做验证

规则：

1. 主控必须复核回传结果。
2. 子代理结论未经主控整合，不得直接作为最终结论对外宣布。
3. 子代理返回的写入结果，主控要先看边界和风险，再决定是否继续整合。

## 11. 回收与留痕规则

1. 子代理完成并整合后，应及时关闭。
2. 不允许把任务子代理长期挂着不回收。
3. 只要本轮启用了有实质工作的子代理，就应在正式记录中保留最小追溯信息：
   - 启用时间
   - 任务目标
   - 模型/能力等级
   - 文件所有权
   - 回传结果摘要
   - 关闭状态

留痕位置以 [07-文档与留档规则.md](/H:/工作/天异/uniapp+vue新架构/规则体系/07-文档与留档规则.md) 为准。

## 12. 禁止事项

1. 禁止让子代理自行定义规则口径。
2. 禁止多个写入代理在同一轮争用同一文件。
3. 禁止把任务子代理常驻化。
4. 禁止只派实现位、不保留主控复核责任。
5. 禁止为了并行而并行。

## 13. 与后续阶段的关系

本文件只定义“子代理如何协作”。

后续分工固定为：

1. `G0` 负责把子代理规则写清楚。
2. 后续实际批次若启用子代理，应按本文件派发和留痕。
3. 历史上旧的子代理台账与治理模式只作迁移参考，不继续作为现行唯一入口。
