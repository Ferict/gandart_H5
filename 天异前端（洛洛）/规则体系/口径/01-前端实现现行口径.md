## 2. 适用范围

当前正式适用：

1. `src/` 现有前端实现结构
2. 页面壳与 page-local composable / helper / component 的当前收口方式
3. `home`、`profile-asset-detail` 等已完成局部结构治理的页面链路
4. `src/services`、`src/models`、`tests/unit` 等已完成内部重分组的目录

当前不扩展：

1. 新的全局状态模型
2. 新的 `uni_modules` 专项接入口径
3. 新的 `uni-app` 页面与配置专题口径
4. 未进入正式规则前的大规模 `src/` 再搬迁

## 3. 当前有效口径

### 3.1 当前 `src/` 仍处于过渡期结构

1. 当前 `src/` 还保留 root 技术目录，但已按目录说明和内部分组治理。
2. 当前不把这种 root 技术目录形态认定为长期理想终态。
3. 在真正的 feature 级重构发生前，新增实现仍优先遵守“业务归属优先、页面邻近优先”的写入规则。

### 3.2 当前页面壳收口方式

1. 页面壳继续承担路由入口、页面级装配和少量生命周期 orchestration。
2. 单页面专属运行时优先下沉到 page-local composable。
3. 单页面专属展示结构优先下沉到 page-local component。
4. 只有跨页面、跨业务稳定复用的运行时才允许进入 root `services/` 或共享层。

### 3.3 当前目录热区与过渡写法

1. `src/services/`、`src/models/`、`tests/unit/` 已完成内部职责分组，后续应沿分组继续写入，不回退到平铺。
2. `src/pages/home/composables/` 已按 `home / activity / profile / shared` 分组，新增首页页面私有实现应优先落在对应子目录。
3. `profile-asset-detail` 这类已完成 page-local 收口的页面，不再把新逻辑塞回页面壳主文件。

### 3.4 当前未完全专题化的三类事项

以下三类目前已有规则边界，但还没有完全重写成独立细则，只能按当前口径执行：

1. `uni-app` 页面与配置治理
   - 当前按目录规则与页面壳规则执行，不另开旧式旁路口径。
2. `uni_modules` 与第三方包接入
   - 当前按工程工具规则、安全边界规则、前端实现规则执行。
   - 未经专项规则补充，不扩写新的 `uni_modules` 专属惯例。
3. 状态管理专题
   - 当前按“组件局部状态 / 页面运行时状态 / 跨页面业务状态 / 持久缓存状态”四层执行。
   - 不默认引入新的全局 store 方案，也不隐式改变现有状态归属规则。

### 3.5 当前仍保留的次级页旁路线

1. `src/services/page-entry/` 当前仍保留 `updating`、`notice-detail` 等次级页占位与 construction 链路。
2. 这条链路当前属于“现行兼容旁路线”，不是废弃死链，也不是新业务默认入口。
3. 在次级页完全接入正式内容链之前，这条旁路线继续保留。
4. 新的次级页若无明确规则补充，不得继续沿这条旁路线扩写新的占位型实现。

## 4. 当前明确不扩展范围

1. 不把当前 root 技术目录继续视为可无限扩张的长期结构。
2. 不新增新的 `src/` 顶层技术目录。
3. 不把页面专属逻辑重新塞回 root `services/`、root `utils/`。
4. 不在没有新规则的前提下引入新的状态管理模型或新的 `uni_modules` 接入支线。

## 5. 何时必须同步更新本口径

命中以下任一情况时，必须同步更新本文件：

1. `src/` 顶层结构或分组蓝图发生实质变化。
2. 页面壳与 page-local 实现边界发生变化。
3. `uni-app / uni_modules / 状态管理` 任一专题被正式重建为独立规则或独立口径。
4. 当前“不扩展范围”被突破或被正式解除。
