# 1. 审查目标

本次只审 `P11.3` 的实施边界：

1. 前端消费链目前到底还有哪些“第二真值”。
2. 哪些冲突应在 `P11.3` 解决。
3. 哪些冲突必须留到 `P11.4 / P11.5`，不能提前混做。

# 2. 当前结论

## 2.1 `content-api.contract.ts` 已经是最强真值候选

当前 [content-api.contract.ts](/H:/工作/天异/uniapp+vue新架构/src/contracts/content-api.contract.ts) 已经集中持有：

1. `ContentSceneRequestDto`
2. `ContentResourceRequestDto`
3. `ContentMarketListRequestDto`
4. `NoticeReadActionRequestDto`
5. `ServiceReminderConsumeActionRequestDto`
6. `ContentEnvelope<T>`
7. 正式场景、资源类型、列表资源类型等基础枚举

所以前端消费链统一时，不需要再发明新的 contract 真值位置。

## 2.2 `content.http.ts` 仍保留第二套 endpoint 真值

当前 [content.http.ts](/H:/工作/天异/uniapp+vue新架构/src/implementations/content.http.ts) 仍直接硬编码：

1. `/api/content/scene`
2. `/api/content/resource`
3. `/api/content/list`
4. `/api/content/action/notice-read`
5. `/api/content/action/service-reminder-consume`

这意味着即使 `content-api.contract.ts` 完成单一真值源冻结，前端运行时代码仍然会保留第二套“真实接口路径真值”。

判定：

1. 这是 `P11.3` 必修项。

## 2.3 `content.port.ts` 与 `content.service.ts` 当前主要问题不是路径，而是上位说明

[content.port.ts](/H:/工作/天异/uniapp+vue新架构/src/ports/content.port.ts) 和 [content.service.ts](/H:/工作/天异/uniapp+vue新架构/src/services/content/content.service.ts) 目前没有重复硬编码 5 个 path。

它们的主要风险在于：

1. 仍可能继续把接口边界解释散落到注释和调用语义里。
2. 若 `content.http.ts` 保留第二真值，`port/service` 层仍无法真正只依赖单一真值源。

判定：

1. `P11.3` 需要回写注释和引用方式。
2. 但这里不是最主要的实施成本。

## 2.4 `scripts/check-content-http-link.mjs` 仍是第二套 endpoint 矩阵，但不应在 `P11.3` 处理

[check-content-http-link.mjs](/H:/工作/天异/uniapp+vue新架构/scripts/check-content-http-link.mjs) 当前维护了自己的 5 接口验证矩阵。

同时，当前 `package.json` 只有：

1. `node ./scripts/check-content-http-link.mjs`

没有：

1. `tsx`
2. `ts-node`
3. 其他直接让 Node 运行 TS 合同文件的正式工具链

所以脚本要想直接消费 [content-api.contract.ts](/H:/工作/天异/uniapp+vue新架构/src/contracts/content-api.contract.ts)，必须额外解决“运行时如何读取 TS 真值源”。

判定：

1. 这是 `P11.5` 的实施对象。
2. 不应为了脚本联动而把 `P11.3` 拖成“又改前端链、又改验证链、又改 Node 运行方式”的混合批次。

## 2.5 当前路径 API 文档仍在指导后端阅读多入口，但这属于 `P11.4`

当前活引用仍明显存在于：

1. [backend-handoff/README.md](/H:/工作/天异/uniapp+vue新架构/backend-handoff/README.md)
2. [backend-handoff/后端移交内容清单.md](/H:/工作/天异/uniapp+vue新架构/backend-handoff/后端移交内容清单.md)
3. [backend-handoff/后端联调环境与启动说明.md](/H:/工作/天异/uniapp+vue新架构/backend-handoff/后端联调环境与启动说明.md)
4. [接口契约/README.md](/H:/工作/天异/uniapp+vue新架构/接口契约/README.md)
5. [规则体系/05-接口与联调规则.md](/H:/工作/天异/uniapp+vue新架构/规则体系/05-接口与联调规则.md)

这些引用当前仍把后端阅读路径指向：

1. `统一内容接口契约.md`
2. `内容域后端统一移交说明.md`
3. `接口契约/`

判定：

1. 这是 `P11.4` 的对象。
2. `P11.3` 不应顺手改文档入口，否则代码实施批次会和当前路径清洁收口混在一起。

# 3. `P11.3` 必须解决的冲突

`P11.3` 只解决下面 4 件事：

1. `content.http.ts` 彻底改成从单一真值源读取 5 个正式 endpoint。
2. `content.port.ts` 和 `content.service.ts` 的职责说明回到单一真值源，不再隐含第二语义。
3. `src/` 内其余直接依赖正式 DTO 的消费链，只认 `content-api.contract.ts` 为上位定义。
4. 为 `P11.5` 保留脚本联动点，但不在本批强行解决 Node 侧读取 TS 的问题。

# 4. `P11.3` 明确不做

1. 不改 `backend-handoff/` 的正式阅读顺序。
2. 不删 `接口契约/`、`统一内容接口契约.md`、`内容域后端统一移交说明.md`。
3. 不改 `scripts/check-content-http-link.mjs` 的运行方式。
4. 不引入新的 API 顶层目录。
5. 不为脚本真值联动新增 `tsx / ts-node` 之类新依赖。

# 5. 审查结论

`P11.3` 的正确边界已经明确：

1. 它是“前端运行时代码统一到单一 API 真值源”的实施批次。
2. 它不承担文档退场，也不承担脚本运行时真值联动。
3. 只要把 `content.http.ts` 的 5 个 path 真值收回 contract 文件，本批就完成了最关键的一刀。
