## 3. 详细事实

### 3.1 当前现行后端接入主链仍只有内容域一套

当前已确认的现行后端接入主链是：

1. [src/contracts/content-api.contract.ts](/H:/工作/天异/uniapp+vue新架构/src/contracts/content-api.contract.ts)
2. [src/ports/content.port.ts](/H:/工作/天异/uniapp+vue新架构/src/ports/content.port.ts)
3. [src/implementations/content.http.ts](/H:/工作/天异/uniapp+vue新架构/src/implementations/content.http.ts)
4. [src/implementations/content.mock.ts](/H:/工作/天异/uniapp+vue新架构/src/implementations/content.mock.ts)
5. [src/services/content/content.service.ts](/H:/工作/天异/uniapp+vue新架构/src/services/content/content.service.ts)

证据：

1. [src/main.ts](/H:/工作/天异/uniapp+vue新架构/src/main.ts) 只在应用启动时装配这一套 `mock / http` content provider。
2. `home-rail`、`home-shell`、`profile-asset-detail` 等现行活跃服务都通过 `resolveContentScene / resolveContentListWithMeta / resolveContentResource / runContentAction` 取数。
3. 当前正式 handoff、联调说明和验证脚本都只回到这一个 contract 真值源。

结论：

1. 当前仓中没有发现第二套“已经进入现行后端接入面”的正式业务域接口体系。
2. 自动任务不应在 `P11.6` 里顺手发明新的正式 API 目录或第二份正式 contract 文件。

### 3.2 当前内容域的伪数据大体已经挂在同一 provider 接口上

当前内容域 `mock` 的挂接方式是：

1. [src/implementations/content.mock.ts](/H:/工作/天异/uniapp+vue新架构/src/implementations/content.mock.ts) 实现同一个 `ContentPort`
2. [src/services/content/content.service.ts](/H:/工作/天异/uniapp+vue新架构/src/services/content/content.service.ts) 用统一 `activeContentPort` 消费 `mock / http`
3. [src/main.ts](/H:/工作/天异/uniapp+vue新架构/src/main.ts) 用 `VITE_CONTENT_PROVIDER` 切换 provider

结论：

1. “正式 content API + 额外挂 mock provider” 这一主结构已经成立。
2. `scene / resource / list / action` 的伪数据并不是散落在页面里直接取的。

### 3.3 仍有两处 mock 快照绕过了统一 provider 接口

当前仍直接从 mock 实现读取快照的地方有：

1. [src/services/content/contentResourceDetail.service.ts](/H:/工作/天异/uniapp+vue新架构/src/services/content/contentResourceDetail.service.ts)
2. [src/services/profile-asset-detail/profileAssetDetailContent.service.ts](/H:/工作/天异/uniapp+vue新架构/src/services/profile-asset-detail/profileAssetDetailContent.service.ts)

它们目前直接导入：

1. `createContentResourceSnapshot`

问题：

1. 这类快照并不经过 `ContentPort`
2. 它们会让“伪数据直接额外挂在正式接口上”的口径出现例外
3. 在 `http` provider 场景下，这些 fallback 仍暗含对 mock 数据实现体的直连

结论：

1. `P11.6` 的 API 实源覆盖，不是再新建接口，而是收掉这种直连 mock 实现体的绕行口
2. 需要把 fallback shell 改成：
   - 要么仅保留纯本地最小壳
   - 要么回收到统一 content 接入链的可解释边界

### 3.4 `page-entry / pageContent` 是冻结兼容旁路线，不是当前正式后端 API 面

当前还有一条与内容域正式接口并行的兼容旁路线：

1. [src/ports/pageContent.port.ts](/H:/工作/天异/uniapp+vue新架构/src/ports/pageContent.port.ts)
2. [src/adapters/pageContent.mock.adapter.ts](/H:/工作/天异/uniapp+vue新架构/src/adapters/pageContent.mock.adapter.ts)
3. [src/services/page-entry/pagePlaceholderContent.service.ts](/H:/工作/天异/uniapp+vue新架构/src/services/page-entry/pagePlaceholderContent.service.ts)

证据：

1. [src/services/page-entry/README.md](/H:/工作/天异/uniapp+vue新架构/src/services/page-entry/README.md) 已明确写成“冻结过渡锚点”
2. [src/pages.json](/H:/工作/天异/uniapp+vue新架构/src/pages.json) 中 `settings / service-entry / action-entry / content-resource / profile-address / profile-assets / notice-detail / updating` 仍注册，但这些目录说明都写明自己不是“真实页面实现”
3. 这些页面当前只做 route redirect 或 construction placeholder，不承接真实内容链

结论：

1. 这条链当前属于“现行兼容旁路线”，不是“当前正式后端 API 面”
2. `P11.6` 不应在没有真实页面接入目标的前提下，为这些 construction 页面强造新后端接口
3. 这条链在自动任务里应先做 `keep / freeze / trash` 分类依据补齐；是否并轨到正式 content API，留到后续真实页面实现批次

### 3.5 `contentResourceDetail.service` 已 formal，但当前没有活跃页面消费

[src/services/content/contentResourceDetail.service.ts](/H:/工作/天异/uniapp+vue新架构/src/services/content/contentResourceDetail.service.ts) 当前已经通过 `resolveContentResource` 对齐正式 content contract，但仓内没有现行页面直接消费它。

对应现状：

1. `content-resource` 页面当前仍是 construction redirect
2. `contentResourceDetail.service` 更像是已经预留好的正式 detail 适配层

结论：

1. 它不是“缺正式 API”的对象
2. 它属于“正式资源 detail 适配层已备好，但页面壳仍未切到真实实现”的对象

## 4. 审查结论

本轮结论固定为：

1. 当前真正“应与后端对接”的现行代码主链仍只有内容域一套，不存在第二套当前正式业务接口体系。
2. 当前内容域的 `mock / http` provider 大体已经挂在同一接口上，但还有两处 detail shell 直接读取 mock 快照，必须在 `P11.6` 收口。
3. `page-entry / pageContent` 及其对应 construction 页面当前仍是冻结兼容旁路线，不应在 `P11.6` 中误判为“马上要补正式后端接口的活页面”。
4. `P11.6` 的正确顺序应是：
   - 先收 API 实源覆盖缺口
   - 再清当前路径冻结壳和生成物目录
   - 最后进 `P11.7` 做收口复核

## 5. 对后续实施的硬约束

1. 不新建新的长期顶层目录。
2. 不新发明第二份正式 API 真值源。
3. 清理动作一律不得删除，只能转入 `垃圾桶/`。
4. `P11.6` 只处理当前已命中的代码/API/目录对象，不顺手扩做新的真实页面功能开发。
