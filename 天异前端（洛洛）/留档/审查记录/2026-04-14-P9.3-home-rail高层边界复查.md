## 1. 复查对象

1. `src/services/home-rail/`
2. `src/pages/home/`
3. `src/models/home-rail/`

## 2. 复查事实

### 2.1 pages/home 对 home-rail 依赖广泛

首页三条 rail 的页面组件和 page-local composable 大量直接消费：

1. `homeRail*Content.service.ts`
2. `homeRailResultWindow.service.ts`
3. `homeRailResultMountWindow.service.ts`
4. `homeRailPageReloadPolicy.service.ts`
5. `homeRailPersistentCacheIntegration.service.ts`
6. `homeRailUpdateCoordinator.service.ts`
7. `homeRailNavigation.service.ts`
8. `models/home-rail/*`

这说明 `home-rail` 不是单页私有实现，而是已经形成稳定的领域级共享 runtime 锚点。

### 2.2 home-rail 没有反向依赖 page-local 实现

复查 `src/services/home-rail/` 后，只发现 `homeRailNavigation.service.ts` 生成 `pages/home/index` 路由 URL，没有发现本目录反向依赖 `src/pages/home/` 下的 page-local component/composable。当前边界方向仍然正确。

### 2.3 当前不适合做物理迁移

如果本批直接把 `home-rail` 再拆散回 `pages/home/*`，会同时引入三类风险：

1. 三条 rail 的共享 runtime 会被重新复制到 page-local 层。
2. `home / profile / activity` 当前已经对称化的运行时结构会退回到页面私有散点状态。
3. 本批收益主要只是目录“看起来更近”，但会破坏已经建立好的稳定锚点。

## 3. 结论

`src/services/home-rail/` 当前应正式定性为：

1. 首页三条 rail 的领域级共享 runtime 锚点。
2. 可以继续被 `src/pages/home/` page-local runtime 消费。
3. 不应回流页面私有展示层、模板适配或单一 panel 私有装配。

本批不做物理迁移，只做目录说明回写和高层边界冻结。
