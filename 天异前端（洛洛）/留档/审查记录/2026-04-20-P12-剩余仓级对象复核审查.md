## 5. 已确认问题

### 5.1 AI 工具配置仍引用退役旧规则入口

`promptfoo/promptfooconfig.yaml` 仍把 `00-开工门禁法则.md`、`04-01-任务命中总表.md` 和 `A06-uni_modules与第三方包接入法则.md` 作为断言目标。

`.continue/checks/rule-compliance-review.md` 也仍要求命中旧规则文件名。

这些文件属于当前跟踪的仓级工具配置，不是历史留档。继续保留旧规则名会让 AI 评测和 AI review 辅助误导后续维护者回到已退场规则体系。

### 5.2 质量工具忽略配置残留旧历史路径

`.eslintrc.cjs`、`.prettierignore`、`.stylelintignore` 和 `.textlintignore` 仍保留 `开发规范/`、`项目索引/`、`审核与修改计划/`、`对话台账/` 等旧路径忽略项。

这些目录当前已不属于现行路径。继续保留忽略项会降低“旧入口重新出现时被发现”的能力。

`.textlintignore` 同时忽略 `references/`，但 `references/README.md` 已是现行索引入口，应允许文档检查覆盖。

### 5.3 `垃圾桶/README.md` 仍是旧外层垃圾桶说明

当前 `垃圾桶/README.md` 写明路径为 `H:\工作\天异\垃圾桶\`，并保留旧文件版本头。

当前仓规则已经固定本应用工作根为 [H:\工作\天异\uniapp+vue新架构](/H:/工作/天异/uniapp+vue新架构/README.md)。本仓 `垃圾桶/` 只应作为项目内隔离入口，并且 Git 只跟踪 README。

### 5.4 根现行说明缺少统一元信息

`源码交付白名单.md`、`质量门禁工具接入说明.md`、`依赖与外链安全基线清单.md` 都是当前仍有效的仓级正式说明，但头部元信息格式不统一。

这不影响运行，但不符合当前“正式手工维护 `.md` 文件应包含文档类型、状态、更新时间和说明”的基线。

### 5.5 根 README 的交付边界可再收紧

`README.md` 已说明 `references/` 不保存接口代码快照，但开头仍写“最小参考索引与真实源码参考代码”，表达上容易让人误以为 `references/` 内仍有代码副本。

同时 README 的“当前不属于交付范围”未显式列出 `.continue/`、`promptfoo/`、`semgrep/`、`vale-styles/`、`.tools/` 和 `新藏品/`，这些排除项已在 [源码交付白名单.md](/H:/工作/天异/uniapp+vue新架构/源码交付白名单.md) 中存在。

### 5.6 `提交清单/README.md` 的历史常见文件措辞可更明确

`提交清单/README.md` 的“常见内容”仍列出旧 `当前提交清单.md`，虽然下文说明它已退出当前路径，但该小节标题容易被理解为当前目录内仍应有该文件。
