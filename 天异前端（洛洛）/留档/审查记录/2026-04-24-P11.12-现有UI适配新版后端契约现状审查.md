# 2026-04-24 P11.12 现有 UI 适配新版后端契约现状审查

> 文档类型：审查记录
> 状态：active
> 更新时间：2026-04-24
> 说明：本文件用于审查“现有 UI 适配新版后端契约”主线的当前结构基线、已有接入链和未来改造阻塞点，为总规划提供事实依据。

## 1. 审查目标

回答三个问题：

1. 当前 UI 哪些面已经有正式内容域接入链
2. 当前 UI 哪些面仍是临时 runtime seam 或 page-local mock
3. 新版后端契约接入后，最合理的切换路径应该落在哪一层

## 2. 审查对象

1. `src/contracts/content-api.contract.ts`
2. `src/ports/content.port.ts`
3. `src/implementations/content.mock.ts`
4. `src/implementations/content.http.ts`
5. `src/services/content/content.service.ts`
6. `src/adapters/content/*`
7. `src/services/home-rail/*`
8. `src/services/profile-asset-detail/*`
9. `src/pages/home/composables/**`
10. `src/pages/profile-asset-detail/**`
11. `src/pages/updating/runtime/*`

## 3. 已确认事实

1. 当前正式内容域已有单一正式契约真值源：`src/contracts/content-api.contract.ts`。
2. 当前正式内容域已有统一 port、mock、http、service 主链。
3. `home / activity / profile / profile-asset-detail` 已有 adapter 归一化层。
4. `home` 三轨和详情页现在主要消费 runtime/service 结果，不再直接沾 DTO。
5. `updating` 抽签页已经抽成 page-local runtime seam，但尚未并入正式内容域主链。
6. 用户已明确：后面会有一份新的后端接口契约，用来对接已有后端；当前工作不能把旧 contract 再固化成 UI 真值。

## 4. 已确认风险

1. `高风险`
   - 新版后端契约如果直接打进页面层，会破坏已经收好的 adapter / runtime / view-model 边界。
2. `高风险`
   - 如果不先做“旧 contract 与新 contract 的 delta 矩阵”，后续切换时极容易在各 panel 中重复改造。
3. `中风险`
   - `updating` 页目前走 page-local runtime seam，若未来新契约把它纳入正式内容域，需要单独决定是否并入 content 主链。
4. `中风险`
   - `home` 三轨各自有大量 page-local composables；如果不先按面分批，单次切换会过大，git 边界会失控。
5. `中风险`
   - 如果 mock/http/provider 切换策略不先写死，后面容易出现一半走旧 mock、一半走新 contract 的混态。

## 5. 审查结论

需要立一份完整总规划，至少固定：

1. 新版契约签收前后分别做什么
2. 旧 contract 到新 contract 的 delta 如何收集
3. adapter / runtime / service / UI view-model 哪一层承担什么迁移责任
4. `home/home`、`home/activity`、`home/profile`、`profile-asset-detail`、`updating` 分别怎么切
5. git 分类、批次边界、并行 ownership 和停止条件
