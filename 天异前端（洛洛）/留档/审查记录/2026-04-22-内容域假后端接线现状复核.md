# 2026-04-22 内容域假后端接线现状复核

> 文档类型：审查记录  
> 状态：active  
> 更新时间：2026-04-22  
> 说明：复核“现有示例数据是否按后端 API 口径接入”的当前状态。  
> 后续状态：未完成；需要单开“HTTP 假后端服务 / 假联调链”计划。

## 1. 结论

当前实现分两层：

1. 三主面板页面没有直接读取 `src/mocks/content-db/`。首页、公告、个人中心通过 `homeRail*Content.service -> content.service -> ContentPort` 获取内容。
2. 当前默认 provider 是进程内 `content.mock.ts`，不是一个可启动、可用 `VITE_CONTENT_PROVIDER=http` 访问的 HTTP 假后端服务。

因此，当前不能说“假后端 API 链路已接上”。更准确的说法是：已有正式 contract、`ContentPort` seam、mock provider、http adapter 和 HTTP smoke 验证入口，但还没有把现有示例数据挂到一个 HTTP 假后端服务上。

## 2. 已确认接线路径

1. `src/main.ts` 根据 `VITE_CONTENT_PROVIDER` 选择 `mock` 或 `http` provider。
2. `src/services/content/content.service.ts` 统一通过 `activeContentPort` 调用 `getScene / getResource / getList / runAction`。
3. `src/services/home-rail/homeRailHomeContent.service.ts` 使用 `resolveContentScene` 与 `resolveContentListWithMeta`。
4. `src/services/home-rail/homeRailActivityContent.service.ts` 使用 `resolveContentScene`、`resolveContentListWithMeta` 与 `runContentAction`。
5. `src/services/home-rail/homeRailProfileContent.service.ts` 使用 `resolveContentScene` 与 `resolveContentListWithMeta`。

## 3. 已确认缺口

1. 仓内没有假 HTTP server。
2. `npm run check:content:http` 需要 `VITE_CONTENT_PROVIDER=http` 和 `VITE_CONTENT_API_BASE_URL`，它验证的是外部 HTTP 服务，不会自动启动本地假后端。
3. `content.mock.ts` 只能证明前端进程内 mock provider 可跑，不能证明后端 API 已经真实可联调。
4. 后续若要满足“假数据按后端 API 接入”，需要让示例数据通过正式 HTTP 路径返回，而不是只走进程内 provider。

## 4. 本轮验证

已执行并通过：

```powershell
npm run test:unit -- tests/unit/content/content.mock.spec.ts
```

该验证只能证明 mock provider 返回结构符合当前 contract baseline，不能替代 HTTP 假后端联调验证。

## 5. 后续建议

单开计划处理，最小范围应包括：

1. 设计本地假 HTTP 服务落点。
2. 复用 `src/contracts/content-api.contract.ts` 的 endpoint、wire query 和 envelope。
3. 使用现有 `src/mocks/content-db/` 示例数据生成 HTTP 响应。
4. 让 `VITE_CONTENT_PROVIDER=http` + `VITE_CONTENT_API_BASE_URL` 能指向该假服务。
5. 跑通 `npm run check:content:http`，并明确它和未来真实后端替换的边界。
