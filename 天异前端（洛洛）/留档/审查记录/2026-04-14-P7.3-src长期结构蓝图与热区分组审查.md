## 4. 审查结论

### 4.1 `src/` 长期蓝图不应理解为“立刻改成 features”

当前更稳的长期蓝图是逻辑分层，而不是立即物理引入 `src/features/`：

1. 页面入口与页面私有实现层
   - 以 `pages/` 为锚点
2. 共享 UI 与共享前端运行时基础层
   - `components / composables / models / stores / utils / static`
3. 领域共享运行时服务层
   - `services/*`
4. 内容接入链
   - `contracts / ports / implementations / adapters`
5. mock 与开发数据支撑层
   - `mocks`

### 4.2 顶层目录应按五类身份理解

#### A. 稳定锚点，允许继续存在

1. `pages`
2. `services`
3. `models`
4. `mocks`
5. `static`

#### B. 稳定共享层，允许继续存在但禁止页面私有逻辑回流

1. `components`
2. `composables`
3. `stores`
4. `utils`

#### C. 内容接入链，短期保留四层，禁止继续横向扩散

1. `contracts`
2. `ports`
3. `implementations`
4. `adapters`

结论：

1. 这四层当前可以保留。
2. 但只允许继续承接与“内容接入 / 页面内容适配”直接相关的新增。
3. 不得把新的业务运行时、页面私有逻辑或旁路试验件继续塞进这四个顶层目录。

#### D. 过渡层，不再向顶层新增同类技术目录

1. 顶层技术分层到此为止。
2. 后续新的跨页共享能力应优先落入：
   - 现有 `services/*`
   - 现有 `models/*`
   - 现有 `components / composables / utils / stores`
3. 后续新的页面私有能力应优先落入对应 `pages/*` 附近。

#### E. 后续并轨候选

1. `contracts / ports / implementations / adapters` 后续可以并轨为更紧凑的接入链分组。
2. 这项并轨不在 `P7` 直接物理实施，只作为后续高级治理候选。

### 4.3 热区顺序已可以一次性写死

后续治理优先顺序固定为：

1. `P7.4`
   - 先把 `src/README` 和相关长期目录说明校正到新蓝图
2. `P7.5`
   - 再按新蓝图补职责头，优先：
     - `pages/*` 重型文件
     - `services/*`
     - `contracts / ports / implementations / adapters`
3. `P7.6`
   - 最后基于新蓝图清理废弃和旁路实现，尤其是：
     - `adapters`
     - `pageContent.mock.adapter`
     - `implementations/content.mock`
     - 旧页面兼容壳

## 5. 本轮明确不做

1. 不引入新的 `src/features/` 物理目录。
2. 不把 `contracts / ports / implementations / adapters` 物理合并。
3. 不把 `components / composables / stores / utils` 合并进新的共享目录。
4. 不在本轮移动 `pages/*` 内已有代码。
