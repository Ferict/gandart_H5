# 2026-04-24 updating 抽签页假数据统一桥接接入审查

> 文档类型：审查记录
> 状态：active
> 更新时间：2026-04-24
> 已检查规则：`AGENTS.md`、`规则体系/04-前端实现规则.md`、`规则体系/05-接口与联调规则.md`、`规则体系/07-文档与留档规则.md`、`规则体系/10-自动化执行边界专题细则.md`

## 背景

当前 `src/pages/updating/UpdatingPriorityDrawPage.vue` 直接消费 `src/pages/updating/updatingPriorityDrawContent.ts` 中的 page-local 静态数组。
这条链路能跑，但不符合当前“假数据集中、页面只吃稳定桥接输出、后续可快速替换为真实后端输入”的目标。

## 现状问题

1. 假数据仍散落在页面目录里，没有进入统一 mock 数据落点。
2. 旧入口文件 `updatingPriorityDrawContent.ts` 同时承担“假数据真值 + 页面兼容导出”，缺少 page-local provider seam。
3. 后续若接新版后端契约，需要直接改页面输入面，不够稳。
4. 当前这条 retained page 不适合直接污染现行正式 `content-api.contract.ts`，否则会把未定稿的新后端契约提前固化。

## 本批结论

1. 本批应把抽签页假数据迁入统一 mock 落点。
2. 本批应建立 `updating` page-local runtime seam，而不是修改当前正式 content contract。
3. 页面保持现有 import 面，不改视觉输出。
4. 保留 `updatingPriorityDrawContent.ts`，但降级为兼容桥，避免出现第二份真值。

## 改造边界

### 必改

1. `src/mocks/updating-db/**`
2. `src/pages/updating/runtime/**`
3. `src/pages/updating/updatingPriorityDrawContent.ts`
4. `tests/unit/updating/priorityDrawBridge.spec.ts`

### 不改

1. `src/contracts/content-api.contract.ts`
2. `src/ports/content.port.ts`
3. `src/implementations/content.mock.ts`
4. `src/implementations/content.http.ts`
5. `src/services/content/content.service.ts`
6. 当前页面任何可见视觉效果

## 验收标准

1. 抽签页继续走原 import 面，但原入口文件已退化为兼容桥。
2. 抽签页数据真值迁入统一 mock 数据目录。
3. 页面通过 page-local runtime / service 桥接拿到稳定 view-model。
4. 兼容文件不再保存第二份真值。
5. 定向测试、构建和留档检查通过。
