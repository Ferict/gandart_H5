## 3. 已确认问题

### 3.1 缺少二线文件化通讯入口

当前 Gemini UI 二线已有执行口径，但没有一个固定目录承载：

1. 主线给二线的当前话术和边界。
2. 二线给主线的回传。
3. 主线按用户要求读取二线回传后的读档记录。

影响：

1. 二线问题容易只停留在聊天上下文。
2. 主线无法稳定知道二线是否发现字段缺口、越权需求或写入边界变化。
3. 后续按需读档没有明确目标文件。

### 3.2 契约入口旧口径仍会误导后续代理

部分当前入口仍把 `src/contracts/content-api.contract.ts` 写成“唯一正式 API 真值源”。

当前实际口径应为：

1. 项目级后端契约入口是 [规则体系/后端契约/README.md](/H:/工作/天异/uniapp+vue新架构/规则体系/后端契约/README.md)。
2. 当前前端已实现内容域契约真值源是 [src/contracts/content-api.contract.ts](/H:/工作/天异/uniapp+vue新架构/src/contracts/content-api.contract.ts)。
3. `P11.12` 完成前，两者允许并存，差异由 adapter、runtime、service 和正式计划吸收。

影响：

1. 二线可能绕过新版后端契约，只读当前前端旧实现。
2. 后续 `P11.12` 可能被错误理解成“继续维护 content-api 单文件即可”。
3. 未确认字段可能被误当成已签收字段写入 UI。

## 4. 风险与遗留

1. 本批只修当前规则入口和二线通讯入口，不做 `P11.12` 差异矩阵。
2. 本批不改二线已经生成的 UI 产物。
3. 本批不创建正式 skill；skill 必须从稳定规则和通讯目录派生，不能反向成为真值源。
4. Codex 不能直接实时对话另一个独立 AI 线程；当前自动监听已取消，可用方式是共享文件和用户触发的按需读档。
