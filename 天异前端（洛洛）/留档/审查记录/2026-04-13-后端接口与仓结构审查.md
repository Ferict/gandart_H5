# 2026-04-13 后端接口与仓结构审查

> 审查类型：只读审查
> 审查人：Codex
> 审查范围：后端接口是否真的留出、当前仓目录结构是否利于人工维护

## 1. 总结结论

结论分两层：

1. 站在“契约与前端调用入口”这一层，仓内确实已经给后端留出了统一内容域接口口径。
2. 站在“后端真实可联调 API 是否已经存在并可用”这一层，仓内没有足够证据证明接口已经真正落地。

更直接地说：

- 有接口文档。
- 有前端 HTTP adapter。
- 没有仓内证据证明后端服务真的已提供并已接通。

因此，如果后端反馈“没有留 API 给他们”，这句话在“没有可验证的真实服务入口”这个意义上基本成立。

## 2. 已经留出的内容

### 2.1 正式接口清单已经写明

以下文档都明确声明当前正式接口只有 5 个：

- `README.md`
- `内容域后端统一移交说明.md`
- `backend-handoff/README.md`

当前正式接口口径为：

1. `GET /api/content/scene`
2. `GET /api/content/resource`
3. `GET /api/content/list`
4. `POST /api/content/action/notice-read`
5. `POST /api/content/action/service-reminder-consume`

同时，正式分页口径仍是：

- `page + pageSize`

不是：

- `cursor`

### 2.2 前端已经预留正式 HTTP 调用入口

前端启动期支持 `mock | http` 两种 provider：

- `src/main.ts`
- `src/services/content.service.ts`

HTTP provider 的正式适配器存在：

- `src/implementations/content.http.ts`

该 adapter 已把前端内容域请求映射到：

- `/api/content/scene`
- `/api/content/resource`
- `/api/content/list`
- `/api/content/action/notice-read`
- `/api/content/action/service-reminder-consume`

### 2.3 给后端的 handoff 文档入口也存在

当前仓里已经有一套明确面向后端的阅读入口：

- `backend-handoff/README.md`
- `backend-handoff/后端联调环境与启动说明.md`
- `backend-handoff/后端移交内容清单.md`
- `references/README.md`

其中 `references/` 是最小参考代码快照，不是空文档壳。

## 3. 没有被证实的内容

### 3.1 仓内没有证据证明真实后端服务已存在

本次审查没有发现任何后端服务实现代码，例如：

- `express`
- `koa`
- `nest`
- `fastify`
- `listen(...)`

也没有发现可直接启动真实内容域服务的后端目录。

当前仓里存在的是：

- 前端源码
- 契约文档
- handoff 文档
- HTTP adapter

不存在的是：

- 已提交的后端服务实现
- 已提交的联调 `.env` 配置
- 已提交的可验证 API 地址

### 3.2 运行时默认仍是 mock，不是 http

`src/main.ts` 当前默认逻辑是：

- `VITE_CONTENT_PROVIDER` 未设置时，默认使用 `mock`

这意味着如果没有额外配置，前端不会自动打后端接口，而是继续走：

- `src/implementations/content.mock.ts`

因此，单从当前仓库出发，后端同学很容易得出一个结论：

- 虽然文档写了接口，但实际联调链默认没有指向后端。

### 3.3 当前测试也没有证明“真实接口可联通”

与 `http` 相关的现有单测主要只验证：

- base URL 格式是否合法
- 生产环境是否强制 `https`
- 开发环境是否只允许本地 `http`

但没有验证：

- 某个真实后端地址可以返回正确 DTO
- 后端 5 个正式接口真的已经存在
- 前端在 `http` provider 下完成过端到端联调

## 4. 对后端反馈的判断

### 4.1 如果后端的意思是“前端有没有给出正式接口定义”

答案是：

- 给出了。

证据是：

- `统一内容接口契约.md`
- `内容域后端统一移交说明.md`
- `backend-handoff/README.md`
- `src/implementations/content.http.ts`

### 4.2 如果后端的意思是“前端有没有给到可直接联调的真实 API 环境”

答案是：

- 目前仓内无法证明已经给到。

所以这条反馈不能简单理解成“后端没看文档”，更可能是：

1. 前端给了契约，但没给真实服务地址。
2. 前端给了 adapter，但默认运行仍是 mock。
3. handoff 文档说明了接口形状，但没有落到一个可验证的联调入口。

## 5. 目录结构审查结论

### 5.1 当前仓结构不利于人工维护

正式文档已经明确以下内容不属于交付范围：

- `审核与修改计划/`
- `对话台账/`
- `全量开发规范法则/`
- `项目台账.md`
- `垃圾桶/`
- `备份/`
- 构建产物与本地依赖

但实际 Git 跟踪结果显示，下面这些目录仍被长期跟踪：

- `审核与修改计划/`：138 文件
- `对话台账/`：15 文件
- `垃圾桶/`：30 文件
- `备份/`：18 文件

以上 4 类合计：

- 201 个 tracked files

占当前 tracked files 的比例约为：

- `34.72%`

这说明仓结构仍明显混杂了：

- 活跃源码
- 正式交付文档
- 内部治理材料
- 备份
- 垃圾桶

### 5.2 当前工作树还有路径层面的隐患

本次工具实际所在 worktree 为：

- `C:\Users\1\.codex\worktrees\55dc\天异`

但 `.gitmodules` 把业务目录 `uniapp+vue新架构` 指向了本机绝对路径：

- `H:\工作\天异\uniapp+vue新架构`

结果是当前 worktree 下的 `uniapp+vue新架构` 目录为空壳，真实源码实际在 `H:` 盘。

这会带来两个问题：

1. 审查、构建、CI 复现很容易指错目录。
2. 新同学或后端同学很难仅凭 worktree 理解“真正源码在哪”。

## 6. 当前最准确的结论表述

建议以后统一对内对外这样描述：

1. 当前已经为后端留出了正式统一内容域接口契约和前端 HTTP adapter。
2. 当前仓内尚不能证明后端真实 API 服务已经落地并可联调。
3. 当前前端默认仍走 mock provider，若要联调，必须额外提供 `http` provider 配置和真实 base URL。
4. 当前仓目录仍混有大量内部治理、备份与垃圾内容，不利于人工维护和对外阅读。

## 7. 建议的后续动作

建议后续单独开批，不在本次只读审查内直接处理：

1. 新增一份正式“联调入口确认单”，明确：
   - 当前联调 base URL
   - 由谁提供
   - 哪 5 个接口已真实可用
2. 补一条 `http provider` 联通性验证链：
   - 至少一个 smoke 脚本
   - 至少一个联调环境示例配置说明
3. 把 `审核与修改计划/`、`对话台账/`、`垃圾桶/`、`备份/` 逐步移出对外源码主仓，或拆到单独治理仓。
4. 处理当前坏子模块/worktree 路径问题，避免继续依赖本机绝对盘符。
