# 2026-04-24 后端契约迁入规则体系与未确认分层审查

> 文档类型：审查记录  
> 状态：active  
> 更新时间：2026-04-24  
> 说明：本审查用于确认 `01-正式契约/` 迁入 `规则体系/后端契约/` 是否必要、迁入后应如何分层，以及哪些内容必须明确标为待确认。
> 已检查规则：`AGENTS.md`、`规则体系/README.md`、`规则体系/05-接口与联调规则.md`、`规则体系/07-文档与留档规则.md`

## 1. 审查结论

结论固定为：

1. 必须迁入规则体系。
2. 必须把 `规则体系/后端契约/` 立成项目现行后端契约入口。
3. 必须保留 `01-正式契约/`，但只能作为历史快照。
4. 必须把未确认项显式分层，而不是混在已确认 contract 旁边默认可用。
5. Gemini 二线程和 `P11.12` 都必须统一从新入口读取。

## 2. 迁入理由

当前 `01-正式契约/` 的问题不是内容不够，而是位置不对：

1. 它已经是项目级后端契约资料，但仍挂在根目录旧抽取位置。
2. 它有 `confirmed / exported-unused / unresolved / inferred` 的天然分层能力，却没有被纳入现行规则体系。
3. 二线和 `P11.12` 如果继续各自读不同入口，会形成并行真值源。

因此迁入规则体系是必要动作，不是纯目录整理。

## 3. 已确认层与待确认层

本轮审查后，固定解释如下：

### 3.1 已确认层

1. `contracts/*.contract.ts`
2. `contracts/index.ts`
3. `api-inventory.md` 中 `status = confirmed` 的条目

这些内容可作为当前适配与差异矩阵的稳定基础。

### 3.2 待确认层

以下内容必须明确标成待确认：

1. `api-inventory.md` 中 `status = exported-unused` 的条目
2. `unresolved.md`
3. `contracts/inferred/*`
4. 缺真实后端样例、缺稳定消费证据、缺后端签收的字段与接口

这些内容只可用于：

1. 线索
2. 差异矩阵
3. 缺口记录
4. 后续确认清单

这些内容不可直接用于：

1. 正式实现契约
2. 二线 UI 新字段落地
3. 视为“后端已经提供”

## 4. 对二线的影响

Gemini 二线程后续必须按以下口径工作：

1. 读取 `规则体系/后端契约/README.md`
2. 读取 `规则体系/口径/06-项目后端契约现行使用口径.md`
3. 仅把 confirmed 项当稳定字段依据
4. 遇到待确认项时，只标缺口，不直接接 UI

## 5. 对 P11.12 的影响

`P11.12` 当前应明确：

1. 项目级后端契约入口已存在
2. `P11.12.1` 不再对着根目录旧快照签收，而是对着 `规则体系/后端契约/` 签收
3. 差异矩阵必须内置 `confirmed / exported-unused / unresolved / inferred` 四层语义

## 6. 风险与边界

本轮迁移不代表：

1. 已经完成新版后端签收
2. 已经完成前端适配
3. 已经证明所有接口都可用

本轮只完成：

1. 入口迁移
2. 现行规则回链
3. 待确认层显式标识
