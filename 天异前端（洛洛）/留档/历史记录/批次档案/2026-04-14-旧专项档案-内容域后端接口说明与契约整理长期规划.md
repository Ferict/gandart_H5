# 内容域后端接口说明与契约整理长期规划

> 建档时间：2026-04-03 22:55:00
> 更新时间：2026-04-03 22:55:00
> Encoding: UTF-8
> 文档定位：专项规划文档，用于后端接口说明整理、契约收束与长期联调路线，不替代《统一内容接口契约》正文

## 1. 目标与结论

当前需要优先做的，不是“全仓代码详注”，而是把内容域接口整理成一套可以直接给后端使用的详细说明包。

原因固定为：

1. 后端真正需要的是“接口用途、字段语义、页面消费位置、联调边界、非目标”，而不是页面实现细节。
2. 全仓详注会制造大体积噪音提交，且大量注释会退化成低价值逐句翻译，后续维护成本极高。
3. 当前仓库内容域已经收敛到统一内容接口体系，最适合优先整理的是：
   - `scene`
   - `resource`
   - `list`
   - `action`
4. 现阶段对后端最有价值的交付，是“统一契约正文 + 详细对接说明 + 页面映射矩阵 + 字段归属矩阵”。

本专项的总目标只有两条：

1. 给后端一份详细、可执行、可联调的内容域接口说明。
2. 给前端留下一份稳定的契约整理路线，后续新增字段或改造时有统一入口，不再零散补文档。

## 2. 当前已确认事实

### 2.1 当前正式接口集合

当前正式内容域接口固定为：

1. `GET /api/content/scene`
2. `GET /api/content/resource`
3. `GET /api/content/list`
4. `POST /api/content/action/notice-read`
5. `POST /api/content/action/service-reminder-consume`

### 2.2 当前正式 sceneId

1. `home`
2. `activity`
3. `profile`
4. `settings`
5. `service_hub`

### 2.3 当前正式 resourceType

1. `notice`
2. `home_banner`
3. `activity`
4. `drop`
5. `market_action`
6. `market_item`
7. `category`
8. `asset`
9. `user_profile`
10. `service_entry`
11. `service_action`
12. `settings_action`
13. `profile_asset`

### 2.4 当前正式 list 资源类型

1. `market_item`
2. `notice`
3. `profile_asset`

### 2.5 当前核心边界

1. `scene` 负责页面首屏聚合，不负责把页面内部实现细节返回给前端。
2. `resource` 负责单资源详情，不返回前端页面路由。
3. `list` 负责查询、过滤、排序、分页，不承载页面局部 UI 状态。
4. `action` 当前只承载已读与提醒消费，不承载通用页面行为。
5. `profile_asset` 当前详情页接入缓存与刷新协议，但并未造成真实后端契约漂移。

## 3. 近期详细计划

### B1 统一盘点批次

目标：把“有哪些接口、给谁用、现在怎么用”一次性拉清。

输出固定为：

1. 接口总览表
   - 接口路径
   - 请求方法
   - 所属类型：scene / resource / list / action
   - 当前调用方服务
   - 当前命中页面
2. 页面到接口映射表
   - `home`
   - `activity`
   - `profile`
   - `profile_asset detail`
   - `settings`
   - `service_hub`
3. 资源类型映射表
   - 每个 `resourceType` 当前由谁消费
   - 哪些走详情页
   - 哪些只作为跳转 target

当前推荐落点：

1. 《统一内容接口契约》保留正式契约正文
2. 新增“后端详细对接说明”文档承接解释性内容

### B2 后端详细说明包批次

目标：给后端一份“看完就知道怎么做、为什么做”的说明文档。

每个正式接口都必须写清：

1. 接口用途
2. 主要服务页面/模块
3. 请求参数说明
4. 参数用途说明
5. 返回结构用途说明
6. 前端消费方式
7. 后端必须保证的语义
8. 非目标与不要返回的内容

特别要补详细说明的接口：

1. `GET /api/content/scene`
   - 各 `sceneId` 的用途
   - `blocks` 的语义
   - 哪些 block 是壳层入口，哪些是内容真值
2. `GET /api/content/resource`
   - `notice`
   - `market_item`
   - `profile_asset`
   - `service_entry / service_action / settings_action`
3. `GET /api/content/list`
   - `market_item`
   - `notice`
   - `profile_asset`
   - 查询、排序、分页、筛选参数含义
4. `POST /api/content/action/notice-read`
5. `POST /api/content/action/service-reminder-consume`

### B3 字段归属与责任矩阵批次

目标：明确哪些字段是后端真实字段，哪些仍然只是前端 preset / 占位。

需要按资源类型逐类写清：

1. 后端真实字段
2. 前端消费字段
3. 仅前端上下文字段
4. 当前 UI preset 占位字段
5. 后续若联调落地，需要新增到后端的字段

当前必须重点收口的资源：

1. `profile_asset`
2. `notice`
3. `market_item`
4. `service_entry`

特别强调：

1. `params.category / params.subCategory` 当前仍是前端详情上下文，不是新增后端 contract。
2. `owner / contract / chain / tokenStandard / traits / provenance` 当前仍以 UI preset 为主，若要后端提供，必须单列新增字段需求。

### B4 联调交付包批次

目标：把后端说明整理成真正可交付的联调包。

交付包至少包含：

1. 契约正文
2. 详细说明文档
3. 页面到接口映射矩阵
4. 字段归属矩阵
5. 示例请求
6. 示例响应
7. 非目标清单
8. 后端待确认项清单

后端待确认项必须集中登记，不得散在对话里。

### B5 代码侧最小注释批次

目标：不是全仓详注，而是给契约代码补最有价值的接口注释。

范围固定为：

1. `src/contracts/content-api.contract.ts`
2. `src/ports/content.port.ts`
3. 必要的内容 service 汇总入口

注释只解释：

1. 接口意图
2. 参数边界
3. 返回语义
4. 非显然限制

不做：

1. 页面实现逐句翻译
2. 大面积样式注释
3. 无业务价值的“这行代码做了什么”注释

## 4. 长远规划

### R1 契约正文与说明文档分层固定

长期固定为两套文档：

1. 《统一内容接口契约》
   - 只保留正式 contract 与联调正文
2. 《后端详细对接说明》
   - 承接页面用途、参数解释、字段意图、联调提醒

不再把“前端运行时规则”“接口详细解释”“批次记录”混写在一份正文里。

### R2 资源域演进治理

后续若新增接口或字段，统一按以下顺序治理：

1. 先在页面和 service 层确认真实需求
2. 再补字段归属矩阵
3. 再修改正式契约正文
4. 再同步详细说明文档

不允许只改实现不改文档，也不允许只在台账提及不落正式说明。

### R3 `profile_asset` 长期演进路线

后续若详情页进入链上字段、藏品来源链、持有证明、二级市场动作，需要按单独批次推进：

1. `owner / contract / chain / tokenStandard`
2. traits 与 provenance 正式字段化
3. detail action 扩充
4. 当前 `page + pageSize` 若演进为 `cursor`，单列契约升级批次

### R4 Mock 与正式契约持续对齐

`content.mock` 后续不能只满足前端页面可跑，还必须持续满足：

1. 接口形状对齐正式 contract
2. 示例数据可直接用于后端说明文档
3. 页面演示数据不与正式字段语义冲突

## 5. 交付标准

本专项最终完成标准固定为：

1. 后端能拿着文档直接理解每个接口的用途与参数语义。
2. 前端能明确哪些字段已经是正式 contract，哪些还只是 UI preset。
3. 页面到接口映射关系不再需要靠口头解释。
4. 后续接口扩展有固定落盘入口，不再散落到台账和聊天记录里。

## 6. 当前默认决策

1. 当前优先级最高的是“后端详细说明包”，不是“全仓代码详注”。
2. 代码注释只作为辅助手段，文档说明才是主交付。
3. 《统一内容接口契约》继续作为正式正文，不在本轮重写前端运行时部分。
4. 本专项后续真正执行时，优先顺序固定为：
   - `B1 盘点`
   - `B2 详细说明包`
   - `B3 字段归属矩阵`
   - `B4 联调交付包`
   - `B5 代码侧最小注释`
