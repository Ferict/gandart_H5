## 2. 批复表

### 2.1 `JS` 还是 `TS`

- 决定什么：
  - 决定我们后续页面、接口、配置、store、adapter 是否带静态类型。
- 可选项：
  - `JS`
    - 起步轻、上手快
    - 规则和模板更简单
  - `TS`
    - 接口字段、页面模型、配置结构更稳
    - 重构和多人协作更安全
- 与后端关系：
  - 关系强度：`弱相关`
  - 原因：
    - 不决定后端怎么写
    - 但会直接影响前后端接口契约、字段约束、adapter 和 model 的稳定性
- 当前建议：
  - 如果当前阶段优先把骨架和页面跑起来，可先 `JS`
  - 如果明确要长期做、接口复杂、多人协作强，建议 `TS`
- 最终批复：
  - `TS`

### 2.2 `npm / pnpm / yarn`

- 决定什么：
  - 决定依赖安装、锁文件、团队环境一致性。
- 可选项：
  - `npm`
    - 最通用
    - 接手成本最低
  - `pnpm`
    - 更省空间
    - 依赖结构更干净
  - `yarn`
    - 可用，但当前不是优先推荐
- 与后端关系：
  - 关系强度：`无直接关系`
  - 原因：
    - 纯前端工程工具链选择，不影响后端接口和服务本身
- 当前建议：
  - 以稳定交接优先：`npm`
  - 以长期仓库整洁优先：`pnpm`
- 最终批复：
  - `npm`

### 2.3 `HBuilderX` 为主还是 `CLI` 为主

- 决定什么：
  - 决定新仓后续主要通过图形工具还是工程化命令行来开发、调试和运行。
- 可选项：
  - `HBuilderX 为主`
    - 对 uni-app 原生支持更直接
  - `CLI 为主`
    - 更像标准前端工程
    - 更利于脚本化和自动化
  - `HBuilderX + CLI 并行`
    - 主链一个，另一个辅助
- 与后端关系：
  - 关系强度：`无直接关系`
  - 原因：
    - 这是前端/客户端开发工具链，不影响后端业务逻辑
- 当前建议：
  - 如果强调工程治理和后续自动化，偏 `CLI 为主`
  - 如果强调 uni-app 原生便利，偏 `HBuilderX 为主`
  - 若两边都要，必须明确主链只选一个
- 最终批复：
  - 用户授权按当前项目风格与执行能力代拍板
  - `CLI 为主`
  - `HBuilderX` 作为辅助工具接入

### 2.4 是否首批引入 `Pinia`

- 决定什么：
  - 决定首批是否建立正式的全局状态管理。
- 可选项：
  - `首批不上`
    - 先做页面和骨架
  - `首批就上`
    - 提前为登录态、用户态、全局配置态留正规归宿
- 与后端关系：
  - 关系强度：`中相关`
  - 原因：
    - 它不决定后端接口本身
    - 但会直接影响登录态、配置下发、用户信息、接口结果缓存这些与后端强相关的数据组织方式
- 当前建议：
  - 如果第一阶段很快要接后端和登录，建议首批就上
  - 如果先只搭静态页面，可先不上
- 最终批复：
  - `首批就上`

### 2.5 是否首批引入第三方 UI 组件库

- 决定什么：
  - 决定基础组件是自己做，还是借第三方库。
- 可选项：
  - `不引入`
    - 风格最可控
    - 需要自己搭基础组件
  - `引入`
    - 开发更快
    - 视觉语气更容易被带偏
- 与后端关系：
  - 关系强度：`无直接关系`
  - 原因：
    - 这是 UI 层实现选择，不影响后端接口
- 当前建议：
  - 当前项目风格要求高，首批不建议引通用 UI 库
- 最终批复：
  - `不引入`

### 2.6 是否首批引入 `ESLint / Prettier / Stylelint`

- 决定什么：
  - 决定代码风格、基础错误检查和样式规范是否从第一天就自动收口。
- 可选项：
  - `首批不上`
  - `首批引入 ESLint + Prettier`
  - `首批三者全上`
- 与后端关系：
  - 关系强度：`无直接关系`
  - 原因：
    - 属于前端代码治理工具链
- 当前建议：
  - 至少首批引 `ESLint + Prettier`
  - `Stylelint` 可视项目节奏再决定
- 最终批复：
  - 用户先授权按建议收口
  - 最终补充拍板：`开始就上`
  - 结果：`ESLint + Prettier + Stylelint` 首批引入

### 2.7 首批必须跑通的目标环境顺序

- 决定什么：
  - 决定我们搭环境和验证时，先跑哪个端。
- 可选项：
  - `H5 -> Android App -> iOS App`
  - `Android App -> H5 -> iOS App`
  - 其他你指定的顺序
- 与后端关系：
  - 关系强度：`弱相关`
  - 原因：
    - 它不决定后端实现
    - 但会影响后端联调开始的时机，以及哪一端先需要真实接口配合
- 当前建议：
  - 如果当前先做前端骨架和页面，建议 `H5 -> Android App -> iOS App`
- 最终批复：
  - `H5 -> Android App -> iOS App`

### 2.8 客户端壳工程何时介入

- 决定什么：
  - 决定原生壳、WebView、状态栏沉浸、桥接、键盘和打包链路什么时候开始同步。
- 可选项：
  - `现在就定义边界，H5 基线后联调`
  - `等前端页面成熟后再介入`
  - `从一开始就深度并行`
- 与后端关系：
  - 关系强度：`中相关`
  - 原因：
    - 它主要属于客户端壳工程
    - 但会影响登录、桥接、推送、配置下发、设备能力获取这些后端与客户端共同参与的链路
- 当前建议：
  - 现在先明确边界，等 H5 基线起来后进入实联调
- 最终批复：
  - 当部分页面稳定后先进行一次交接对齐
  - 尽量确保不影响后续新增
  - 接口方式尽量统一
