## 1. 用途

1. 本文件承接内容域在当前项目中的实施边界、项目级落地建议、分批路线与摘要结论。
2. 本文件不是后端接口契约，也不替代 `全量开发规范法则/` 中的现行强制条款。
3. 后端与联调正文仍以 [统一内容接口契约.md](/H:/工作/天异/uniapp+vue新架构/统一内容接口契约.md) 为准；前端运行时与刷新更新实体约束以 `A17-内容域前端接线与刷新更新法则.md` 为准。

## 2. 当前前后端落地建议

### 2.1 前端当前应完成的事项

1. 把当前首页、后续活动页、个人中心页的场景 service 统一成“场景聚合 service”。
2. 把详情页全部统一成“按 `resourceType + resourceId` 拉详情”。
3. 把当前 `route meta` 逐步收口到：
   - `targetType`
   - `targetId`
4. 把页面继续保持为稳定 model 消费层。
5. 提前给图片资源留 `asset` 结构接口位。

### 2.2 前端当前不建议继续扩张的事项

1. 不要为每个藏品做独立页面协议。
2. 不要继续往页面里硬编码内容。
3. 不要让页面直接吃后端 DTO。
4. 不要自己实现图片派生图服务。

### 2.3 后端当前应完成的事项

1. `scene` 聚合接口。
2. `resource` 详情接口。
3. `list` 列表接口。
4. `notice-read` 与 `service-reminder-consume` 动作接口。
5. 统一资源 ID 体系。
6. 统一图片资源输出。

## 3. 当前项目的推荐落地顺序

### 第 1 批：先稳定首页主链路

前端：

1. 统一 `targetType + targetId`
2. 保持页面只消费 model
3. 保留 adapter 接口位

后端：

1. `GET /api/content/scene?sceneId=home`
2. `GET /api/content/resource?resourceType=notice|home_banner|drop|market_action|market_item`
3. `POST /api/content/action/notice-read`

### 第 2 批：扩到活动页、个人中心页与首页壳层二跳

后端：

1. `GET /api/content/scene?sceneId=activity`
2. `GET /api/content/scene?sceneId=profile`
3. `GET /api/content/scene?sceneId=settings`
4. `GET /api/content/scene?sceneId=service_hub`
5. `GET /api/content/resource?resourceType=activity|service_entry|service_action|settings_action|profile_asset`
6. `GET /api/content/list?resourceType=market_item|notice|profile_asset`
7. `POST /api/content/action/service-reminder-consume`

前端：

1. 活动页改为场景聚合消费。
2. 个人中心改为场景聚合消费。
3. 设置页、服务入口页、动作页、资产页统一走 `scene/resource/list` 链路。

### 第 3 批：图片和后台配置继续收口

后端：

1. 资源主图管理
2. 派生图输出
3. 后台配置接入内容域

前端：

1. 切换到 `asset.variants`
2. 不再依赖本地假图

## 4. 当前结论

当前更稳妥的方案是：

1. 不按页面建接口。
2. 按资源域建接口。
3. 用场景聚合接口解决首页、活动页、个人中心首屏。
4. 用统一详情接口解决各内页。
5. 用统一列表接口解决市场、公告、活动、资产列表。
6. 个人资产继续走统一 `profile_asset` 资源，不再额外发明页面专属协议。
7. 用统一动作接口解决公告已读等行为。
8. 图片只维护一张主图，但前端最终吃派生规格图，不长期吃最大原图。

按这套口径推进时：

1. 前端当前工作不会废。
2. 后端实现路径最短。
3. 后续页面再多也不会重新发明协议。
4. 资源互联天然统一。
