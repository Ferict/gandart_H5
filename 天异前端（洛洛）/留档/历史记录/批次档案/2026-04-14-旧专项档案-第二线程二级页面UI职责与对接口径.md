## 1. 适用范围

本口径只用于后续“第二线程”承接二级页面任务时的默认协作边界。

本口径不新增现行角色体系；实际落地时仍映射到治理型子代理模式下的 `task-worker-*`，可使用 `ui-task-worker` 作为轻量标签。

本口径默认服务于“已有业务主链、需要把二级页面 UI 正式落到现有 uni-app 工程”的场景，不服务于产品规则裁决、数据契约制定或全局架构重排。

## 2. 角色映射

- `main-controller`
  - 负责命中规则、任务拆分、文件所有权划分、业务口径确认、最终验收与统一回复。
- 第二线程
  - 负责二级页面纯 UI 落地。
  - 默认只承接可切片、非业务裁决型页面实现任务。
  - 不自行解释、改写或裁决业务规则。

## 3. 第二线程负责范围

第二线程只负责“看得见的页面”，包括：

- 二级页面的页面结构、视觉层级、布局、样式与静态交互态落地。
- 设计稿或参考图到页面骨架、模块编排、展示组件拼装的实现。
- 纯展示所需状态的 UI 落地：
  - 加载态
  - 空态
  - 错误态
  - 禁用态
  - 占位骨架
- 页面内可复用的纯展示组件沉淀，但仅限展示职责，不绑定业务链。
- 用假数据、占位文案、占位图把页面 UI 先跑通，便于主控后续接线。
- 在不改变业务主链的前提下，对现有二级页做 clean-room 视觉重写。

## 4. 第二线程不负责范围

第二线程默认不负责以下事项，这些统一由主控决策或主控另行派发：

- 接口请求、数据联调、字段映射、缓存策略、状态管理。
- 权限、鉴权、埋点、路由守卫、业务提交流程。
- 业务规则判断、表单校验规则、后端契约定义。
- 全局架构调整、目录重排、技术选型变更。
- 现行规范正文修订、规则裁决、批次收口结论。
- 新增依赖、全局 token 重构、壳层结构改写。

## 5. 与主控的对接方式

### 5.1 主控下发给第二线程的最小输入

- 页面名称与页面用途。
- 目标路由或目标落点路径。
- 设计稿、参考图或来源页。
- 信息架构与模块边界。
- 字段展示口径与交互说明。
- 明确哪些块要做，哪些块不做。

### 5.2 第二线程回传给主控的标准输出

- 已完成的 UI 页面或组件代码。
- 页面使用的假数据结构或占位数据形状。
- 需要主控继续接线的位置说明。
- 当前未决问题、阻塞项与需要拍板的分歧点。

### 5.3 派单粒度

- 默认按“一个页面一个包”派发。
- 同一文件同一轮只允许一个写入代理。
- 若页面拆成多个局部块，必须由主控先划清文件所有权。

## 6. 默认文件边界

第二线程默认优先承接以下文件：

- 目标二级页面目录下的页面视图文件。
- 该页面私有展示组件。
- 与该页面直接绑定的局部样式文件。

第二线程默认不主动触碰以下文件：

- `src/services/**`
- `src/stores/**`
- `src/adapters/**`
- `src/main.ts`
- `src/pages.json`
- `src/manifest.json`
- `src/uni.scss`
- `全量开发规范法则/**`

若实际任务必须跨出上述边界，需由主控重新下发明确授权。

## 7. 通用落地方法

第二线程承接二级页面时，默认按以下顺序落地：

### 7.1 先找正式结构基线

优先复用当前仓已稳定存在的独立页结构，不另起一套页面壳：

- 独立页壳层优先参考 `StandalonePageScaffold`
- 交互热区优先接入 `HomeInteractiveTarget`
- 标题、副标题、卡片、胶囊标签、图片壳优先沿用仓内既有语义

### 7.2 先复用结构，再做视觉增强

默认顺序是：

1. 先把头部、主滚动区、内容流、底部操作区的结构摆正。
2. 再把 hero、信息卡、属性区、说明区的视觉层级补齐。
3. 最后才加轻量动画、局部扫描、局部材质和局部强调线。

禁止反过来先做炫技效果，再倒推结构。

### 7.3 旧参考与外部参考的使用口径

React 参考稿、旧项目截图、外部视觉稿只能提供：

- 视觉语义
- 信息层级
- 模块关系
- 交互目标

不得直接照搬：

- 页面壳结构
- 全局样式注入方式
- 桌面鼠标专属交互
- 与当前仓壳层冲突的 safe-area 方案
- 与共享交互基元冲突的私有按钮算法

### 7.4 动效口径

允许：

- 模块级轻量入场
- 局部扫描
- 局部状态增强
- 小范围材质变化

禁止：

- 整页高成本连续炫技动效
- 依赖 `mousemove` 才成立的主交互
- 全量当前可见项统一重播动画
- 为视觉效果牺牲可读、可点、可滚

## 9. 资产详情页视觉参考落地口径

### 9.1 页面定位

当前这页默认对应“资产详情页”二级独立页，默认落点为：

- `src/pages/profile-asset-detail/index.vue`

除非主控明确指定新路由或新页面文件，否则第二线程不主动新建另一份平行详情页。

### 9.2 允许保留的视觉语义

从用户提供的 React 参考稿中，只保留以下视觉目标：

- 档案感
- 终端感
- 已验证资产感
- 博物馆展柜式 hero 感
- 链上印签 / 资产凭证感
- 底部主操作有明确主次关系

### 9.3 明确不迁入主链的参考实现

以下内容禁止直接迁入：

- 页面内假状态栏
- 页面自管 safe-area 逻辑
- `dangerouslySetInnerHTML`
- 页面内全局 CSS 注入
- `lucide-react`
- React 状态实现方式
- 依赖 `mousemove` 的 3D 视差主交互
- 只为桌面 hover 服务的主反馈链
- 未接共享交互基元的私有按钮结构

### 9.4 页面结构口径

资产详情页默认按以下结构组织：

1. 顶部头部
   - 返回按钮
   - 居中标题
   - 右侧菜单或占位
2. Hero 档案卡
   - 主视觉图或展示壳
   - 资产编号
   - 已验证状态或系列标签
   - 主标题与副标题
3. 资产核心信息卡
   - 当前估值
   - 币种
   - 法币换算或补充状态
4. 链上溯源卡
   - owner
   - contract
   - standard
   - chain
5. 属性矩阵卡
   - traits 列表
   - rarity 或补充属性
6. 说明卡
   - 档案说明
   - 备注说明
7. 底部操作区
   - 主操作按钮
   - 次操作按钮

默认采用“多段卡片流”，不做整页浮层拼贴式结构。

### 9.5 视觉语言口径

资产详情页应从现有仓内语言继续演化，而不是完全跳风格：

- 背景：
  - 继续使用当前独立页浅灰舞台基线
- 卡片：
  - 继续使用白底实体卡 + 轻阴影 + 圆角
- 强调色：
  - 继续使用青色系强调线、强调字和状态色
- 英文副标题：
  - 继续使用仓内缩放小字体系
- 图标：
  - 继续走 `AetherIcon`

可在 hero 卡和链上卡内适度增加：

- 扫描线
- 终端小字
- 局部描边
- 局部印签感

但不能把整页都做成高噪声视觉。

### 9.6 交互口径

必须：

- 返回按钮接 `HomeInteractiveTarget`
- 右上角按钮接 `HomeInteractiveTarget`
- Hero 可点击区域接 `HomeInteractiveTarget`
- 底部主次操作接 `HomeInteractiveTarget`
- 所有小控件真实热区不低于 `44px x 44px`

不做：

- 依赖 hover 才能理解的信息显隐
- 依赖鼠标移动的核心交互
- 误导性 `cursor-pointer` 但无动作

### 9.7 滚动与底部操作区口径

资产详情页必须保持：

- 单一主滚动区
- 底部操作区与内容滚动终点同步处理
- 底部吸附区不遮挡正文和按钮

不得：

- 再造一套页面级固定定位补丁去硬顶底部区域
- 让正文滚动终点和底部操作区终点各写一套 magic number

### 9.8 文件落地建议

默认写入范围：

- `src/pages/profile-asset-detail/index.vue`

必要时允许拆以下纯展示局部组件，但仅限当前页私有：

- hero 展示壳
- traits 纯展示网格
- provenance 纯展示卡
- bottom actions 纯展示壳

默认不在本批改：

- `src/services/profileAssetDetailContent.service.ts`
- 路由 query schema
- 导航 helper
- 共享壳层

### 9.9 验收标准

资产详情页落地完成时，至少满足：

- 头部、hero、核心信息、链上信息、属性区、说明区、底部操作区全部可见
- 页面只存在一个主滚动区
- 按钮和图标按钮热区符合当前仓规则
- 独立页结构与现有 `content-resource / notice-detail / profile-asset-detail` 处于同一产品体系
- 视觉上明显吸收了参考稿的“档案感”，但技术结构仍完全属于当前仓

## 10. 停手条件

出现以下情况时，第二线程应停止继续扩写并回传主控：

- 设计稿缺失导致业务含义无法确定。
- 页面改动将跨入接口、状态、路由或权限逻辑。
- 需要修改全局规范、全局 token 或高风险配置。
- 与主控或其他写入代理发生文件所有权冲突。
- 主控临时改变页面落点、路由主键或详情页职责边界。

## 11. 验收口径

第二线程交付合格的标准如下：

- 页面可独立打开并看到完整 UI。
- 假数据驱动下能展示主要模块和关键状态。
- 交互仅保留前端展示反馈，不触发真实业务提交。
- 主控能够在不重做 UI 的前提下继续接业务接线。
- 页面热区、按压反馈、结构尺寸遵守仓内现行 UI 法则。
- 页面结构、视觉、动效和交互都能回扣到本口径文档，而不依赖口头补充解释。

## 12. 一句话口径

第二线程只做二级页面纯 UI 落地，不做业务裁决与业务接线；当前资产详情页默认在现有 `profile-asset-detail` 页面上按仓内独立页结构 clean-room 重写，保留参考稿的视觉语义，不照搬其实现结构。

## 13. 复用优先级总则（避免从零开始）

二级页面统一采用以下优先级：

1. 第一优先：复用当前仓已有正式基座（组件、服务、路由协议、交互基元）。
2. 第二优先：借用 uni-app 官方“能力级模板”（组件能力与事件链），不直接搬运视觉模板。
3. 第三优先：仅在前两级无法覆盖时，新增页面私有 UI 组件。
4. 禁止直接引入和当前设计系统冲突的外部 UI 组件库来替代现有语言体系。

## 15. uni-app 官方模板与能力借用策略

### 15.1 可直接用（能力级，不改造）

1. `scroll-view + refresher` 官方链路：作为二级页刷新能力的标准底座。
2. `page-meta` 页面样式能力：用于统一二级页背景与滚动策略。
3. `uni.getWindowInfo` 与 `env(safe-area-inset-*)`：作为安全区补偿来源。

### 15.2 只参考，不直接照搬（模板级）

1. uni-app 官方示例里的导航栏、列表页视觉模板，只参考结构，不直接复制样式。
2. 官方 demo 的“纯组件堆叠式页面”不直接进入本仓主链，必须先套入 `StandalonePageScaffold`。

### 15.3 当前阶段不引入

1. `uni-ui` 新组件依赖（例如额外导航条、卡片库）默认不引入，先用仓内组件体系完成。
2. 任何会导致“风格跳出首页家族”的官方视觉模板默认不引入。

## 16. 二级页统一骨架模板（执行基线）

统一骨架遵循：

1. `StandalonePageScaffold` 外层壳。
2. 固定顶部导航栏（毛玻璃样式）。
3. 单一 `scroll-view` 主滚动区。
4. 底部操作区与内容终点联动留白。

推荐结构（伪代码）：

```vue
<StandalonePageScaffold :route-source="routeSource">
  <template #default="{ canOpenDrawer, openDrawer }">
    <view class="secondary-page-shell">
      <StandalonePageTopbar
        :title="pageTitle"
        :subtitle="pageSubtitle"
        :can-open-drawer="canOpenDrawer"
        :is-scrolled="isScrolled"
        @back="handleBack"
        @open-drawer="openDrawer"
      />
      <scroll-view class="secondary-page-scroll" scroll-y :refresher-enabled="true">
        <view class="secondary-page-content">
          <!-- 页面模块 -->
        </view>
      </scroll-view>
    </view>
  </template>
</StandalonePageScaffold>
```

## 17. 资产详情双变种落地规划

### 17.1 变种 A：市场藏品详情（Market Variant）

- 页面落点：`src/pages/content-resource/index.vue`
- 生效条件：`resourceType=market_item`
- 数据来源：`resolveContentResourceDetail('market_item', resourceId)`
- 导航入口：`buildContentResourceUrl(...)`（已在首页市场卡与部分公告卡命中）
- UI 重点：价格、流动性状态、交易维度、版本与发行量、档案说明

### 17.2 变种 B：个人藏品详情（Profile Variant）

- 页面落点：`src/pages/profile-asset-detail/index.vue`
- 生效条件：`itemId + category`（现有 query 协议）
- 数据来源：`resolveProfileAssetDetailContent(itemId, category)`
- 导航入口：`buildProfileAssetDetailUrl(...)`（已在个人中心资产卡命中）
- UI 重点：持有态、链上溯源、藏品属性、档案注记、主次操作

### 17.3 双变种共享块

1. 顶部导航栏框架。
2. Hero 展示壳。
3. 信息卡与属性网格卡。
4. 底部双按钮操作区。
5. 状态态样式（loading / empty / error）。

## 18. 与首页三主页面的连接规划

### 18.1 首页（Home）到市场详情

## 21. 风险与质量门禁清单

### 21.1 安全

1. 所有路由参数必须走现有 schema sanitize。
2. 页面不渲染未清洗 HTML，不使用 `v-html` 承载外部内容。
3. 不在日志中输出敏感地址全值（沿用仓内安全日志口径）。

### 21.2 性能

1. 主滚动区只保留一个，避免多重滚动与回流放大。
2. 动效只做模块级轻量动画，禁止整页长链路高频动画。
3. 图片懒加载与占位显影优先复用现有显影组件。

### 21.3 UI 一致性

1. 导航栏、背景色、卡片圆角、强调色与首页家族一致。
2. 英文辅助小字统一 `12px + scale` 策略。
3. 触达控件统一 `HomeInteractiveTarget` 与 `44px` 热区下限。
