# 2026-04-13 P4.3 首页主 rail 模板层拆分边界计划

> 文档类型：计划记录
> 状态：active
> 更新时间：2026-04-13 16:24:10
> 说明：本记录用于在 `P4.2` 脚本层治理达到阶段停点后，给出首页主 rail 后续模板层治理的建议顺序与拆分边界，避免下一轮继续机械拆脚本。

## 1. 前提

当前前提已经成立：

1. [HomeRailHomePanel.vue](/H:/工作/天异/uniapp+vue新架构/src/pages/home/components/HomeRailHomePanel.vue) 的 page-local 脚本组合层已经基本成型。
2. 剩余重量主要转移到模板层。
3. 再继续拆 `useHomeRailHome*` 微文件的收益已经明显下降。

## 2. 下一阶段目标

下一阶段如果继续推进首页主 rail，应从模板层治理入手，目标固定为：

1. 减少 [HomeRailHomePanel.vue](/H:/工作/天异/uniapp+vue新架构/src/pages/home/components/HomeRailHomePanel.vue) 模板长度。
2. 保持 page-local 语义，不引入跨 rail 抽象组件。
3. 保留现有脚本 composable 结构，不反向回流逻辑。

## 3. 建议拆分顺序

### 3.1 第一优先级

先考虑抽离以下块之一：

1. `featured` 组合区
2. `banner + notice` 头部组合区

原因：

1. 这两块模板边界天然清晰。
2. 交互输入较少，prop 面相对可控。
3. 不会立刻碰到 `market` 结果区的强耦合 window / overlay 逻辑。

### 3.2 第二优先级

再考虑 `market` 结果区拆分：

1. `market header + search + sort + tag`
2. `market results grid + overlay + footer`

原因：

1. `market` 区体量最大。
2. 但它的交互和运行时耦合也最重，应后于头部组合区。

## 4. 不建议的拆法

1. 不把单个 `swiper-item` 或单张 market card 再拆成大量超小组件。
2. 不做跨 `home / profile / activity` 的模板抽象复用。
3. 不为了降行数去拆只有一层 `view` 包裹的空壳组件。

## 5. 建议的停点判断

出现以下任一情况就应停下：

1. 新组件 prop 数量已经接近原模板局部节点数量。
2. 新组件只能复用一次，且不降低阅读成本。
3. 需要把大量事件透传才能维持现有行为。

## 6. 当前结论

1. `P4.2` 之后继续推进首页主 rail，正确方向不是再拆脚本，而是模板层边界治理。
2. 下一轮最稳的起点是 `featured` 或 `banner + notice` 组合区。
