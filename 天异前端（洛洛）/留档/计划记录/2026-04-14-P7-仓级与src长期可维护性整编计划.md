# 2026-04-14 P7 仓级与 src 长期可维护性整编计划

> 文档类型：计划记录
> 状态：active
> 更新时间：2026-04-14
> 说明：本计划用于承接 `P6` 完成后的下一条治理主线，统一处理仓级目录边界、素材落点、废弃代码混入、`src/` 长期结构、目录说明文件和源码职责头缺口。

## 1. 目标

`P7` 的目标固定为：

1. 把仓级目录中“现行源码、依赖、文档、规则、素材、生成物、历史与垃圾桶”的边界收清楚。
2. 把 `src/` 从“可运行但长期维护成本高”的状态，收向“人工可维护、职责可解释、落点可预期”的状态。
3. 系统治理废弃代码混入目录、素材散放、目录说明缺口和文件职责头缺口，而不是继续零散补丁。
4. 对后续目录移动、文件迁移、垃圾桶投递建立明确的先后顺序和验收标准。

## 2. 当前已知问题基线

本计划以以下事实为起点：

1. 仓根仍混入 `.codex-tmp/`、`dist/`、`test-results/`、`.github/` 这类需要重新界定职责的路径。
2. `src/` 一级目录虽然已补齐 `README`，但整体仍是旧技术分层：
   - `adapters / components / composables / contracts / implementations / mocks / models / pages / ports / services / static / stores / utils`
3. 素材与静态资源当前分散在：
   - [新藏品/README.md](/H:/工作/天异/uniapp+vue新架构/新藏品/README.md)
   - [src/static/README.md](/H:/工作/天异/uniapp+vue新架构/src/static/README.md)
   - [src/mocks/README.md](/H:/工作/天异/uniapp+vue新架构/src/mocks/README.md)
4. 按当前宽松扫描口径，`src/` 下约 `365` 个 `.ts/.js/.vue` 文件里，约 `270` 个疑似缺少显式职责头。
5. 废弃代码与旁路实现只做过零散清理，尚未进入系统性“保留 / 迁移 / 冻结 / 删除”分级。

## 2.1 本批固定默认决策

为避免执行中频繁停下来拍板，`P7` 先固定以下默认决策：

1. `P7` 不在本批内正式引入新的 `src/features/` 物理目录。
   - 本批只产出长期蓝图与分批迁移顺序，不直接做 feature 化大迁移。
2. [新藏品/README.md](/H:/工作/天异/uniapp+vue新架构/新藏品/README.md) 继续保留为候选原图层，不整体并入 `src/static/`。
3. 文件职责头按分级治理，不做全仓机械补头。
   - 强制优先级先放在：
     - 非平凡 `.vue`
     - 重型 page-local runtime / composable
     - `service / implementation / contract / provider` 入口文件
4. `.codex-tmp / dist / test-results` 统一按非现行长期目录处理。
5. `.github` 继续保留为现行基础设施目录。
6. 根目录正式文档保持“必要工程入口 + 正式契约文档 + 少量现行说明”的收敛状态。

## 3. 范围

本阶段纳入：

1. 顶层目录职责复核与说明补齐。
2. 生成物、临时产物和非现行目录边界复核。
3. `src/` 长期结构蓝图和热区分组。
4. 素材与静态资源统一落点。
5. 目录级 `README.md` 和文件职责头补齐机制。
6. 废弃代码与旁路实现分级治理。

本阶段不纳入：

1. 后端契约或联调协议变更。
2. 新页面功能开发。
3. 已完成的 `P5` 历史档案迁移主线重做。
4. 仅为对称而继续拆微粒度 helper。

## 4. 固定执行顺序

### P7.1 仓级目录与混入内容审查

目标：

1. 逐类审查顶层目录与仓级混入内容。
2. 明确哪些是现行长期目录，哪些是生成物、临时目录或冻结入口。
3. 对 `.codex-tmp/`、`dist/`、`test-results/`、`.github/` 的当前定位给出明确结论。

交付产物：

1. 一份仓级目录分级审查。
2. 一份仓级目录整编边界计划。

### P7.2 素材与静态资源统一落点治理

目标：

1. 统一“候选素材、运行时静态资源、mock 数据引用层、历史素材”的放置规则。
2. 明确 [新藏品/README.md](/H:/工作/天异/uniapp+vue新架构/新藏品/README.md) 的长期定位。
3. 清理散落在非正式目录中的素材说明。

交付产物：

1. 一份素材分层审查。
2. 一份素材与静态资源统一落点规则定稿。
3. 相关现行目录说明回写。

### P7.3 `src/` 长期结构蓝图与热区分组

目标：

1. 在现有 `src/` 基础上，定义长期可维护的逻辑蓝图，而不是继续按旧技术分层无限扩散。
2. 明确哪些顶层目录属于：
   - 稳定锚点
   - 共享层
   - 内容接入链
   - 过渡层
   - 后续并轨候选
3. 形成后续 `P7.4-P7.6` 的统一上位边界。

交付产物：

1. 一份 `src` 长期结构蓝图与热区分组审查。
2. 一份 `src` 长期结构蓝图与默认分组定稿。

### P7.4 目录级 README 系统补齐与校正

目标：

1. 把顶层、`src/` 二级及长期维护热区目录的 `README.md` 状态补齐或校正。
2. 清理“有 README 但职责已过时”的目录。
3. 对新增或调整后的目录，同步回写入口索引。

### P7.5 文件职责头分级治理

目标：

1. 建立源码文件职责头的分级清单，而不是对全仓一刀切。
2. 优先治理：
   - 非平凡 `.vue`
   - 重型 page-local runtime / composable
   - `service / implementation / contract / provider` 入口文件
3. 对可豁免文件建立白名单。

### P7.6 废弃代码与旁路实现分级治理

目标：

1. 系统识别：
   - 已弃用源码
   - 重复实现
   - 只剩兼容壳的中间层
   - 应冻结或送垃圾桶的旁路文件
2. 明确哪些应删、哪些应迁、哪些应保留并写明原因。

### P7.7 收口复核与提交边界整理

目标：

1. 复核 `P7` 是否真正提升了人工维护性，而不是只增加目录层级。
2. 整理提交边界，避免把仓级目录治理、素材治理、职责头治理和代码重排混成不可复查的大提交。

## 5. 验收标准

完成 `P7` 后必须满足：

1. 顶层目录语义更清楚，现行目录、生成物、临时目录、历史与垃圾桶边界不再混淆。
2. 素材文件有统一规则和明确落点，不再散落在流程目录、历史目录和源码目录之间。
3. `src/` 的长期结构蓝图明确，后续不再继续按“看到哪里乱就补哪里”推进。
4. 长期目录的 `README` 达到可维护状态，不只占位。
5. 文件职责头治理有优先级、有白名单、有阶段结果，不做机械补头。
6. 废弃代码有明确分级和处置方式，而不是继续混在现行目录里。

## 6. 待拍板项

1. 本批暂无额外待拍板项。
2. 如执行中出现现有结构无法承接的新类型目录需求，按现行规则先请求用户明确许可。
