## 1. 目标

本阶段只解决两件事：

1. 把内容域 API 从“多文档、多入口、半过渡”收成**一个真实源文件**。
2. 把当前路径从“仍混有历史/过渡/生成/旧入口”收成**干净、清洁、无歧义**的状态。

## 2. 目标状态

`P11` 完成后，目标状态固定为：

1. 后端只需要面对一个正式源文件，就能按注释、类型和规则接上当前前端需要的全部 API。
2. 当前路径不再并行保留多套 API 真值源。
3. 当前路径不再保留“冻结入口、映射表、回链入口”这类过渡性结构。
4. 现行规则、现行口径、现行源码、联调入口、历史记录、垃圾桶之间边界清晰。
5. 应用仓处于可解释、可提交、可审查的干净状态。

## 3. 默认决策

本阶段默认决策直接定为：

1. **唯一正式 API 真值源**：
   [src/contracts/content-api.contract.ts](/H:/工作/天异/uniapp+vue新架构/src/contracts/content-api.contract.ts)
2. **唯一正式 API 真值源必须承载的内容**：
   - 五个正式接口的路径、方法、请求参数、响应 envelope
   - 正式 DTO 与字段归属
   - 前后端边界注释
   - 当前正式分页口径、正式资源类型与场景范围
   - 当前明确不属于后端字段的内容
3. **`接口契约/` 当前的拆分式文档不再继续作为正式真值源存在**。
4. **`统一内容接口契约.md` 与 `内容域后端统一移交说明.md` 不再作为当前路径正式 API 真值源存在**。
5. **`backend-handoff/` 仅保留 handoff 与联调环境职责，不再承载正式 API 真值正文**。
6. **`P11` 的代码实施必须排在 `P10` 边界收口后**，不在当前大积压改动上继续叠加；`codex exec` 连续执行时，`P10` 收口后可直接顺延进入 `P11`，不需要再次等待用户复核。
