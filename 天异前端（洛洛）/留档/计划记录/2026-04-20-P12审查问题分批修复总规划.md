## 6. 自动模式执行边界

本规划允许在用户明确授权“按规划执行某一批”后进入自动模式。自动模式包括 `codex exec`、云端任务、或当前会话中用户明确要求的非交互连续执行。

自动模式默认只允许推进当前被点名的单个批次，不自动跨批次。即使第一批完成且验证通过，也必须在交付、回写、提交后停止，等待用户确认是否进入第二批；第二批、第三批同理。

自动模式允许自动完成：

1. 阅读当前批次计划和对应审查记录。
2. 修改当前批次计划明确列出的文件。
3. 运行当前批次计划列出的验证命令。
4. 写当前批次交付记录。
5. 回写对应审查记录和总规划中的批次状态。
6. 在工作区只包含当前批次变更且验证通过时形成独立 Git 提交。

自动模式不得自动完成：

1. 调整三批执行顺序。
2. 把未点名批次顺手合并执行。
3. 新建长期顶层目录。
4. 删除、移动或重命名活跃源码入口。
5. 升级生产依赖或重写 `package-lock.json`。
6. 引入真实后端地址、真实密钥或真实模型评测结论。
7. 修改本规划未列入的业务源码、接口契约或页面实现。
8. 执行分支删除、历史改写、强制推送等 Git 仓库高风险操作。

自动模式的每批固定停止点为：

1. 当前批次完成标准全部满足。
2. 当前批次验证命令已执行并记录结果。
3. 当前批次交付记录、审查回写和总规划回写完成。
4. 当前批次独立提交完成，或已在留档中写明暂不提交原因。

到达固定停止点后，只能汇报结果和下一步建议，不得自动进入下一批。

## 7. 全局停止条件

任一批次命中以下情况必须停止：

1. 需要改动真实业务源码。
2. 需要新增长期顶层目录。
3. 需要删除或移动 `src/` 活跃入口。
4. 需要真实后端 base URL 或真实 Promptfoo 密钥才能判断。
5. 发现修复会让 GitHub 联调源码仓和内部治理仓边界冲突。
6. 发现依赖安全基线需要实际升级依赖才能继续。
7. 发现本批变更范围会覆盖未列入计划的文件。
8. 发现工作区存在与当前批次冲突的用户未提交改动。
9. 任一必须验证命令失败且无法在当前批次范围内解释或修复。
10. 发现需要用户重新拍板的策略选择，例如“交付仓是否保留质量规则目录”或“audit 是否立即硬阻断”。

## 8. 自动模式完成汇报要求

每批自动模式结束时必须汇报：

1. 实际停止位置。
2. 已修改文件范围。
3. 已运行验证命令和结果。
4. 是否形成 Git 提交及提交 hash。
5. 未解决阻塞项。
6. 下一步建议，但不直接进入下一批。

## 9. 全局验证基线

每批至少执行：

1. `git diff --check`
2. `npm run lint:format`
3. `npm run lint:docs`

如改动 `.github/`、`scripts/quality/` 或 `package.json`，还必须按对应详细计划补充专项验证。
