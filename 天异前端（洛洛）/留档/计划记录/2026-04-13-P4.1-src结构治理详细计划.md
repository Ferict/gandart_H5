`P4.1` 只解决 `src/` 里的结构性问题，不再继续做“是否有死代码”的排查。

本阶段目标固定为：

1. 降低 `src/services/` 与 `src/pages/home/composables/` 的人工理解成本
2. 让页面私有实现更接近页面目录
3. 在不引入大范围行为漂移的前提下，开始从“技术分层堆叠”过渡到“业务语义更清晰”的结构

## 3. 实施顺序

### 3.1 第一批：`src/pages/home/composables/` 目录内部分组

先处理风险最低、收益最高的一块：

1. 把 `src/pages/home/composables/` 按 `home / activity / profile / shared` 分成子目录
2. 只做目录内部分组，不改文件职责
3. 通过导入路径更新完成物理收拢

当前理由：

1. 这里已有 33 个文件
2. 命名前缀天然已经分群
3. 改动范围主要在首页链路内部，风险可控

### 3.2 第二批：`src/services/` 根目录收口

在首页 composable 分组稳定后，再处理 root `services/`：

1. 保留真正跨页面、跨业务的 `content* / homeRail* / homeShell* / routeQuery* / secondaryPage*`
2. 重新审查 `homeNoticeState.service.ts`、`pagePlaceholderContent.service.ts` 等边界模糊文件
3. 如确认只服务于单一路径，再迁回对应页面或业务邻近目录

### 3.3 第三批：mock 目录口径收口

最后处理 mock 语义问题：

1. 保持 `src/mocks/content-db/` 作为 mock 数据源目录
2. 保持 `src/implementations/content.mock/` 作为 mock provider 实现目录
3. 若后续仍证明理解成本高，再决定是否改名或改层级

当前规则：

1. 在没有充分收益前，不因为“看起来分裂”就机械搬目录

## 4. 当前不做

`P4.1` 明确不做：

1. 不把 `src/` 直接整体重构成 `app/features/shared`
2. 不重写页面逻辑
3. 不调整接口契约与联调链
4. 不把首页、详情页、设置页的大块业务行为混成同一批改造

## 5. 验证要求

每一批都至少要跑：

1. `git diff --check`
2. `npm run knip`
3. `npm run type-check`

必要时再补：

1. `npm run test:unit`
2. `npm run test:e2e`

## 6. 当前结论

当前最稳的下一步，不是继续找死代码，而是进入 `P4.1` 第一批，把 `src/pages/home/composables/` 做目录内部分组收拢。
