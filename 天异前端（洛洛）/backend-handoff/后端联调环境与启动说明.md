# 后端联调环境与启动说明

## 1. 文档定位

本文件只说明三件事：

1. 后端联调时怎样拉起当前前端源码仓
2. 当前最少需要哪些环境和命令
3. 仓内最小验证入口怎样跑

本文件不承担 API 正文职责；接口语义统一看：

- [src/contracts/content-api.contract.ts](../src/contracts/content-api.contract.ts)

## 2. 推荐环境

当前仓库 `package.json` 未单独锁定 `engines` 字段，但结合现有依赖与工具链，推荐按下面口径准备环境：

1. Node.js：推荐 `20.x LTS`
2. npm：使用随 Node 20 一起安装的 npm 即可
3. 操作系统：Windows 为当前主验证环境；H5 本地联调对 macOS / Linux 也应可用

## 3. 安装与启动

### 3.1 安装依赖

```bash
npm install
```

### 3.2 启动 H5 本地开发

```bash
npm run dev:h5
```

固定端口版本：

```bash
npm run dev:h5:5180
```

### 3.3 常用校验命令

```bash
npm run type-check
npm run test:unit
npm run test:e2e
npm run quality:check
npm run check:content:http
```

如需补齐 Windows 质量工具：

```bash
npm run setup:quality:windows
```

### 3.4 内容域五接口最小验证矩阵

`npm run check:content:http` 当前会：

1. 检查 `VITE_CONTENT_PROVIDER=http`
2. 检查 `VITE_CONTENT_API_BASE_URL`
3. 直接读取 [src/contracts/content-api.contract.ts](../src/contracts/content-api.contract.ts) 中的正式 endpoint 描述
4. 默认直接验证：
   - `GET /api/content/scene`
   - `GET /api/content/list`
     - 默认 `resourceType=market_item`
     - 默认 `sortField=listedAt`
     - 默认 `sortDirection=desc`
5. 在补齐样本参数后继续验证：
   - `GET /api/content/resource`
   - `POST /api/content/action/notice-read`
   - `POST /api/content/action/service-reminder-consume`

样本参数和字段语义统一以 [src/contracts/content-api.contract.ts](../src/contracts/content-api.contract.ts) 为准。

当前接口口径只在 contract 中维护，后端联调时重点确认这些边界：

1. `GET /api/content/list` 的 wire query 是扁平字段：`sortField / sortDirection / startDate / endDate`。
2. 用户态接口不新增 `userId / accountId` 参数，后端从登录态、token 或 session 解析当前用户。
3. 业务失败优先返回 HTTP 2xx + `ContentEnvelope.code != 0`；当前前端 adapter 不解析非 2xx body。
4. scene/resource 可在有效空结果时返回 `data:null`；list 成功时必须返回分页壳和 `items: []`。
5. `tone / visualTone / placeholderIconKey / badgeType / badgeLabel / indicatorTone / englishTitle / badges` 是内容展示元数据，不是前端运行时字段。

如果缺少以下样本参数，脚本会给出 `skip`，而不是伪造“已真实联通”：

1. `VITE_CONTENT_SMOKE_RESOURCE_TYPE`
2. `VITE_CONTENT_SMOKE_RESOURCE_ID`
3. `VITE_CONTENT_SMOKE_NOTICE_ID`
4. `VITE_CONTENT_SMOKE_SERVICE_ID`
5. 可选 `VITE_CONTENT_SMOKE_LATEST_MESSAGE_ID`

### 3.5 新版后端 wrapper 接驳启动

当需要直接接新版后端 190 个业务 wrapper，而不是当前 5 个 `/api/content/*` 聚合接口时，使用 `backend-http` provider：

```bash
VITE_CONTENT_PROVIDER=backend-http
VITE_CONTENT_BACKEND_API_BASE_URL=https://后端域名
npm run dev:h5
```

Windows PowerShell 示例：

```powershell
$env:VITE_CONTENT_PROVIDER='backend-http'
$env:VITE_CONTENT_BACKEND_API_BASE_URL='https://后端域名'
npm run dev:h5
```

说明：

1. `backend-http` 不要求后端先实现 `/api/content/*` 聚合接口。
2. 前端会在 `src/implementations/content.backend-http.ts` 内部把多个后端 wrapper 归一成当前 `ContentPort`。
3. 若没有配置 `VITE_CONTENT_BACKEND_API_BASE_URL`，前端会回退读取 `VITE_CONTENT_API_BASE_URL`。
4. 当前尚无独立 `backend-http` smoke 脚本；最小自动验证先看 `tests/unit/content/content.backend-http.spec.ts`。
5. 当前已接范围与待后端补充项看 [P11.12-后端接口接入分级清单.md](./P11.12-后端接口接入分级清单.md)。
6. 未覆盖 wrapper 必须保持显式未支持，不允许通过 mock 或空数据伪装成真实后端成功。

## 5. 当前联调边界

1. GitHub 仓库当前应作为“前端源码联调仓”，不是仅文档仓。
2. 后端可以查看真实 `src/` 与 `tests/`，但接口真相只以 [src/contracts/content-api.contract.ts](../src/contracts/content-api.contract.ts) 为准。
3. `references/` 不再保留接口代码快照副本，只保留索引说明，避免和真实源码漂移。
4. 当前正式列表分页仍是 `page + pageSize`，不是 `cursor`。
5. `route/query`、刷新协议、缓存键、hydrate 细节仍不是后端 contract。
6. `owner / contract / chain / tokenStandard / traits / provenance` 仍不是当前正式后端字段。
7. `backend-http` 是前端适配旧 wrapper 的 provider，不改变当前 5 个正式内容域聚合接口的 contract。

## 6. 环境与联网注意事项

1. 当前仓库存在 `Playwright`、`Semgrep`、`Textlint` 等质量工具依赖，首次安装可能需要稳定外网。
2. 若本地访问外网依赖受限，可在开发环境内使用代理/VPN，但不要把本机代理账号、私有凭据或 `.env.local` 一起提交。
3. 当前仓库没有要求后端提交本机环境文件；只需要按本说明本地安装依赖即可。

## 7. 不应提交到联调源码仓的内容

1. 已送桶的旧历史入口（原 `审核与修改计划/`、`对话台账/`、`项目索引/`）
2. 已送桶的旧历史总账入口（原 `项目台账.md`）
3. `规则体系/`
4. `留档/`
5. `提交清单/`
6. `垃圾桶/`
7. 历史回档材料与冻结快照
8. 已送桶的旧回档目录（原 `备份/`）
9. 已送桶的本地生成物与临时工位目录（原 `.codex-tmp/`、`dist/`、`test-results/`）
10. `node_modules/`

以上边界的正式清单以 [源码交付白名单.md](../源码交付白名单.md) 为准。
