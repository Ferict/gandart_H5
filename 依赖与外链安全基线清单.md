# 依赖与外链安全基线清单

> 文档类型：安全与依赖基线说明
> 状态：active
> 更新时间：2026-04-20
> 说明：本文件记录当前直接依赖、开发依赖和 H5 外链脚本/样式的安全边界。

本文档用于回答 3 个问题：

1. 这个包是干什么的。
2. 它落在哪一层。
3. 它现在能不能替换，替换边界是什么。

适用范围：

- 当前仓库直接生产依赖
- 当前仓库直接开发依赖
- 当前仓库 H5 入口外链脚本/样式基线

本文档不覆盖：

- 间接依赖的逐包治理
- 后端依赖、部署依赖、网关依赖
- 普通图片链接与业务内容里的外部图片地址

## 1. 当前外链资源基线结论

当前仓库 H5 入口未接入外部脚本或样式 CDN 资源。

现状口径：

- [index.html](./index.html) 当前只加载本地入口脚本 `/src/main.ts`
- 当前仓库不存在已登记的第三方脚本/样式外链白名单
- 当前仓库不需要为现有入口补 `SRI`

后续准入口径：

- 若未来新增第三方脚本、第三方样式、`preload`、`modulepreload` 等会执行或参与构建链的外链资源，必须先登记来源
- 登记后必须同时满足：
  - 命中仓库外链白名单
  - 带 `integrity`
  - 带 `crossorigin="anonymous"`
- 普通内容图片链接不在本清单的外链脚本/样式治理范围内

### 2.1 生产依赖使用边界说明

- `@dcloudio/*` 这一组依赖属于平台与运行时基础，不应被普通业务组件直接当作“可随时更换的工具包”对待。
- `lucide-vue-next` 属于展示依赖，替换成本最低，但替换时要同步核对现有图标命名、视觉风格与 bundle 体积。
- `pinia` 与 `vue` 共同构成当前业务层的主要编程模型，替换时不只是改导入，还会影响状态流、测试和组件组织方式。

### 3.2 开发依赖替换口径

- 可以相对独立替换的，主要是“格式化、文档、报告、图形化评测”这类工具型依赖。
- 需要成组评估的，主要是“ESLint 体系”“Stylelint 体系”“uni 构建链”“Vue 类型检查链”。
- 不建议在没有明确收益时替换的，是会带来全链路适配成本的基础工具，如 `vite`、`typescript`、`vue-tsc`、`@dcloudio/vite-plugin-uni`。

## 4. 当前仓库依赖口径总结

如果只用一句话概括当前依赖结构：

- 生产层是 `uni-app + Vue 3 + Pinia + lucide-vue-next`
- 开发层是“`Vite + TypeScript + ESLint/Stylelint/Prettier + Vitest/Playwright + Promptfoo + 提交流程治理`”这一整套质量与构建工具链

如果只回答“这个包落在哪层、能不能替换”：

- 平台运行时类：能换，但本质上等于迁移平台，不是小改
- 状态管理与框架类：能换，但改造成本高
- 展示类工具：可以换，相对低风险
- lint / format / docs / report 类：可以换，但要成组保留等价能力
- 构建与类型系统类：可以换，但风险高，必须整体评估

## 6. 生产依赖 audit 接受基线

当前正式审计命令固定为：

```bash
npm audit --omit=dev --json
```

当前基线采集时间：2026-04-20。

当前生产依赖 audit 统计：

| 严重级别   | 当前数量 | CI 阻断口径                              |
| ---------- | -------- | ---------------------------------------- |
| `critical` | 0        | 超过 0 立即阻断                          |
| `high`     | 18       | 超过 18 立即阻断                         |
| `moderate` | 2        | 持续输出，不作为当前 P12 批次阻断条件    |
| `low`      | 0        | 持续输出，不作为当前 P12 批次阻断条件    |
| `total`    | 20       | 以 `critical` 与 `high` 是否超过基线为准 |

当前直接受影响的生产依赖：

1. `@dcloudio/uni-app`
2. `@dcloudio/uni-app-plus`
3. `@dcloudio/uni-components`
4. `@dcloudio/uni-h5`

当前主要传递风险集中在：

1. `@dcloudio/uni-cli-shared`
2. `@dcloudio/uni-cloud`
3. `@dcloudio/uni-console`
4. `@dcloudio/uni-h5-vite`
5. `@intlify/*`
6. `vite`
7. `esbuild`
8. `picomatch`

风险接受原因：

1. 当前生产依赖风险集中在 uni-app 运行时、H5/App 平台适配和构建链依赖树。
2. 当前可用修复路径多显示为 semver-major 或平台链迁移，不能在 P12 审查修复批次中作为局部补丁安全落地。
3. 本批只登记当前生产依赖 audit 基线，并防止新增 high/critical 回退；不声称这些漏洞已经修复。

升级边界：

1. 不在 P12 审查修复批次中升级 `@dcloudio/*` 平台链。
2. 后续若要降低当前 high 基线，必须单独规划 uni-app 平台链、Vite 和相关构建依赖升级，并跑完整构建、H5 E2E、App 构建链验证。
3. 如出现生产依赖 `critical > 0` 或 `high > 18`，视为安全基线回退，`npm run audit:deps` 应返回非零退出并阻断 CI。

必须重新评估基线的触发条件：

1. 新增或升级任何生产依赖。
2. 升级 uni-app 平台链、Vite 或构建工具链。
3. `npm audit --omit=dev --json` 统计发生变化。
4. GitHub advisory 对当前直接依赖新增 `critical`。
5. 发版前需要形成新的安全接受结论。
