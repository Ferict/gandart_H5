## 2. 本轮补强的重点

### 2.1 接口规则回答了“什么才算真的留出 API”

本轮已经把“已留出 API”的判定从模糊说法收成正式规则，要求同时满足：

1. 有正式 contract
2. 有字段归属
3. 有 `http` adapter
4. 有联调入口说明
5. 有可切到 `http` 的运行方式
6. 有正式验证链

因此：

1. 只有契约，不等于后端可直接接入。
2. 只有 `mock + adapter`，也不等于后端可直接接入。

### 2.2 接口文档集中目标已经定死

本轮已经把长期正式目标目录定为：

1. `接口契约/README.md`
2. `接口契约/内容域接口总表.md`
3. `接口契约/内容域字段字典.md`
4. `接口契约/内容域联调说明.md`

同时明确：

1. 当前过渡期仍以根目录正式契约文档和 `backend-handoff/` 为准。
2. 本轮不提前物理迁移接口文档。

### 2.3 目录规范与接口规范的关系已经对齐

当前已经明确：

1. 先定目录规则和接口规则
2. 后做仓级目录治理
3. 后做 `src/` 物理重构
4. 后做接口文档集中迁移

### 2.4 工程与工具规则已经补齐基础门禁层

当前已经明确：

1. `package.json` scripts 是正式命令真值源
2. 基础开发验证是 `type-check + lint + test:unit`
3. 页面链路改动应追加 `test:e2e`
4. `quality:check`、`semgrep`、`vale`、`audit` 等工具各自有边界，不互相替代

### 2.5 前端实现边界已经开始从“技术目录思维”转向“人工维护思维”

当前已经明确：

1. 页面壳应薄
2. 业务实现应靠近业务域
3. `shared` 只承载真正共享内容
4. `service / adapter / provider` 各自有清晰边界
5. `src/` 的问题先写成规则，后续再按规则物理治理

### 2.6 安全、子代理与留档口径已经并入新规则体系

当前已经明确：

1. 敏感信息、外链、缓存和第三方接入有统一安全边界
2. 子代理是常用能力，但按需使用、即用即弃
3. 新计划、新审查、新交付、新即时思路有明确落点
4. 历史记录体系的目标与最小字段已经正式写入规则
