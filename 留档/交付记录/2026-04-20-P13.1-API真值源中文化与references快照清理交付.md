## 1. 问题状态

1. `src/contracts/content-api.contract.ts` 虽已是唯一正式 API 真值源，但原注释仍以英文为主，且 5 个正式 endpoint 的说明不够具体。
2. `references/` 仍保留 `content-api.contract.ts`、`content.port.ts`、`content.http.ts`、`content.service.ts` 代码快照，容易形成第二套接口理解。
3. `references/user-pinned-rules.md` 仍作为旧规则冻结入口停留在当前参考目录中，不符合 `references/` 只保留当前参考索引的口径。
4. 用户复核后指出当前合约仍存在字段注释不清、wire 形态混写、示例不是合法 JSON、用户身份和错误口径不明确等问题，尚不足以让后端直接照文件接入。

## 2. 修复方式

1. 将 `src/contracts/content-api.contract.ts` 中面向后端阅读的注释改为中文。
2. 为 `CONTENT_API_ENDPOINTS` 中 5 个正式接口补齐中文 `summary`、`requestFields`、`responseFields` 和 `notes`。
3. 后续按用户反馈继续加固：补齐后端接入最短路径、统一响应壳示例、每个 endpoint 的 `backendSteps`、`requestExample`、`successExample` 和 `commonMistakes`。
4. 为统一响应壳、target、资源详情、列表、请求 DTO 和 action 结果补字段级中文注释。
5. 保持 5 个正式接口路径、方法、DTO 名称和运行时 adapter 行为不变。
6. 将 `references/` 下的代码快照和旧规则冻结入口退出当前路径，并保留本地归档。
7. 回写 `references/README.md`、`backend-handoff/`、规则体系、总计划和 P12-P16 主线计划，统一说明 `references/` 只保留索引，不再承载接口代码副本。
8. 按用户后续要求加固 `src/contracts/content-api.contract.ts`：补齐资源类型字典、场景块白名单、身份上下文、错误/空数据策略、内容展示元数据归属和 list wire query DTO。
9. 将 endpoint 成功示例改为可解析 JSON，并为 POST 接口补 `requestBodyJsonExample`，由单测直接解析保护。
10. 同步回写 `backend-handoff/` 的阅读和验证口径，只说明从 contract 读什么、怎么验，不复制 API 正文。

## 3. 当前结论

1. 当前唯一正式 API 真值源仍是 [src/contracts/content-api.contract.ts](/H:/工作/天异/uniapp+vue新架构/src/contracts/content-api.contract.ts)。
2. `references/` 不再承载 `contract / port / http / service` 代码快照副本。
3. 后端如需核对真实调用链，应直接查看 `src/ports/content.port.ts`、`src/implementations/content.http.ts`、`src/services/content/content.service.ts` 和相关测试。
4. 真实后端联通仍不在本批范围内，后续仍以真实环境、base URL 和样本参数验证为准。
5. `src/contracts/content-api.contract.ts` 现在不仅声明类型，也直接写明后端实现每个接口应按什么顺序取参、组装返回、避免哪些错误。
6. `GET /api/content/list` 的正式后端 wire query 已明确为扁平字段 `sortField / sortDirection / startDate / endDate`；前端内部 `sort.field / dateRange` 只保留在 port 内部模型。
7. 用户态接口已明确不新增 `userId / accountId`，后端从登录态、token 或 session 获取当前用户。
8. 业务失败口径已明确为 HTTP 2xx + `ContentEnvelope.code != 0`；HTTP 非 2xx 只表达传输、鉴权或服务异常。
