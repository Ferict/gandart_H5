# GitHub 源码交付与后端联调长期规划

> 建档时间：2026-04-04 12:30:00
> 更新时间：2026-04-04 12:30:00
> Encoding: UTF-8
> 文档定位：专项规划文档，用于约束 GitHub 联调源码仓的保留/排除边界、环境说明和后续同步路线

## 1. 目标与结论

当前不再把 GitHub 承载仓定义成“只给后端看契约的文档仓”，而是定义成“后端可直接联调、可下钻真实消费链的前端源码仓”。

本专项固定目标：

1. 给后端一个可直接查看 `src/`、`tests/`、构建配置和统一正文的源码仓。
2. 明确哪些目录必须保留，哪些内部治理材料必须排除。
3. 明确环境与启动说明，避免后端拿到仓库后仍然不知道如何运行或验证。

当前固定结论：

1. GitHub `main` 可以作为后端联调源码仓承载。
2. 交付仓应保留前端源码与必要配置，不再只交文档。
3. 台账、规则、专项档案、垃圾桶、备份等内部治理内容必须排除。
4. 接口真相仍以《统一内容接口契约》和《内容域后端统一移交说明》为准，源码只是联调与核对参考。

## 2. 当前已确认交付口径

### 2.1 必须保留

1. `src/`
2. `tests/`
3. `scripts/`
4. `.github/`
5. 根目录构建与质量配置
6. `统一内容接口契约.md`
7. `内容域后端统一移交说明.md`
8. `backend-handoff/`
9. `references/`

### 2.2 必须排除

1. `审核与修改计划/`
2. `对话台账/`
3. `全量开发规范法则/`
4. `项目台账.md`
5. `垃圾桶/`
6. `备份/`
7. `node_modules/`
8. `dist/`
9. `test-results/`
10. `promptfoo/`
11. `semgrep/`
12. `vale-styles/`
13. `.continue/`
14. `.codex-tmp/`
15. `.tools/`

### 2.3 环境说明要求

交付仓必须同步提供：

1. 推荐 Node 版本
2. 安装命令
3. 启动命令
4. 单测与 E2E 命令
5. 质量门禁命令
6. 联网与代理边界说明

不允许把本机私有环境一并提交，例如：

1. `.env.local`
2. token / key
3. 本机代理账号
4. 个人路径配置

## 3. 近期执行计划

### G1 边界与入口收口

目标：

1. 把 `源码交付白名单.md` 从“最小源码包”升级成“GitHub 联调源码仓白名单”。
2. 把 `backend-handoff` 从“文档仓入口”升级成“源码仓阅读入口”。
3. 补齐联调环境与启动说明。

当前输出：

1. `源码交付白名单.md`
2. `backend-handoff/README.md`
3. `backend-handoff/后端移交内容清单.md`
4. `backend-handoff/后端移交说明模板.md`
5. `backend-handoff/后端联调环境与启动说明.md`

### G2 源码仓实际整理

目标：

1. 按白名单核对当前仓库内容。
2. 形成“可推 GitHub”的实际目录集。
3. 确保内部治理材料未被误带。

执行时必须复核：

1. `src/`、`tests/`、`scripts/` 是否完整。
2. handoff 文档是否与源码现实一致。
3. 白名单与 GitHub 实际目录是否一致。

### G3 GitHub 同步与验证

目标：

1. 将整理后的源码仓内容推送到 GitHub 指定承载仓。
2. 以固定提交号给后端交付。
3. 再次核对 GitHub 页面上的目录与文档入口是否清晰。

必须确认：

1. GitHub 页面可直接看到统一正文与 handoff 入口。
2. `src/` 和 `tests/` 已同步。
3. 内部治理目录没有出现在 GitHub 交付仓里。

### G4 后续维护路线

目标：

1. 后续新增对后端有价值的源码、测试、配置时，同步评估是否进入源码交付白名单。
2. 后续新增治理目录或内部工具时，默认排除，不得自动进入交付仓。
3. GitHub 交付仓与本地主仓若长期分化，必须建立单独的导出流程或同步脚本，不再靠手工口头约定。

## 4. 长远规划

### R1 契约正文与源码仓双入口固定

长期固定为两条入口：

1. 契约与说明入口
   - `统一内容接口契约.md`
   - `内容域后端统一移交说明.md`
2. 源码与验证入口
   - `src/`
   - `tests/`
   - `backend-handoff/`
   - `references/`

### R2 白名单长期治理

后续所有“给外部协作方的 GitHub 交付”都必须先过白名单，而不是先推再解释。

白名单更新顺序固定为：

1. 先确认新增内容是否真的对联调有价值
2. 再更新 `源码交付白名单.md`
3. 再更新 `backend-handoff` 入口说明
4. 最后才允许同步 GitHub

### R3 环境说明长期维护

每次下面内容发生变化，都必须同步更新 `后端联调环境与启动说明.md`：

1. Node 或 npm 要求变化
2. 启动命令变化
3. 测试命令变化
4. 质量门禁变化
5. 代理或联网要求变化

## 5. 当前默认决策

1. GitHub 当前应作为“源码联调仓”，不是“只交契约的文档仓”。
2. 后端应该能看到真实源码，但不应被治理台账和规则目录淹没。
3. `源码交付白名单.md` 是当前唯一正式保留/排除清单。
4. 若后续要真正执行仓库清理或导出，必须以本专项文档和白名单为准，不再临时口头决定。
