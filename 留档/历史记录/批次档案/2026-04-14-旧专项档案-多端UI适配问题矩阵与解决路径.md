## 4. 详细问题与解决路径

### 4.1 顶部安全区、状态栏、导航栏

**典型问题**

- 顶部内容被状态栏或刘海屏遮住
- 自定义导航栏和状态栏之间断层
- 误以为前端可以直接“改系统状态栏”

**同类风险**

- 我们现在会做 H5 + App 壳双环境，最容易把“系统状态栏”和“前端顶部安全区延伸层”混掉。

**解决路径**

1. H5 侧使用 `viewport-fit=cover` 与 `env(safe-area-inset-top)`。
2. 在 uni-app 中，如果 `navigationStyle: custom` 或 `titleNView: false`，必须自行处理顶部状态栏占位。
3. App 沉浸式是宿主能力，不是前端自己“改状态栏”。
4. 前端应统一成：
   - `topChromeMode = web`
   - `topChromeMode = immersive-app`

**uni-app + Vue 落点**

- `pages.json` 中的导航配置
- `App.vue` / 壳层组件中的顶部安全区延伸层
- 运行时配置控制是否启用沉浸式顶栏

**主要责任**

- 前端：样式与安全区消费
- 客户端壳：是否 edge-to-edge / 透明状态栏

**来源**

- uni-app `pages.json`：`https://zh.uniapp.dcloud.io/collocation/pages.html`
- WebKit iPhone X：`https://webkit.org/blog/7929/designing-websites-for-iphone-x/`
- MDN `env()`：`https://developer.mozilla.org/en-US/docs/Web/CSS/env`
- Android Edge-to-edge：`https://developer.android.com/design/ui/mobile/guides/layout-and-content/edge-to-edge`

### 4.2 底部安全区、标签栏、home indicator

**典型问题**

- 底部标签栏或吸底按钮被 home indicator 挡住
- 手势导航区域与自定义底栏冲突

**同类风险**

- 我们项目大量使用自定义底部主标签栏，这个是必中的。

**解决路径**

1. 吸底区域统一接 `env(safe-area-inset-bottom)`。
2. 交互元素本体不要落在系统手势区。
3. Android edge-to-edge 时不能把可点击区直接压到 gesture inset 里。

**uni-app + Vue 落点**

- 共享底部 chrome 组件
- `uni.scss` 或全局共享变量中的 safe-area 变量

**来源**

- Android edge-to-edge：`https://developer.android.com/design/ui/mobile/guides/layout-and-content/edge-to-edge`
- Apple safe area：`https://developer.apple.com/documentation/UIKit/positioning-content-relative-to-the-safe-area`

### 4.3 Android WebView / App 的窗口 inset 传递

**典型问题**

- Android App 里 safe-area 在不同 WebView 版本表现不一致
- 键盘出现后视觉视口和安全区更新不同步

**同类风险**

- 我们后面要给客户端壳工程交接，这一条必须先明确，不然后面会互相甩锅。

**解决路径**

1. 明确告诉客户端壳：WebView 的 inset 传递不是所有版本都一样。
2. Android 官方已说明：
   - M136：fullscreen WebView 才开始转发 `displayCutout/systemBars`
   - M139：IME 通过 visual viewport resize 影响所有 WebView
   - M144：safe-area 才推广到所有 WebView
3. 若宿主提前消费了 insets，WebView 可能拿不到更新。

**uni-app + Vue 落点**

- 壳工程交接文档
- 前端运行时能力探测与降级策略

**责任**

- 客户端壳优先
- 前端做兼容消费与降级

**来源**

- Android WebView window insets：`https://developer.android.com/develop/ui/views/layout/webapps/understand-window-insets`

### 4.4 键盘、VisualViewport 与输入区

**典型问题**

- 输入框被键盘遮住
- fixed 底栏跟键盘打架
- 键盘收起/弹出时页面抖动

**同类风险**

- 设置页、后续表单页、搜索页都会遇到。

**解决路径**

1. 前端区分 `layout viewport` 和 `visual viewport`。
2. 优先用 `window.visualViewport` 处理键盘占位和视口变化。
3. Android 宿主壳建议启用 `adjustResize`，保证键盘动画和 inset 同步。
4. 不要假设所有平台都有 VirtualKeyboard API；它只能作为增强能力。

**uni-app + Vue 落点**

- 输入区 composable
- 页面滚动与定位服务
- 客户端壳交接项：键盘模式

**来源**

- MDN VisualViewport：`https://developer.mozilla.org/en-US/docs/Web/API/VisualViewport`
- Android software keyboard：`https://developer.android.com/develop/ui/views/layout/sw-keyboard`
- Android edge-to-edge setup：`https://developer.android.com/develop/ui/views/layout/edge-to-edge-manually`
- MDN VirtualKeyboard：`https://developer.mozilla.org/en-US/docs/Web/API/VirtualKeyboard/overlaysContent`

### 4.5 动态视口单位：`vh / svh / lvh / dvh`

**典型问题**

- `100vh` 在手机浏览器里高度不稳定
- 顶部/底部工具栏展开收起时页面尺寸跳变

**同类风险**

- 我们有大量全屏壳层和底部栏，必须避免只写 `100vh`。

**解决路径**

1. 优先使用 `dvh / svh / lvh` 理解视口，而不是把 `vh` 当固定真值。
2. `vh` 等同于 `lvh`，在移动端往往不是“当前可见高度”。
3. 需要稳定最小高度时用 `svh` 思路，需要实时跟随时用 `dvh`。

**uni-app + Vue 落点**

- 全局壳层高度基线变量
- 页面主滚动区高度策略

**来源**

- web.dev viewport units：`https://web.dev/blog/viewport-units`
- MDN `<length>` / `dvh`：`https://developer.mozilla.org/en-US/docs/Web/CSS/length`

### 4.6 fixed / sticky / overflow / scroll 容器

**典型问题**

- sticky 失效
- 顶栏吸附不对
- 局部滚动把全页滚动链路打断

**同类风险**

- 我们大量有固定顶栏、底栏和局部横向滚动分区。

**解决路径**

1. `position: sticky` 受最近的带滚动机制祖先影响，祖先一旦有 `overflow` 就要重新判断。
2. 页面应尽量维持单一主滚动容器。
3. 只有明确需要区域滚动时才引入额外 scroll 容器。

**uni-app + Vue 落点**

- 页壳与主滚动区规范
- 页面 section 内的横向滚动区

**来源**

- MDN `position: sticky`：`https://developer.mozilla.org/en-US/docs/Web/CSS/position`
- MDN scroll snap：`https://developer.mozilla.org/en-US/docs/Web/CSS/scroll-snap-type`

### 4.7 uni-app 的 `scroll-view` 与页面滚动差异

**典型问题**

- 页面级下拉刷新、触底、透明标题渐变失效
- 区域滚动和页面滚动混用导致体验断裂

**同类风险**

- 我们如果后面图省事到处用 `scroll-view`，这一条会频繁命中。

**解决路径**

1. 区分“页面滚动”与“区域滚动”。
2. `scroll-view` 只用于确实需要局部滚动的地方。
3. 需要依赖页面滚动链路的页面，不要随意把主体改进 `scroll-view`。

**uni-app + Vue 落点**

- 页面级滚动规则
- 分区横滑规则

**来源**

- uni-app `scroll-view`：`https://zh.uniapp.dcloud.io/component/scroll-view.html`

### 4.8 鼠标、触摸、轨迹板的 hover / focus / active 差异

**典型问题**

- 触摸设备上 hover 样式误触发
- 桌面 Web 缺 hover / focus 反馈
- tooltip 只在鼠标 hover 可见，触摸端不可达

**同类风险**

- 我们同时支持手机、iPad、桌面 Web，这一条是长期问题。

**解决路径**

1. 用 `hover / any-hover / pointer / any-pointer` 区分交互能力。
2. 触摸端优先 `active / pressed`，桌面端补 `hover / focus-visible`。
3. 不要把信息只放在 hover tooltip 里，因为触摸端不可达。

**uni-app + Vue 落点**

- 交互状态语义类
- 组件交互规范

**来源**

- MDN `pointer`：`https://developer.mozilla.org/en-US/docs/Web/CSS/@media/pointer`
- MDN `any-hover`：`https://developer.mozilla.org/en-US/docs/Web/CSS/@media/any-hover`
- MDN tooltip role：`https://developer.mozilla.org/en-US/docs/Web/Accessibility/ARIA/Reference/Roles/tooltip_role`

### 4.9 点击热区、视觉容器、高亮容器

**典型问题**

- 图标在热区里不居中
- 视觉按钮被硬做成 44×44，导致间距崩坏
- hover / active 高亮和真实热区不一致

**同类风险**

- 我们项目已经多次碰到这类问题，属于必写细则。

**解决路径**

1. 区分三层：
   - 物理热区
   - 视觉壳
   - 内容图标/文字
2. 热区至少保证 `44×44`。
3. 视觉壳不必等于热区尺寸，但要始终居中于热区。
4. 高亮容器应与视觉壳语义一致，不能长方形乱套在圆形图标外。
5. Android 官方原生可点击建议是 `48dp × 48dp`。我们当前 WebUI 规范统一采用 `44 × 44px` 作为跨端最低物理热区基线；如果某一端后续发现误触率偏高，应优先把该端专属热区再向 `48dp` 靠拢，而不是缩小热区。

**责任**

- 前端

**来源**

- Apple 触控设计通用建议：`https://developer.apple.com/design/tips/`
- Android target（48dp）：`https://developer.android.com/guide/topics/ui/accessibility/apps`

### 4.10 小字号、最小字号、文字膨胀

**典型问题**

- 10px / 11px 被浏览器或系统放大
- 标签、角标、辅助字被撑爆

**同类风险**

- 我们现在大量有小标签、小英文、小角标，必须稳住。

**解决路径**

1. 常规正文不低于 12px。
2. 特殊装饰性小字也要保证容器和缩放策略稳定。
3. 必要时评估 `text-size-adjust`，但要注意它是实验性能力，不能盲目全局关掉。

**uni-app + Vue 落点**

- 文字与容器细则
- 小标签 / 角标组件规范

**来源**

- MDN `text-size-adjust`：`https://developer.mozilla.org/en-US/docs/Web/CSS/text-size-adjust`

### 4.11 毛玻璃、backdrop-filter 与降级

**典型问题**

- 毛玻璃效果在不同 WebView 上失真
- 背后没有真实内容时像一块发白面板
- 低性能设备掉帧

**同类风险**

- 我们当前 UI 已大量使用“毛玻璃”语义，这一条必须纳入细则。

**解决路径**

1. 毛玻璃要区分：
   - 真的 blur 背景
   - 半透明实底 fallback
2. backdrop-filter 不能作为唯一视觉基础。
3. 前景文本必须在毛玻璃上保持足够对比度。
4. 沉浸式顶部的玻璃效果必须依赖正确的内容层级，而不只是调颜色。

**来源**

- MDN filter / backdrop-filter：`https://developer.mozilla.org/en-US/docs/Web/CSS/CSS_filter_effects/Using_filter_effects`

### 4.12 橡皮筋、scroll chaining、下拉刷新

**典型问题**

- iOS 下拉露白
- 局部滚动带动整页滚动
- 弹层滚到底后把底页带起来

**同类风险**

- 我们有抽屉、局部横滑和长页面，必然会遇到。

**解决路径**

1. 主滚动容器与弹层滚动容器都要明确 `overscroll-behavior`。
2. 不要让多个滚动层默认串联。
3. 若依赖原生下拉刷新，要避免和前端自定义导航、局部滚动直接冲突。

**来源**

- MDN `overscroll-behavior`：`https://developer.mozilla.org/en-US/docs/Web/CSS/overscroll-behavior`
- uni-app `pages.json` 自定义导航说明：`https://zh.uniapp.dcloud.io/collocation/pages.html`

### 4.13 弹层、原生组件、层级冲突

**典型问题**

- popup 被原生层盖住
- nvue 或原生组件与 web 视图层级冲突

**同类风险**

- 如果后续混入 `uni-popup`、原生组件、广告组件或 nvue，这条会立刻出现。

**解决路径**

1. 事先约束“纯 WebUI 页面”与“原生组件混排”的边界。
2. 弹层不能只靠 CSS `z-index` 假设能压过一切。
3. 如确需与原生层混排，必须让客户端壳工程参与方案。

**来源**

- uni-popup：`https://zh.uniapp.dcloud.io/component/uniui/uni-popup.html`
- uni-ad / 原生组件层级说明：`https://zh.uniapp.dcloud.io/uni-ad/ad-component.html`

### 4.14 宽屏主列与 iPad 竖屏宽度分档

**典型问题**

- Web 端内容拉太宽
- iPad 竖屏字小、间距稀、像“大号手机图”

**同类风险**

- 这是我们当前三端适配的主问题之一。

**解决路径**

1. 宽屏舞台和主内容列分层：
   - 舞台层
   - 主列
   - 两侧背景层
2. iPad 竖屏不整页缩放，按宽度分档进入 `single-page-with-nav` 或 `double-page`。
3. 桌面 Web 与 iPad 横屏共用 `wide-stage`。

**来源**

- Android large screens：`https://developer.android.com/guide/topics/large-screens/user-interface`
- Android adaptive quality：`https://developer.android.com/docs/quality-guidelines/adaptive-app-quality`

### 4.15 动画、动效与 reduced motion

**典型问题**

- 动画太多导致眩晕或性能下降
- 桌面和触摸设备反馈不一致

**同类风险**

- 我们当前 UI 有切换动画、角标震动、浮现动画，后面会越来越多。

**解决路径**

1. 所有关键动画都要有低动效降级策略。
2. 使用 `prefers-reduced-motion` 作为浏览器端基础能力。
3. 动画的职责要清楚：
   - 状态反馈
   - 层级引导
   - 上下文切换
     不能把所有元素都做动。

**来源**

- MDN `prefers-reduced-motion`：`https://developer.mozilla.org/en-US/docs/Web/CSS/@media/prefers-reduced-motion`

### 4.16 分屏、多窗口、旋转与窗口重设

**典型问题**

- iPad 分屏后布局突然断裂
- Android 大屏或桌面窗口缩放后主列规则失效
- 横竖屏切换后顶部/底部 chrome 占位错乱

**同类风险**

- 我们后续明确要支持 iPad 竖屏、iPad 横屏和桌面 Web，同一页面在窗口尺寸变化时必须维持同一套适配模型。

**解决路径**

1. 不以“设备名”作为适配核心，而以窗口尺寸和当前 presentation mode 作为唯一判断依据。
2. 任何需要跨端稳定的布局，都必须在窗口变化后重新计算：
   - `presentationMode`
   - `topChromeMode`
   - 主列限宽
   - 宽度分档状态
3. iPad 分屏、桌面窗口缩放、Android 多窗口都要进入同一验证矩阵，不能只测全屏。

**uni-app + Vue 落点**

- 共享壳层的 presentation model 服务
- 全局共享视觉变量（非尺寸 token 层）
- 页面级不要自行猜测设备类型

**来源**

- Android adaptive app quality：`https://developer.android.com/docs/quality-guidelines/adaptive-app-quality`
- Android large screens UI：`https://developer.android.com/guide/topics/large-screens/user-interface`

### 4.17 桌面 Web 的焦点、键盘与鼠标路径

**典型问题**

- 桌面端只有 hover，没有可见 focus
- 鼠标与触摸走不同状态机，导致高亮不一致
- 键盘导航无法清楚感知当前焦点

**同类风险**

- 我们未来桌面 Web 会成为正式浏览入口，不只是调试壳。只做触摸态会导致桌面体验明显“像劣质手机页搬运”。

**解决路径**

1. 组件至少要区分：
   - `hover`
   - `active / pressed`
   - `focus-visible`
2. 焦点态应与视觉壳兼容，不能只依赖浏览器默认 outline，也不能完全去掉 outline 却不补替代样式。
3. 交互说明不能只藏在 hover 中；触摸端不可达的信息要有替代表达。

**uni-app + Vue 落点**

- 共享交互状态类
- 组件级焦点态约束
- 桌面 Web 样式分支仅允许在交互态层面出现，不能把页面结构做成双份

**来源**

- MDN `:focus-visible`：`https://developer.mozilla.org/en-US/docs/Web/CSS/:focus-visible`
- MDN `@media (pointer)`：`https://developer.mozilla.org/en-US/docs/Web/CSS/@media/pointer`
- MDN `@media (any-hover)`：`https://developer.mozilla.org/en-US/docs/Web/CSS/@media/any-hover`
