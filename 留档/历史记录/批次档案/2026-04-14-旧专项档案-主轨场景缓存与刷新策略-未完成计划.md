# 主轨场景缓存与刷新策略-未完成计划

更新时间：2026-03-28 16:30:14

## 1. 当前已落地口径

- `home`、`activity`、`profile` 三个场景服务已支持 `resolve...Content(options?: { force?: boolean })`
- 默认调用优先复用同场景 in-flight promise，避免同一轮重复打服务
- `force: true` 用于下拉刷新，允许当前页主动绕过已有 in-flight 并发起一次新请求

## 2. 后续缓存分层

### A. 静态配置层

- 内容：标签、排序项、入口文案、placeholder/icon 配置、分类元数据
- 建议：磁盘 TTL `24h`
- 策略：启动可直接读盘，页面首开不阻塞，再做后台静默校验

### B. 半动态运营层

- 内容：banner、公告列表文案、首发卡视觉与说明、活动列表结构、个人中心分类结构
- 建议：软 TTL `10min`，硬 TTL `1h`
- 策略：优先读缓存渲染，进页和切回前台时后台 revalidate

### C. 高频市场 / 时间层

- 内容：地板价、市场价格、首发藏品时间、活动时间、倒计时相关字段
- 建议：软 TTL `30s`，硬 TTL `5min`
- 策略：始终按 stale-while-revalidate 处理；下拉刷新强制重拉

### D. 用户敏感状态层

- 内容：未读数、提醒态、资产总额、持仓汇总、登录态相关派生数据
- 建议：先不做磁盘持久化，保留内存态 + 请求去重
- 策略：进页主动校验；下拉刷新强制重拉；失败时继续保留当前页已显示内容

## 3. 预期实现骨架

- 新增主轨场景缓存仓：按 `sceneId` 分 bucket，至少包含 `payload`、`updatedAt`、`softExpiresAt`、`hardExpiresAt`
- 服务层统一入口：
  - 普通进页：`cache first + stale-while-revalidate`
  - 下拉刷新：`force network`，但失败时允许回退到现有缓存内容
- 缓存命中后先适配为页面内容模型，再异步对比新旧快照，避免页面无意义闪动

## 4. 这轮先不做的事

- 不在本轮把磁盘缓存、TTL 清理、前后台切换重校验全部一起上
- 不在本轮扩散到详情页、设置页、service hub 之外的独立页面
- 不把市场高频字段和静态配置混用同一 TTL

## 5. 验收口径

- 后续真正落缓存时，普通切页不应重复拉取可缓存静态块
- 下拉刷新必须只刷新当前页，并能绕过软缓存
- 地板价、首发时间、活动时间这类字段不得长期停留在过期缓存
- 请求失败时页面继续保留当前可见内容，不回落空白 shell
