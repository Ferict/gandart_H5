## 2. 决策项清单

### 2.1 决策项 A：单位体系最终基线

- 决策问题：规范层是否继续只用 `px + 宽度驱动`，还是保留极小范围尺寸 token。
- 影响范围：
  - `开发规范/UI与适配规范.md`
  - `开发规范/工程与流程规范.md`
  - 后续页面尺寸写法与评审口径
- 责任方：
  - 方案整理：前端规范负责人（当前由我执行）
  - 落地执行：前端开发
  - 最终拍板：用户
- 通俗解释：就是“尺寸怎么写”。选纯 `px` 最直接；保留小范围 token 可以减少重复改值。
- 选项：
  - 选项 A：只保留 `px + 宽度驱动`，不再引入 token 尺寸层
    - 优点：简单、直接、认知成本最低
    - 风险：同类尺寸改动时容易多点手改
  - 选项 B：保留极小范围 token（仅全局公共尺寸），其余仍用 `px`
    - 优点：全局调节更快，规范弹性更好
    - 风险：会增加一层规则和维护成本
- 当前建议：选项 A（你当前多次强调链路简洁、直接）。
- 用户批复：`选项 A：只保留 px + 宽度驱动，不再引入尺寸 token 层`

### 2.2 决策项 B：对话台账记录粒度

- 决策问题：对话台账记录到“关键进展”，还是“每条消息逐段记录”。
- 影响范围：
  - `对话台账/对话台账-YYYYMMDD.md`
  - `开发规范/工程与流程规范.md`
  - 台账维护成本与可追溯深度
- 责任方：
  - 方案整理：前端规范负责人（当前由我执行）
  - 落地执行：前端开发/助手
  - 最终拍板：用户
- 通俗解释：就是台账记多细。越细越可追溯，但维护越重。
- 选项：
  - 选项 A：关键进展粒度（节点化记录）
    - 优点：维护轻，主线清晰
    - 风险：部分细节追溯不足
  - 选项 B：每条消息逐段完整粒度
    - 优点：追溯最完整
    - 风险：台账增长极快，维护负担大
- 当前建议：选项 A（同时保留关键争议点逐段记录作为例外）。
- 用户批复：`选项 B：每条消息逐段完整粒度`

### 2.3 决策项 C：`scroll snap` 适用范围

- 决策问题：`scroll snap` 仅用于横向标签条，还是扩展到更多横滑模块。
- 影响范围：
  - `开发规范/UI与适配规范.md`
  - 横向滚动组件交互一致性
  - 后续性能与可控性
- 责任方：
  - 方案整理：前端规范负责人（当前由我执行）
  - 落地执行：前端开发
  - 最终拍板：用户
- 通俗解释：就是横滑时是否统一“磁吸停靠”。范围越大，体验统一度越高，但约束也更强。
- 选项：
  - 选项 A：只在横向标签条启用
    - 优点：风险低、成本低
    - 风险：不同横滑模块体验不统一
  - 选项 B：扩展到所有横滑内容模块（可按白名单）
    - 优点：体验一致
    - 风险：个别模块可能不适合强磁吸，需要白名单豁免
- 当前建议：选项 B（白名单启用，不一刀切）。
- 用户批复：`自定义：当前不扩展，维持现行“仅横向标签条/导航性横滑区”口径；后续如需扩展再单开白名单批次`

### 2.4 决策项 D：Web 与大屏适配口径

- 决策问题：保持“宽度分档 + 页面轨道”，还是回退到“单栏 + 限宽”。
- 影响范围：
  - `开发规范/UI与适配规范.md`
  - `src/pages/**` 页面布局策略
  - 未来平板/桌面 Web 体验一致性
- 责任方：
  - 方案整理：前端规范负责人（当前由我执行）
  - 落地执行：前端开发
  - 最终拍板：用户
- 通俗解释：大屏上是“多轨并排展示”还是“手机单栏放大看”。前者信息密度高，后者实现最稳。
- 选项：
  - 选项 A：宽度分档 + 页面轨道
    - 优点：大屏利用率高，扩展到折叠屏更自然
    - 风险：实现复杂度更高，规则需要更严
  - 选项 B：单栏 + 限宽
    - 优点：实现简单，回归快
    - 风险：大屏空间浪费，平板体验上限低
- 当前建议：选项 A（与你当前“按页面宽度适配、不按设备名分端”方向一致）。
- 用户批复：`自定义：按宽度分档 + 页面轨道执行，并以下述细分规则作为正式口径`

正式执行说明：

1. 旧的“单页 + 抽屉、平板竖屏单页 + 独立侧栏”只保留为当前过渡基线，不再作为长期主方案继续补丁式扩展。
2. 新主方案改成统一的“页面轨道模式”，主判断不再看设备名，只看页面总可用宽度。
3. 折叠屏 / 双面屏 / 三折叠屏不是单独设备类型，只在宽度判断外增加 `viewport segment` 安全边界修正。

正式展示模式：

1. `single-page`：总宽 `< 430`
2. `single-page-with-nav`：`430 <= 总宽 < 750`
3. `double-page`：`750 <= 总宽 < 1125`
4. `triple-page`：总宽 `>= 1125`
5. `triple-page-with-stage`：当三页主轨道已经达到单页上限且仍有剩余宽度时，额外宽度只做舞台背景填充

每种模式的正式表现：

1. `single-page`：单页，保留底部导航，保留抽屉
2. `single-page-with-nav`：左侧一个页面，宽度最大 `430`；右侧为独立导航轨道；保留底部导航；取消抽屉
3. `double-page`：默认显示两页；左页固定首页，右页在活动页 / 个人页之间切换；保留底部导航
4. `triple-page`：首页 + 活动页 + 个人页三页并排；取消底部导航
5. `triple-page-with-stage`：三页并排；外围剩余空间只做舞台背景，不再放大主内容

轨道模型：

1. 首页轨道
2. 活动页轨道
3. 个人页轨道
4. 导航轨道
5. 舞台背景轨道

折叠屏兼容规则：

1. 主判断仍然是总宽度
2. 如果支持 `viewport segments`，横向双分段优先进入双页模式
3. 如果支持 `viewport segments`，横向三分段优先进入三页模式
4. 纵向分段仍按单页类模式处理，不做上下双页
5. 页面内容轨道不能跨铰链 / 折痕硬铺开

滚动与交互规则：

1. 每个页面轨道各自滚动
2. 导航轨道不承担正文滚动
3. 舞台背景层不滚动
4. 抽屉只保留在 `single-page`
5. `single-page-with-nav` 不再使用抽屉
6. 三页模式取消底部导航
