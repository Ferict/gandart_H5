# 计划 - 单位体系细则与项目骨架规则落地及专项启动

## 1. 目标

1. 落地 `单位体系细则`
2. 落地 `正式项目骨架规则`
3. 把两份新规则接回主规范入口
4. 为 `UI 视觉专项优化` 与 `环境部署` 建立正式专题与待决清单

## 2. 允许修改范围

- `开发规范/`
- `审核与修改计划/`
- `项目索引/`
- `提交清单/`
- `项目台账.md`
- `对话台账/`

## 3. 明确不改范围

- 不改 `../旧架构文件/**`
- 不改新仓业务源码
- 不创建正式 uni-app 代码骨架
- 不安装依赖
- 不做任何打包、构建、原生壳接入

## 4. 批次顺序

1. 落两份新规则正文
2. 回改 `README.md` 与 `前端开发规范整理版.md`
3. 建立 `UI 视觉专项优化` 与 `环境部署` 专题文档
4. 将所有待决事项写入台账
5. 更新索引、当前计划、提交清单、对话台账
6. Git 提交

## 5. 风险

1. 若在本批擅自决定工具链，会违反“待决事项不得自行拍板”的要求。
2. 若本批直接开始建源码骨架，会把“规则批次”和“环境/源码批次”混在一起。

## 6. 验证方式

1. 新规则是否已接回规范入口
2. 待决事项是否明确写入台账
3. 本批是否只动规范与管理文件
4. `git status` 是否边界清晰

## 7. 下一批衔接

1. 先根据用户决策收口环境部署方案
2. 再开始正式项目骨架落地
3. `UI 视觉专项优化` 按页面逐页审查推进
