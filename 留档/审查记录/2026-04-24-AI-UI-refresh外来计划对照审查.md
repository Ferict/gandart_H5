# 2026-04-24 AI UI Refresh 外来计划对照审查

> 文档类型：审查记录
> 状态：active
> 更新时间：2026-04-24
> 说明：本文用于把外来 “AI UI Refresh Pipeline” 计划与当前仓现实结构做一轮对照，判断哪些可直接采用、哪些必须改写后采用、哪些当前不适配。

## 1. 审查目标

本次审查只回答三件事：

1. 外来计划与当前仓的匹配度如何。
2. 当前仓若向该目标靠拢，优先应治理什么。
3. 哪些条目属于可保留的治理意图，哪些条目属于旧项目假设，不应直接照搬。

## 2. 审查对象

本次审查对象包括：

1. 用户提供的 “AI UI Refresh Pipeline 执行计划”
2. 当前仓源码结构
3. 当前仓规则、口径与留档体系

## 4. 已确认可直接采用的部分

以下内容与当前仓目标一致，可直接吸收为治理方向：

1. 先做文档与边界治理，再动大规模 UI。
2. 不默认改全部 `.vue` 文件，而是先做页面与组件分级。
3. UI 改造必须保护请求链、接口契约、页面取数和敏感逻辑。
4. token 与高复用组件应先行治理，不应直接逐页散改。
5. 敏感文件需要阻断规则。
6. 在正式 skill 投产前先做试跑和验证，这个顺序是正确的。

## 5. 已确认不能直接照搬的部分

以下内容与当前仓现实明显不匹配，不能直接采用：

1. 外来计划假设的是大体量旧项目：
   - 225 个 `.vue`
   - 98 条路由
   - 大量 `pages/**` 与 `components/w-*`
   - 当前仓现实是 56 个 `.vue`、3 个正式注册页面、10 个全局组件
2. 外来计划假设的是 Vue 2 + `@/api` 读取链。
   - 当前仓正式请求链是 `contracts -> ports -> implementations -> services -> main`
3. 外来计划假设 `docs/ai-ui-refresh/` 是默认文档落点。
   - 当前仓已有正式 `规则体系/口径/留档` 体系，且新顶层目录需单独许可
4. 外来计划假设当前没有可复现的 npm H5 构建脚本。
   - 当前仓 `package.json` 已有 `dev:h5`、`build:h5`、`build:app`
   - `npm run build:h5` 已可执行通过
5. 外来计划中的首批页面、首批组件、敏感页面名单，几乎全部对应旧仓目录，不适用于当前仓

## 6. 需要改写后采用的部分

以下内容方向可保留，但必须改写成当前仓版本后才能落地：

1. Phase 1 文档体系
   - 保留 “先沉淀治理文档” 的方向
   - 但默认落点改为：`规则体系/口径/` + `留档/`
2. Phase 2 Vue 文件分级
   - 保留 P0/P1/P2/P3/P4 分级思想
   - 但分级对象必须重建为当前仓页面与组件清单
3. Phase 3 页面契约与接口消费说明
   - 保留“字段置信度”思路
   - 但不能替代正式 API 真值源；只能作为消费侧契约补充
4. Phase 4 token 与组件策略
   - 保留 “先 token、再高复用组件、再页面回归” 的顺序
   - 但 token 起点必须换成当前仓的 `src/uni.scss` 与现有组件体系
5. Phase 5 构建与验证策略
   - 保留 “H5 / App / 宽度回归 / 敏感文件阻断” 的验证框架
   - 但脚本、平台现实与敏感文件名单必须换成当前仓版本
6. Phase 6 试跑
   - 保留 “先小范围试跑” 的思路
   - 但试跑页必须换成当前仓注册页面与核心 panel
7. Phase 7 以后 skill 化阶段
   - 保留“先 draft、后正式”的顺序
   - 但当前阶段不直接创建正式 skill，也不先建新顶层 skill 文档目录

## 7. 对当前仓的已确认问题

对照新增规则后，当前仓至少存在以下专项治理缺口：

1. `src/adapters/` 仍为空壳，与 “UI / 请求彻底解耦” 目标不一致。
2. 当前还没有正式的 AI UI Refresh 页面分级清单。
3. 当前还没有正式的页面消费契约清单和字段置信度清单。
4. 当前还没有把 “Full Project Mode / Selected Pages Mode” 改写成适用于本仓的现行口径。
5. 当前还没有针对本仓的 UI 刷新敏感文件阻断清单与试跑入口清单。

若后续向该目标推进，影响范围至少包括：

## 9. 风险与遗留项

当前主要风险如下：

1. 如果直接照搬旧仓页面与组件名单，会把当前仓带回错误基线。
2. 如果把页面消费说明写成“新的正式接口文档”，会冲掉现有单一契约真值源。
3. 如果在未完成页面分级前直接全仓 UI 改造，容易把视觉改动扩散进请求链和 provider seam。
4. 如果先建 `docs/ai-ui-refresh/` 顶层目录，会违反当前目录治理基线。
