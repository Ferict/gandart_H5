这代表当前至少存在 `12` 个当前路径文件共同承载 API 真值、解释或 handoff 语义。
这不满足“后端只面对一个源文件即可接上”的要求。

### 3.3 单一源文件迁移成本

[src/contracts/content-api.contract.ts](/H:/工作/天异/uniapp+vue新架构/src/contracts/content-api.contract.ts) 当前被 `32` 个 `src/` 源码文件直接消费。
同时，当前 [tsconfig.json](/H:/工作/天异/uniapp+vue新架构/tsconfig.json) 的 `include` 只覆盖 `src/**`。

结论：

1. 如果把唯一真值源迁出 `src/`，需要同步改编译边界与大量导入。
2. 如果以最小风险收口，默认应继续以 `src/contracts/content-api.contract.ts` 作为最终唯一正式真值源。

[规则体系/05-接口与联调规则.md](/H:/工作/天异/uniapp+vue新架构/规则体系/05-接口与联调规则.md) 当前仍把以下内容视为正式真值源或过渡主入口：

这与本次新增目标冲突：

1. API 应统一到一个真实源文件。
2. 后端不应继续在多个当前路径之间拼装理解。
3. 当前路径不应继续保留多套并行入口。

### 3.5 “真实留出接口”仍未闭环

[scripts/check-content-http-link.mjs](/H:/工作/天异/uniapp+vue新架构/scripts/check-content-http-link.mjs) 已提供统一验证入口，但当前默认仍依赖 `scene / list` 最小验证和额外样本参数。
所以当前仍不能宣称：

1. 五个正式接口都已通过真实后端联通验证。
2. 后端只靠当前一个文件就能无歧义完成接入。

## 4. 审查结论

### 4.1 已确认问题

1. 当前 API 真值源仍然分裂，不符合“一个真实源文件”的目标。
2. 当前仓级目录仍有大量过渡、历史和生成路径并存，不符合“当前路径干净无歧义”的目标。
3. 现行规则、接口目录和后端 handoff 文档需要整体改口，不能继续沿用“多文件共同真值”的说法。

### 4.2 默认决策

本轮默认决策固定为：

1. **唯一正式 API 真值源默认定为** [src/contracts/content-api.contract.ts](/H:/工作/天异/uniapp+vue新架构/src/contracts/content-api.contract.ts)。
2. **`接口契约/` 不再作为多文件真值目录继续扩写**；该目录后续要么退出当前路径，要么降为非真值辅助层，但不再承载正式分裂真值。
3. **`统一内容接口契约.md`、`内容域后端统一移交说明.md` 不再保留为当前路径正式 API 真值源**；后续进入历史或垃圾桶，或仅保留短暂迁移态。
4. **`backend-handoff/` 仅保留联调环境、启动方式、验证入口等 handoff 信息**，不再承载合同式 API 真值正文。
5. **新阶段开始前先完成 `P10` 的边界收口**，否则继续在 `641` 条未提交改动上叠加新 API 主线是不负责任的。

## 5. 后续建议

建议立刻进入新的顶层阶段：

1. 先用一个新阶段计划把 API 单一真值源、hand-off 收口、当前路径去歧义和仓级清洁收口写成同一条剩余主线。
2. 新阶段只在用户过目并确认默认决策后再开工实现。
