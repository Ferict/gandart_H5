## 2. 观察结果

### 2.1 `home` 当前剩余复杂度主要集中在四簇

1. 切换入口与立即替换路径：
   - `applyResolvedMarketListResult`
   - `replaceMarketCollectionImmediately`
   - `queueMarketResultSwitch`
2. `switch / queue` 主编排：
   - `startMarketResultSwitch`
   - `flushQueuedMarketSwitch`
3. `enter / retained replay / queued insertion`：
   - `startMarketEnterPhase`
   - `scheduleMarketRetainedReplayEnter`
   - `scheduleMarketQueuedInsertions`
   - `finalizeMarketEnter`
4. 清理与 inactive reset：
   - `resetHomeMarketResultWindowForInactive`
   - 多个 `clear*Timeout`

### 2.2 `profile` 复杂度比 `home` 更偏 refresh replay

1. 除了 `switch / queue / enter` 外，`profile` 还叠加了：
   - `requestProfileAssetRefreshReplay`
   - `reconcileProfileAssetRender`
   - `applyResolvedProfileAssetList`
2. 也就是 `profile` 的危险点不只是动画切换，而是：
   - 手动查询切换
   - refresh replay
   - retained image changed
     三者耦合在一起

### 2.3 `activity` 的时序层最简单

1. `activity` 的 `startNoticeResultSwitch` 是当前最短、最线性的实现。
2. 它更像是高耦合层的参考样板，而不是优先重构对象。

## 3. 当前判断

1. 下一批如果继续切时序层，第一刀不该从 `profile` 开始。
2. 也不该一上来拆 `retained replay`，因为这是最容易引入视觉回退的一层。
3. 当前最稳的第一刀应先切出 `home` 的：
   - 切换入口与立即替换路径
   - `switch / queue` 主编排
     让 `enter/replay` 继续留在主文件

## 4. 建议顺序

1. `P4.10` 第一刀：
   - `home` 的 `switch gateway / queue runtime`
2. `P4.10` 第二刀：
   - `profile` 的 `switch gateway / refresh replay gateway`
3. `activity` 保持参考实现，不作为当前主改动目标

## 5. 结论

1. 进入 `P4.10` 后，首刀目标应固定在 `home` 的切换入口与 queue 主编排。
2. 在这一刀完成前，不建议动：
   - `scheduleMarketRetainedReplayEnter`
   - `scheduleProfileAssetRetainedReplayEnter`
   - `activity` 的线性切换实现
