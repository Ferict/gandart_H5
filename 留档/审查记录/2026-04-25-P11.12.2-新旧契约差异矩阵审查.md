## 2. 已确认事实

当前事实：

1. 当前前端内容域契约只有 5 个正式接口。
2. 当前 `ContentPort` 把 UI 消费稳定在 `getScene / getResource / getList / runAction`。
3. 当前 `content.http.ts` 只实现 `/api/content/*` 抽象接口。
4. 项目级后端契约冻结基线是 190 个旧 H5 业务 wrapper。
5. 旧 H5 后端 wrapper 大多是具体 domain POST 接口，不是 scene/resource/list/action 抽象接口。
6. 当前 adapter 已经承担 DTO 到 home/profile/detail view-model 的归一化职责。

## 3. 审查结论

结论：

1. 两套契约不是同构替换关系。
2. 正确方向不是让页面直接消费 190 个 wrapper。
3. 正确方向是保留当前 `ContentPort` 和稳定 view-model，在 implementation/adapter/runtime-service 层做 wrapper 组合与字段归一化。
4. `notice-read` 和 `service-reminder-consume` 不能直接用旧 wrapper 硬接，必须等待确认。
5. `profile-asset-detail` 的五卡片字段可能存在后端字段缺口，需要在后续迁移线单独确认。

## 4. 风险

当前风险：

1. `高风险`
   - 如果直接把旧 H5 wrapper import 到页面，会破坏 UI 与请求链解耦。
2. `高风险`
   - 如果不先裁决 implementation/adapter/runtime/service 责任层，后续会在各页面重复造映射。
3. `中风险`
   - 活动、公告、资讯、抽签旧接口之间语义重叠，可能导致 activity 线选错来源。
4. `中风险`
   - profile_asset 主列表候选 wrapper 不止一个，需要先确认首选入口。
