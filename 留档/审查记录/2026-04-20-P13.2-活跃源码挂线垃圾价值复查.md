## 1. 本轮修正点

P13 第一轮把“仍在 `pages.json` 注册、仍被导航构建函数引用”作为强保留理由，结论偏保守。

本轮修正为：

1. 被路由或导航挂到，只说明直接移走会有破坏风险。
2. 被挂到不等于有业务价值。
3. 如果某段代码只是为了让 AI 生成的占位入口不报错，且没有用户确认的真实页面目标，应列为“挂线垃圾候选”。
4. 挂线垃圾候选不能直接删除；后续必须先拆线、验证，再按计划转移到 `垃圾桶/`。
5. 活跃但过度拆分、过度抽象的代码不应直接送桶，应进入简化或归并重构计划。

## 2. 新判定标准

### 2.1 `keep`

满足以下任一条件，才算真正保留：

1. 支撑当前用户明确要求的主流程。
2. 支撑当前可见、可操作、已验收的页面功能。
3. 属于正式后端接口真值、真实 mock/http provider 或必要测试链。
4. 已有明确后续产品目标，且短期不能替代。

### 2.2 `挂线垃圾候选`

满足以下特征，应列为治理候选：

1. 代码仍被 route、导航、mock 或 schema 接上。
2. 实际页面只跳转到 construction 或 updating。
3. 没有真实业务渲染、真实接口消费或明确用户需求。
4. 保留理由只是“不拆会影响当前接线”。

### 2.3 `defer`

只有在确实需要用户决定产品方向时才标记为 `defer`。不能把“我不确定”或“被挂到了”都写成 `defer`。

### 2.4 `refactor`

如果代码真实参与当前主页面，但文件拆分过细、抽象过重、命名或落点不利于维护，应标为 `refactor`，不应直接作为垃圾桶对象处理。

## 3. 复查结果

### 3.1 仍然确认 `keep`

1. `src/pages/home/`
   - 当前主页面，不能在 P13 作为垃圾清理对象处理。
   - 但其内部大量 page-local composable 和组件拆分较重，后续应进入“首页代码简化/归并”主题，不属于本轮直接送桶。
2. `src/pages/profile-asset-detail/`
   - 当前有真实页面、真实路由参数、页面 runtime、组件和测试覆盖。
   - 保留。
3. `src/contracts/content-api.contract.ts`
   - 唯一正式 API 真值源。
   - 保留。
4. `src/implementations/content.mock.ts`、`src/implementations/content.http.ts`、`src/services/content/`、`src/ports/content.port.ts`
   - 当前承接真实内容域 mock/http provider 和后端接口预留。
   - 保留，但后续可以做结构简化，不按垃圾处理。

### 3.2 挂线垃圾候选

以下对象当前被接入，但业务价值不成立。它们不是“有用”，只是“直接拿走会影响现有接线”。

1. `src/pages/settings/`
   - 当前只 redirect 到 `updating`，没有设置页实现。
   - 候选处理：确认是否保留设置入口；如不保留，先从导航构建与 `pages.json` 拆线，再转移。
2. `src/pages/service-entry/`
   - 当前只 redirect 到 `updating`，没有服务入口实现。
   - 候选处理：确认抽屉服务入口是否当前需要；如不需要，拆线后转移。
3. `src/pages/action-entry/`
   - 当前是泛 action 占位入口，没有真实 action 页面。
   - 候选处理：优先移除泛 action 中转页，而不是继续为它维护 route query schema。
4. `src/pages/content-resource/`
   - 当前是泛内容资源占位入口，没有真实内容资源详情页。
   - 候选处理：真实内容详情未来应单独设计，不应靠当前泛占位页冒充。
5. `src/pages/profile-address/`
   - 当前只 redirect 到 `updating`，没有地址或二维码页面实现。
   - 候选处理：若当前没有账户地址页需求，拆线后转移。
6. `src/pages/profile-assets/`
   - 当前只 redirect 到 `updating`；资产列表目前主要在首页 profile rail 内呈现。
   - 候选处理：如果不做独立资产列表页，拆线后转移。
7. `src/pages/notice-detail/`
   - 当前只 redirect 到 `updating`；公告详情内容链没有接入页面。
   - 用户已确认：notice 预留内容链标记为需要移除。
   - 候选处理：后续先拆 `notice-detail` 与 notice preset 内容链，再验证并转移。
8. `src/pages/updating/`
   - 当前是 construction 占位页本体。
   - 用户已确认：建设中展示收为一个可引用组件，通过文案适配未完成入口；仓促上线时该能力可能继续使用。
   - 候选处理：暂留统一承载页，但页面本身不再复制建设中 UI 细节。
9. `src/services/page-entry/`
   - 当前主要服务上述占位入口。
   - `parseProfileAssetDetailRouteQuery` 是实际详情页仍用的部分；其余解析和 redirect service 多数跟占位入口绑定。
   - 候选处理：先拆占位入口，再把详情页 query 解析迁回 `profile-asset-detail` 邻近目录，最后评估本目录是否整体转移。
10. `src/ports/pageContent.port.ts`
    - 当前不是正式后端 API，不是内容域主接口，只服务占位内容。
    - 候选处理：如果不保留 `updating` 的复杂 preset，转移。
11. `src/adapters/pageContent.mock.adapter.ts`
    - 当前只是占位页文案 preset 和公告详情假内容。
    - 候选处理：如果不做公告占位详情、不保留复杂 construction 文案，转移。
12. `src/models/route-query/routeQuery.model.ts` 与 `src/models/route-query/routeQuery.schema.ts` 中的次级页 query 类型
    - 当前大量 query 类型是为了占位页和中转页存在。
    - 候选处理：拆线后删除对应类型，只保留真实页面需要的 query 模型。

## 4. 第一批建议治理顺序

不直接动源码，先请用户确认以下方向：

1. 当前是否保留独立 `settings`、`service-entry`、`action-entry`、`content-resource`、`profile-address`、`profile-assets` 页面入口。
2. 除 notice 之外的占位入口是否全部拆线。
3. 已确认统一建设中展示保留为可复用组件，`updating` 只作为可能的统一承载页。

确认后按批次处理：

1. 第一批：从导航构建函数和页面入口说明中拆掉不需要的入口。
2. 第二批：更新 `pages.json`，移除不再需要注册的页面。
3. 第三批：把对应页面目录、page-entry 旁路线和 pageContent 占位内容按计划转移到 `垃圾桶/`。
4. 第四批：精简 route query 模型和测试。
5. 第五批：跑 `type-check`、`test:unit`、`knip`、`build:h5`，必要时补 E2E。

## 5. 本轮不做

1. 不执行删除。
2. 不转移文件到 `垃圾桶/`。
3. 不改业务源码。
4. 不把“仍被挂到”写成“有用”。
5. 不把首页内部过度拆分问题混入本轮直接垃圾清理；它应进入后续简化重构。
