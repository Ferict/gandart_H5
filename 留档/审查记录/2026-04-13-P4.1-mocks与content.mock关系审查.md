# 2026-04-13 P4.1 mocks 与 content.mock 关系审查

> 文档类型：审查记录
> 状态：active
> 更新时间：2026-04-13
> 说明：本审查用于判断 `src/mocks/` 与 `src/implementations/content.mock/` 是否属于同一语义分裂在两处，以及是否需要在当前批次立即做物理治理。

## 1. 审查范围

本次只审查：

1. [src/mocks](/H:/工作/天异/uniapp+vue新架构/src/mocks/README.md)。
2. [src/implementations/content.mock](/H:/工作/天异/uniapp+vue新架构/src/implementations/content.mock/README.md)。
3. `content.mock.ts` 与两处目录的职责边界。

本次不做：

1. 目录物理迁移。
2. mock 数据改名。
3. provider 接口改签名。

## 2. 当前现状

`src/mocks/` 当前承载的是静态 mock 数据库：

1. `content-db/`
2. `content-db/scenes/`
3. 各类 `assets / categories / drops / market-items / notices / service-*` 数据文件。

`src/implementations/content.mock/` 当前承载的是 mock provider 的实现层：

1. `scene.ts`
2. `resource.ts`
3. `list.ts`
4. `action.ts`
5. `shared.ts`

而 [src/implementations/content.mock.ts](/H:/工作/天异/uniapp+vue新架构/src/implementations/content.mock.ts) 作为总入口，将上述实现层收口成统一 mock implementation。

## 3. 依赖关系

当前依赖方向是单向的：

1. `src/implementations/content.mock/*` 读取 `src/mocks/content-db/*`。
2. `src/services/content/content.service.ts` 与少量内容域 service 只依赖 `content.mock.ts` 入口，不直接读 `src/mocks/`。

这说明两处目录当前不是“同一职责重复摆放”，而是：

1. `src/mocks/`
   - 静态数据源层。
2. `src/implementations/content.mock/`
   - mock provider 组装实现层。

## 4. 审查结论

当前这两处目录虽然分开，但职责边界是成立的。

问题不在于“必须立即合并”，而在于：

1. 目录名容易让第一次接手的人误以为两边都在放 mock 实现。
2. 只有在同时阅读两个 README 和 `content.mock.ts` 后，才能完全理解数据层与实现层的区别。

但从结构质量看，当前不属于必须立即治理的重复目录。

## 5. 建议

本批次建议：

1. 保持当前双层结构不动。
2. 继续将其视为“数据源层 + 实现层”的合法分层。
3. 不把这条线插入当前 `P4.1` 物理治理主线。

若后续要继续优化，只建议在更大的命名重构批次里评估：

1. 是否把 `src/mocks/` 改名为更直白的 `src/mock-data/` 或等价目录。
2. 是否把 `content.mock.ts` 与 `src/implementations/content.mock/` 的入口说明再收紧。

## 6. 结论

`src/mocks/` 与 `src/implementations/content.mock/` 当前不需要立即做物理迁移或合并。

当前更值得继续推进的结构治理热区仍是：

1. `src/pages/` 路由壳一致性。
2. `src/composables/` 与 `src/services/` 的职责边界复核。
