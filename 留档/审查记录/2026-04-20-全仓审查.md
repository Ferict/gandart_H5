## 5. 已确认问题

### 5.1 P2：交付白名单会切掉仍被 CI/脚本依赖的目录

证据：

1. `源码交付白名单.md` 第 50-55 行保留 `.github/**`、`scripts/**`、`src/**`、`tests/**`。
2. `源码交付白名单.md` 第 74-79 行明确排除 `promptfoo/**`、`semgrep/**`、`vale-styles/**`、`.continue/**`、`.tools/**`。
3. `package.json` 第 19-24 行仍让 format/docs 扫描 `promptfoo/`、`.continue/checks/`、`semgrep/`、`vale-styles/`。
4. `.github/workflows/quality-gate.yml` 第 38-82 行仍运行 docs、lint、Semgrep、Vale 与 Promptfoo smoke。
5. `scripts/quality/run-semgrep.mjs` 第 26 行读取 `semgrep/rules`。
6. `scripts/quality/run-vale.mjs` 第 5 行读取 `.continue/checks` 和 `promptfoo`，`.vale.ini` 依赖 `vale-styles`。
7. `scripts/quality/run-promptfoo-smoke.mjs` 第 15-21 行读取 `promptfoo/promptfooconfig.yaml`。

影响：

1. 当前本地仓内这些目录存在，所以当前质量门禁可以通过。
2. 但按白名单裁剪交付后，交付仓会缺质量规则目录和 Promptfoo 配置，CI 或本地质量脚本会失败。

后续建议：

1. 要么把这些质量规则目录纳入 GitHub 联调源码仓。
2. 要么把内部治理检查从交付仓 workflow/scripts 中拆出去，或改成缺目录时显式跳过。

### 5.2 P2：Promptfoo workflow 条件直接引用 secrets 会让步骤条件不可用

证据：

1. `.github/workflows/quality-gate.yml` 第 77-82 行在 step `if:` 中直接引用 `secrets.OPENAI_API_KEY` 和 `secrets.PROMPTFOO_PROVIDER`。
2. GitHub Docs 明确说明 secrets 不能直接在 `if:` 条件中引用；需要先设置为 job 级 env，再通过 `env.*` 判断。

影响：

1. Promptfoo smoke 步骤在 PR workflow 中可能无法按预期跳过或运行。
2. 这会让当前 AI smoke gate 处于“看起来接入，但 GitHub 条件表达式不可靠”的状态。

后续建议：

1. 将 secret 是否存在的判断转为 job 级 env，例如先写 `env: HAS_PROMPTFOO_PROVIDER: ${{ secrets.PROMPTFOO_PROVIDER != '' }}`，再在 step `if:` 中判断 `env.*`。
2. 或者取消 `if:` 中的 secret 判断，在脚本内部根据环境变量决定跳过还是失败。

### 5.3 P2：依赖审计有高危生产漏洞但质量门禁显式放行

证据：

1. `npm audit --omit=dev --json` 报告生产依赖范围仍有 high 18。
2. 受影响的直接生产依赖包括 `@dcloudio/uni-app`、`@dcloudio/uni-app-plus`、`@dcloudio/uni-components`、`@dcloudio/uni-h5`。
3. `.github/workflows/quality-gate.yml` 第 84-87 行对 `npm run audit:deps` 设置了 `continue-on-error: true`。
4. `scripts/quality/run-npm-audit.mjs` 第 84-90 行会打印漏洞数量，但第 150-151 行固定以 0 退出。
5. `依赖与外链安全基线清单.md` 只记录直接依赖用途和替换边界，没有记录当前 audit 漏洞基线、接受原因或升级计划。

影响：

1. CI 名义上有 Dependency audit baseline，但高危生产依赖不会阻断合并。
2. 后续无法区分“已接受的安全基线”与“新引入的安全回退”。

后续建议：

1. 至少把当前 audit 基线写入安全清单，包括漏洞数量、影响包、接受原因和下一步升级边界。
2. 若当前阶段要把 audit 作为真正门禁，应让 high/critical 在生产依赖范围内阻断 CI。

### 5.4 P3：正式留档未进入 format/docs 质量覆盖

证据：

1. `package.json` 第 19-20 行只覆盖根目录 Markdown、`规则体系/**`、`backend-handoff/**`、`references/README.md` 和 `留档/README.md` 等路径。
2. `留档/审查记录/**`、`留档/计划记录/**`、`留档/交付记录/**`、`留档/历史记录/**` 没有进入 `lint:format` 或 `lint:docs`。
3. 本轮新增的审查记录曾在 `npm run lint:format` 通过后，被单文件 `npx prettier --check --ignore-unknown` 检出格式问题，说明现行门禁确实无法覆盖新增正式留档。

影响：

1. 正式审查、计划、交付记录可能格式漂移，但仓级质量命令不会发现。
2. 这与“关键行为可追溯、可反查”的留档目标存在覆盖缺口。

后续建议：

1. 至少对本轮新增或变更的 `留档/**/*.md` 做单文件 Prettier/textlint 检查。
2. 若全量 `留档/**/*.md` 噪声过大，可新增一个只覆盖 active 记录或近期开工记录的脚本。
