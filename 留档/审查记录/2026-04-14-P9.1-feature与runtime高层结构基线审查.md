## 1. 审查范围

本次审查只评估 `P7/P8` 之后仍然保留的高层结构问题：

1. `src/pages/*` 内页私有 runtime 的长期落点。
2. `src/services/*`、`src/models/*` 与 `pages/*` 之间的领域边界。
3. `contracts / ports / implementations / adapters` 这条内容接入链在长期维护视角下的可解释性。
4. 是否需要在后续阶段引入更明显的 feature-first 物理组织。

## 2. 基线事实

### 2.1 当前稳定锚点

当前 `src/` 顶层仍是以下稳定锚点：

1. `pages`
2. `services`
3. `models`
4. `components`
5. `composables`
6. `contracts`
7. `ports`
8. `implementations`
9. `adapters`
10. `mocks`
11. `stores`
12. `utils`
13. `static`

### 2.2 物理治理已完成的部分

1. 首页 `components` 已按 `home / activity / profile / shared` 分组。
2. 详情页 `profile-asset-detail` 已按 `components / runtime / helpers` 分组。
3. 明确跨页复用的首页共享壳已回流到 `src/components/`。
4. `services` 已按 `app / content / home-rail / home-shell / page-entry / profile-asset-detail` 分组。

### 2.3 仍然存在的高层问题

1. `src/pages/*` 内仍承载大量 page-local runtime 与结果窗口时序逻辑，虽然局部已拆，但高层归属仍偏“页内实现平铺”。
2. `src/services/*` 与 `src/pages/*` 之间还存在“领域运行时”与“页面私有装配”边界模糊的问题，尤其是 `home-rail` 与 `pages/home/composables/*`。
3. `contracts / ports / implementations / adapters` 在维护上仍是接入链合理、但目录语义偏架构化；新成员需要先理解抽象层，才能定位改动。
4. `src/components` 与 `src/pages/*/components` 之间虽然已按“跨页共享 vs 页面私有”分层，但还没有更高层的 feature 视图。

## 3. 结论

`P7/P8` 解决的是“目录可解释”和“物理落点明显错误”的问题，还没有处理“高层 feature/runtime 视图是否要更直接”。

因此下一条主线应聚焦：

1. 输出高层 feature/runtime 蓝图。
2. 明确哪些领域适合继续保留在现有锚点下，哪些只应在后续批次做更明显的 feature-first 迁移。
3. 先用分级治理法，避免再次进入大规模无边界搬目录。
