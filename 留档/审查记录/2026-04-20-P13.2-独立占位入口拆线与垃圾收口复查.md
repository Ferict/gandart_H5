## 1. 复查结论

1. `settings`、`service-entry`、`action-entry`、`content-resource`、`profile-address`、`profile-assets`、`notice-detail` 当前都是独立占位页入口，没有真实业务页面价值。
2. 上述页面的真实行为均是进入建设中提示，不应继续保留七套页面壳。
3. `updating` 仍有价值，应作为仓促上线时统一未完成功能承载页保留。
4. `profile-asset-detail` 是真实藏品详情页，保留。
5. `notice` 正式内容域 DTO、mock、http、service 链仍属于正式内容接口范围，不随本批移除。

## 2. 已确认问题

1. 独立占位页会制造“看起来有页面，实际只有建设中”的假入口。
2. `page-entry` 旁路线把路由 query、建设中跳转和 notice 假内容混在一起，已经超过当前价值。
3. `pageContent` 端口和 mock adapter 不是正式后端 API，只服务占位内容。
4. `route-query` 共享模型主要服务已退场占位入口，真实详情页 query 应回到页面附近。
5. notice placeholder 内容链容易被误读成正式公告详情能力，需明确退出。

## 3. 处置分级

1. `keep`：`src/pages/updating/`、`src/pages/profile-asset-detail/`、正式内容域接口链。
2. `trash`：七个独立占位页目录、`src/services/page-entry/`、`src/models/route-query/`、`src/ports/pageContent.port.ts`、`src/adapters/pageContent.mock.adapter.ts`。
3. `defer`：首页内部过度拆分问题，后续单开“首页简化/归并重构”，不在本批处理。

## 4. 风险与遗留项

1. 旧独立页面 URL 不再注册，外部如果手输旧路径会失效；当前仓内导航已统一改到 `updating`。
2. 本批只移除 notice placeholder 内容链，不移除正式内容域公告接口。
3. 本批不处理首页内部过度拆分，不把“代码多”直接等同垃圾。

## 5. 验证要求

1. `pages.json` 不再注册七个独立占位入口。
2. 现行源码不再 import `page-entry`、`route-query`、`pageContent` 或旧页面路径。
3. 未完成功能入口统一生成 `updating` URL。
4. 类型、单测、冗余、质量和构建检查按交付记录执行。
