## 1. 审查目标

记录并固定当前问题：项目目标不是“页面继续读取前端内部假数据”，而是应当先有正式后端 API 契约，再通过一批假后端对接到正式 API 链路，用它展示现有示例数据。

## 2. 审查对象

1. 内容域正式 API 契约与联调口径。
2. 当前页面示例数据的来源方式。
3. `mock / http` provider 与真实联调准备程度的边界。

## 3. 证据来源

1. 用户明确指出：当前应当是“做了后端 API 接口，然后做一批假对接后端接上，用来展示现有示例”，而不是继续做内部假数据。
2. [规则体系/05-接口与联调规则.md](/H:/工作/天异/uniapp+vue新架构/规则体系/05-接口与联调规则.md) 已明确：`mock` 不能证明后端接口真实存在、联调入口完整或字段 contract 已兑现。
3. 当前已知实现仍存在前端内部 mock/provider 示例数据链，不能直接等同于“假后端按正式接口返回数据”。

## 4. 已确认问题

当前偏差是：

1. 已有正式 API 契约与部分 `http` adapter，不等于页面展示已经走了后端 API 链。
2. 当前示例展示如果仍依赖前端内部假数据，就绕过了正式 HTTP 请求、正式响应 envelope、正式字段校验和接口 smoke 验证。
3. 这种实现会造成错误结论：页面能展示示例，不代表“后端 API 已可接入”。

## 5. 正确目标口径

后续应按以下目标修复：

1. 保留 [content-api.contract.ts](/H:/工作/天异/uniapp+vue新架构/src/contracts/content-api.contract.ts) 作为唯一正式 API 真值源。
2. 建立一批假后端或假 HTTP 服务，按正式接口路径、请求参数和响应 envelope 返回现有示例数据。
3. 前端展示示例时，应能切到 `http` provider 并连接该假后端，而不是只调用进程内 mock。
4. 示例数据可以继续是假数据，但它必须通过正式 API 形态返回，才能作为接口联调前的展示和验证链。
5. 后续只有在假后端、`http` adapter、provider 切换方式和验证脚本都打通后，才可对外宣称“假后端 API 链路已接上”。

## 6. 影响范围

1. 后端 readiness 判断。
2. 页面示例展示链路。
3. `content.http.ts`、`content.service.ts`、provider 切换方式和 smoke 验证脚本。
4. `backend-handoff/` 中对后端如何启动、如何验证的说明。
5. 当前关于“已留出真实接口”的历史结论需要后续复核，不应继续按内部 mock 通过来证明。

## 7. 风险与遗留项

1. 如果继续以内置 mock 展示页面，后端联调时可能才暴露 envelope、字段、分页、错误码和接口路径不一致。
2. 如果不补假后端 HTTP 链，`http` adapter 可能长期只存在于代码中，缺少真实验证。
3. 如果文档继续写“已接好 API”，会误导后续开发和交付判断。

## 8. 后续处置建议

建议单开计划处理，最小范围应包含：

1. 复查当前内容域 provider 默认接线和页面数据来源。
2. 设计假后端落点和启动方式，优先复用正式 content API contract。
3. 用现有示例数据填充假后端返回，不在页面层继续新增内部假数据。
4. 让前端能通过配置切到 `http` provider 访问假后端。
5. 增加至少一条 HTTP smoke 验证，证明示例展示链路不是只走内部 mock。
6. 回写 `backend-handoff/`，说明假后端和未来真实后端的替换边界。
