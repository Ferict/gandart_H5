## 3. 详细事实

### 3.1 五个正式接口的单一真值源已经稳定

[src/contracts/content-api.contract.ts](/H:/工作/天异/uniapp+vue新架构/src/contracts/content-api.contract.ts) 当前已经明确导出：

1. `CONTENT_API_ENDPOINTS`
2. 五个正式接口的 request / response DTO
3. 当前正式分页口径、正式场景范围、正式列表资源范围
4. 当前字段归属和不纳入后端 contract 的字段

结论：

1. `P11.2` 和 `P11.4` 已把“正式接口正文”收口到单一契约文件。
2. `P11.5` 不需要再发明新的 API 文档结构。

### 3.2 `content.http.ts` 已对齐正式接口，但验证入口还没完全对齐

[src/implementations/content.http.ts](/H:/工作/天异/uniapp+vue新架构/src/implementations/content.http.ts) 当前已经直接消费 `CONTENT_API_ENDPOINTS`，并覆盖：

1. `getScene`
2. `getResource`
3. `getList`
4. `runAction.notice-read`
5. `runAction.service-reminder-consume`

已有证据：

1. [tests/unit/content/contentApiContractEndpoint.spec.ts](/H:/工作/天异/uniapp+vue新架构/tests/unit/content/contentApiContractEndpoint.spec.ts) 已保护 adapter method/path 与 contract endpoint export 的一致性。
2. [tests/unit/content/content.http.spec.ts](/H:/工作/天异/uniapp+vue新架构/tests/unit/content/content.http.spec.ts) 已保护 `http` base URL 边界。

结论：

1. “契约 + adapter” 这一半已经成立。
2. `P11.5` 的主缺口已经转移到“验证入口”和“可解释的 readiness 结论”。

### 3.3 `check-content-http-link.mjs` 仍保留第二套 endpoint 矩阵

[scripts/check-content-http-link.mjs](/H:/工作/天异/uniapp+vue新架构/scripts/check-content-http-link.mjs) 当前虽然服务于统一联调入口，但仍存在两个结构性问题：

1. 脚本内部自己维护 5 条接口的 `method + pathname + label`。
2. 这些 endpoint 信息不是直接从 [src/contracts/content-api.contract.ts](/H:/工作/天异/uniapp+vue新架构/src/contracts/content-api.contract.ts) 读取。

这意味着：

1. 仓内仍同时存在“contract 真值源”和“脚本隐形 endpoint 矩阵”两套口径。
2. `P11.3` 已解决源码消费链，`P11.5` 仍必须解决验证链消费真值源的问题。

### 3.4 `GET /api/content/list` 的默认 smoke 目前不完全符合正式 market_item 口径

[src/contracts/content-api.contract.ts](/H:/工作/天异/uniapp+vue新架构/src/contracts/content-api.contract.ts) 当前规定：

1. `market_item` 列表请求必须带 `sort.field`
2. `market_item` 列表请求必须带 `sort.direction`

[src/implementations/content.http.ts](/H:/工作/天异/uniapp+vue新架构/src/implementations/content.http.ts) 当前也会在 `market_item` 分支发送：

1. `sortField`
2. `sortDirection`

但 [scripts/check-content-http-link.mjs](/H:/工作/天异/uniapp+vue新架构/scripts/check-content-http-link.mjs) 当前默认验证 `GET /api/content/list` 时，没有补 `market_item` 的正式排序参数。

结论：

1. 当前默认 list smoke 不是严格按正式 contract 在验证。
2. 这是 `P11.5` 必须修的接口留出口径缺口。

### 3.5 仓内已经有统一联调入口，但还缺“本轮正式 readiness 交付结论”

当前 `backend-handoff/` 已完成去歧义，后端知道：

1. 先看 [src/contracts/content-api.contract.ts](/H:/工作/天异/uniapp+vue新架构/src/contracts/content-api.contract.ts)
2. 再看联调环境与交付清单
3. 通过 `npm run check:content:http` 进入仓内最小验证链

但当前还缺：

1. 一份明确以“单一真值源 + adapter + 联调入口 + 验证脚本”为证据的 `P11.5` 交付记录
2. 一份回填到 `P11` 主计划的本轮 readiness 结论

### 3.6 真实后端联通证据仍允许保持“环境阻塞”

按 [规则体系/05-接口与联调规则.md](/H:/工作/天异/uniapp+vue新架构/规则体系/05-接口与联调规则.md)，当前可以证明“已真实留出接口”，但不能伪造“真实后端已联通”。

因此 `P11.5` 的正确结论边界固定为：

1. 可以证明 5 个正式接口都已有：
   - 单一真值源
   - adapter 对齐
   - 统一联调入口
   - 仓内验证入口
2. 如果缺真实后端 URL 或有效样本，最终真实联通结论仍应写为“环境阻塞”。

## 4. 审查结论

`P11.5` 当前的核心缺口只有 3 个：

1. `check-content-http-link.mjs` 仍是第二套 endpoint 真值。
2. 默认 `market_item list` smoke 缺正式排序参数。
3. 仓内缺少一份回到当前单一真值源口径的 readiness 交付结论。

因此 `P11.5` 应收口为：

1. 脚本直接消费当前 contract endpoint 描述，而不是自写 endpoint 矩阵。
2. 脚本默认 list smoke 改为符合正式 `market_item` 查询口径。
3. 补充脚本/验证入口的单测与 handoff 说明。
4. 形成本批正式交付记录，并回填 `P11` 总计划索引。

## 5. 对后续实施的约束

1. `P11.5` 不新建新的长期顶层目录。
2. `P11.5` 不回退到旧 API 文档或冻结入口。
3. `P11.5` 若出现任何清理动作，当前口径固定为“没有删除权限，只能转入 `垃圾桶/` 视为清理完成”。
4. `P11.5` 不伪造真实后端联通成功；没有真实环境证据时，必须明确写成“环境阻塞”。
