## 1. 审查目标

确认当前是否可以直接进入全量接口结构改造，以及该目标中哪些说法必须被收窄，避免把页面、组件、后端 wrapper 和 mock 数据搅成新的混态。

## 3. 审查结论

这件事可以做，但原话如果理解成“一次性把整个前端直接改成调用 190 个后端 wrapper”，就是有问题的。

正确目标应改成：

1. 先盘点全仓当前实际接口消费点。
2. 再把后端契约接进 `provider / implementation / adapter / runtime / service`。
3. 页面和组件继续消费稳定 view-model。
4. mock 数据集中到同一接口链路，不能再和真实接口链路分叉。
5. 按当前前端实际消费优先迁移，不把未确认、未使用、推断层 wrapper 直接暴露给 UI。

## 4. 已确认事实

1. 当前前端内容域已实现契约只有 5 个抽象聚合接口。
2. 当前项目级后端契约冻结基线包含 190 个旧 H5 业务 wrapper。
3. 两者不是同构关系，不能一一替换。
4. `ContentPort`、`content.service.ts` 和现有 adapters 是当前页面稳定消费链的核心边界。
5. `规则体系/后端契约/` 是项目级后端契约入口，但不等同于后端最终签收。
6. `exported-unused`、`unresolved`、`contracts/inferred/*` 只能作为待确认线索，不能直接写进 UI 稳定依赖。
7. 当前目标仍受 P11.10 视觉冻结约束，不允许借接口改造顺手改变视觉表现。

## 5. 主要风险

1. 直接让页面调用后端 wrapper，会让 UI 重新理解 transport 语义。
2. 直接替换 `src/contracts/content-api.contract.ts`，会让已运行的 content port/http/mock/service 链失去稳定真值源。
3. 把 190 个 wrapper 全部实现进当前 UI，会把未使用和待确认接口变成维护负担。
4. 保留散装假数据，会导致后续真实后端切换时无法判断页面到底走的是 mock 还是 http。
5. 如果不先定义 provider 过渡策略，可能出现一半旧 mock、一半新 backend、一半 page-local 的长期混态。

## 6. 必须收窄的口径

1. “整个前端现用接口”只指当前仓真实消费或即将按计划消费的接口，不包括所有后端 wrapper。
2. “彻底改造接口结构”不等于重写视觉层，也不等于页面直连后端 DTO。
3. “后端接驳”必须先经过 adapter 和 runtime/service，不允许跳过稳定 view-model。
4. “假数据接后端接口”必须理解为 mock provider 与真实 provider 共享同一 port/service 形态，不是继续散装 page-local mock。
