# 2026-04-25 多线并行脏链与提交边界审查

> 文档类型：审查记录
> 状态：active
> 更新时间：2026-04-25
> 说明：本记录用于复核当前多线并行工作树是否存在 ownership、提交边界、规则入口或视觉/接口混批问题。

## 1. 审查范围

本次审查对象：

1. 当前 Git 工作树全部 modified / deleted / untracked 路径。
2. 仓根 `通讯/README.md` 消息流。
3. P11.12 规则、口径、计划和交付记录脏链。
4. 二线 UI / updating / 背景 token 脏链。
5. `来源源码/` 未跟踪来源包。

## 2. 已确认情况

已确认：

1. `规则体系/二线通讯/**` 退出当前路径，仓根 `通讯/README.md` 成为唯一通讯入口；旧拆分内容已有垃圾桶冻结落点，本次不读取垃圾桶内容作为现行依据。
2. `P11.12` 口径层已拆成后端口径、前端改造后口径、需要后端补充口径，真实 API 仍应按现行接口与联调规则落到 contract / port / implementation / service / handoff / 验证链。
3. `src/pages/updating/**` 当前新增 `priority-draw` 详情、结果弹层和 `asset-merge` 页面；页面没有直接 import mock，而是经 page-local runtime/service/mock port 承接。
4. `src/components/**`、`src/pages/home/components/shared/**`、`src/uni.scss` 同时存在背景 token、顶栏 blur、字号和共享壳变动。
5. `来源源码/` 当前未跟踪，约 7173 个文件、244.68 MB，且仓根 `来源源码/README.md` 缺失。

## 3. 发现的问题

### 3.1 通讯迁移与规则修正同文件交织

`AGENTS.md`、`规则体系/README.md`、`规则体系/01-项目基线与目录规则.md` 同时承载了通讯入口迁移和 P11.12 / 接口真值源口径修正。

风险：

1. 如果拆开提交，容易出现规则引用新 `通讯/README.md` 但新入口未提交，或旧通讯入口已删除但规则仍回链旧路径。
2. 如果混入源码提交，会把目录治理和二线 UI 变动绑在一起。

处理建议：

1. 作为 docs governance 批次单独提交。
2. 不与 `src/**` 源码、二线视觉或测试改动混批。

### 3.2 二线 UI 批次触碰 shared/global 与 page-local 多类边界

当前二线相关脏链同时包含：

1. `src/uni.scss` 全局背景 token。
2. `src/components/SecondaryPageTopbar.vue`、`HomeShellDrawer.vue`、`HomeShellNavRail.vue`、`SecondaryPageFrame.vue`、`StandalonePageScaffold.vue` 等共享壳。
3. `src/pages/home/components/shared/HomeRailTopbar.vue` 和 `HomeActivityDateFilterSheet.vue`。
4. `src/pages/updating/**` 新页面和 page-local runtime。
5. `src/mocks/updating-db/**` 新增 page-local mock 数据。

风险：

1. 这不是单纯 page-local visual 批。
2. `SecondaryPageTopbar.vue` 和 `HomeRailTopbar.vue` 不只改背景 token，还移除了玻璃 blur 并调整了字号/Logo 高度，属于明确视觉变化。
3. 若未经用户或主线单独批准，不应作为 P11.12 接口主线提交。

处理建议：

1. 二线应单独形成 UI token / shared shell 批次，并明确视觉授权和验证结果。
2. `updating` page-local 新页面应单独形成 page-local structure / visual 批次。
3. 不得把这些文件混进 backend-adaptation 或 P11.12 adapter 主线提交。

### 3.3 来源源码包不能默认入 Git

`来源源码/` 当前是未跟踪大体量来源包，且缺仓根目录说明。

风险：

1. 直接提交会把 244.68 MB 来源包和大量静态资源、二进制包混入当前主仓。
2. 顶层长期目录缺 `README.md`，不满足目录治理规则。
3. 来源包用途、是否只作参考、是否需要压缩包与解包目录同时保留，尚未裁决。

处理建议：

1. 提交前必须先补正式归属裁决。
2. 若保留为长期顶层目录，必须补 `来源源码/README.md` 和规则/留档入口。
3. 不得默认混入二线 UI、P11.12 或规则修正提交。

### 3.4 测试脏链很小但归属未绑定

`tests/unit/home/*` 仅把 `priceUnit` 从 `楼` 改成 `￥`。

风险：

1. 该变动可能是修复错误测试数据，也可能是 P11.12 字段口径的一部分。
2. 若与二线 UI 或规则治理混提，会丢失真实原因。

处理建议：

1. 单独作为 tests-only / P11.12 数据口径测试修正提交。
2. 提交前至少跑对应 home 单测。

## 4. 当前停止边界

本轮不能为了追求工作树清空而直接做以下动作：

1. 不提交 `来源源码/`。
2. 不把二线 UI / shared/global 视觉改动混入 P11.12 接口主线。
3. 不把通讯目录迁移和源码改动放进同一提交。
4. 不执行任何物理删除或 `git clean`。

## 5. 建议提交顺序

建议后续按以下顺序收口：

1. `docs governance`：通讯入口合并、规则 review 修正、P11.12 三层口径和后端契约元信息补齐。
2. `docs p11.12 decision`：P11.12 决策批复清单、工作树边界审查和交付记录。
3. `tests-only`：home 测试中 `priceUnit` 修正。
4. `二线 UI token/shared shell`：背景 token、共享壳和顶栏视觉变动，必须带视觉授权和验证。
5. `updating page-local`：priority-draw / asset-merge 页面、runtime 和 page-local mock seam。
6. `来源源码`：等待用户裁决是否入 Git、是否只保留压缩包、是否需要来源目录 README 和规则入口。
