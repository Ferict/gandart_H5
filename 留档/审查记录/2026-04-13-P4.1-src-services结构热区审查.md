# 2026-04-13 P4.1 src/services 结构热区审查

> 文档类型：审查记录
> 状态：active
> 更新时间：2026-04-13
> 说明：本审查用于判断 `src/services/` 是否已形成可直接分组的自然簇，并为下一批物理目录治理提供低风险切分依据。

## 1. 审查范围

本次只审查：

1. [src/services](/H:/工作/天异/uniapp+vue新架构/src/services/README.md) 当前一级平铺结构。
2. 文件命名簇与职责簇的一致性。
3. 下一批物理分组是否存在低风险切分方案。

本次不做：

1. 实际移动 `src/services/` 文件。
2. 修改 `src/` 顶层分层。
3. 处理 `src/mocks/` 与 `src/implementations/content.mock/` 的关系。

## 2. 当前现状

`src/services/` 当前共有 `28` 个现行 service 文件，另有目录说明与占位文件。

按职责与命名前缀观察，已经形成以下自然簇：

1. `content*`：`6` 个
   - 内容域 provider、缓存、图片缓存、用户域、详情解析。
2. `homeRail* + homeNotice*`：`10` 个
   - 首页三条 rail 的内容解析、结果窗口、持久化、导航、协调器与首页公告已读状态。
3. `homeShell*`：`6` 个
   - 首页壳层 DOM、布局模式、菜单、占位内容与壳状态。
4. `profileAssetDetail*`：`2` 个
   - 详情页内容与持久缓存。
5. `routeQuery* + secondaryPageConstruction* + pagePlaceholder*`：`3` 个
   - 次级页 query 解析、跳转构建、占位内容。
6. `appIdentity*`：`1` 个
   - 应用身份与环境标识。

## 3. 审查结论

`src/services/` 当前的主要问题不是职责完全混乱，而是：

1. 已经存在稳定职责簇，但仍平铺在同一层。
2. 维护者需要先记住命名前缀，才能在一层目录里人工检索。
3. 首页相关 service 数量已经过密，`homeRail*` 与 `homeShell*` 虽然职责不同，但路径层面没有显式隔离。

同时也确认了一个重要事实：

1. 这批 service 的自然簇足够稳定，已经适合做第二批低风险物理分组。
2. 当前不需要直接跳到 `app / features / shared` 级别的大重构。

## 4. 推荐分组蓝图

下一批建议只把 `src/services/` 内部分为以下六组：

1. `content/`
2. `home-rail/`
3. `home-shell/`
4. `profile-asset-detail/`
5. `page-entry/`
6. `app/`

## 5. 风险判断

本批分组的主要风险不是职责归属不清，而是：

1. 源码与单测中的相对导入路径较多。
2. 部分正式留档与规则文档引用了现有 service 路径。

因此下一步应采用：

1. 先确定分组蓝图。
2. 再按簇做整批移动与整批导入回写。
3. 完成后统一跑 `type-check / knip / test:unit`。

## 6. 结论

`src/services/` 已具备进入 `P4.1` 第二批物理结构治理的条件。

建议顺序：

1. 先落分组蓝图。
2. 再执行整批迁移。
3. 最后再评估是否继续处理 `src/mocks/`。
