## 2. 观察结果

### 2.1 overlay 卡片和主网格卡片确实存在模板重复

重复主要体现在：

1. `home-market-card-shell`
2. `home-market-card-visual`
3. badge 区
4. `HomeMarketCardImageReveal`
5. `home-market-card-copy`
6. `name / editionCode / issueCount / price` 展示壳

### 2.2 但重复块的运行时耦合不对称

overlay 卡片依赖：

1. `marketRemovedOverlayItems`
2. `resolveMarketRemovedOverlayItemStyle`
3. `resolveMarketRemovedOverlayRevealPhase`
4. 固定 `lazy-load="false"`

主网格卡片依赖：

1. `mountedMarketItems`
2. `isMarketCardPlaceholder`
3. `resolveMarketCardEntryClass`
4. `resolveMarketCardEntryStyle`
5. `resolveMarketCardPresentationPhase`
6. `isHomeVisualImageRevealReady`
7. `handleHomeVisualImageLoad / error / retrying`
8. `handleCollectionClick`

### 2.3 如果现在硬抽“通用卡片壳”，会明显提高 prop 和 callback 数量

预估至少要传：

1. item
2. imageUrl
3. placeholderIcon
4. phase
5. fallbackText
6. lazyLoad
7. load / error / retrying 三组事件
8. class / style
9. badge tone / label
10. click activate

再叠加 overlay 与主网格的分支差异后，组件会变成“名义上展示组件，实际上仍然携带大量行为接线”的高 prop 壳。

## 3. 当前判断

1. `market` 结果主区已经不再属于低风险模板层。
2. 当前最像样的下一步不是继续拆 `section`，而是单独设计 `market card shell` 的组件边界和事件模型。
3. 这件事应作为新批次处理，不应继续算在当前 `P4.3` 的低风险模板层收口里。

## 4. 结论

1. `P4.3` 到此可以视为阶段收口。
2. 下一步若继续处理 `HomeRailHomePanel.vue`，应先开一个新的高风险批次，专门审查：
   - `market card shell` 是否值得抽象
   - overlay 与主网格是否应共用同一展示组件
   - 结果窗口运行时是否需要先进一步收口
3. 在未开新批次前，不建议继续顺手拆 `market results` 主区。
