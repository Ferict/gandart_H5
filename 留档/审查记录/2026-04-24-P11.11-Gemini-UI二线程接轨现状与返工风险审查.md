# 2026-04-24 P11.11 Gemini UI 二线程接轨现状与返工风险审查

> 文档类型：审查记录
> 状态：active
> 更新时间：2026-04-24
> 说明：本文件用于审查 Gemini UI 二线程当前接轨现状、潜在返工点和与主线冲突点，为二线程执行口径与 git 分类提供依据。

## 1. 审查目标

确认以下问题：

1. Gemini 二线程现在最容易在哪些地方返工
2. Gemini 二线程最容易误碰哪些不属于它的边界
3. Gemini 二线程需要什么固定工作形式，才能避免把视觉稿直接打穿到正式契约链

## 2. 审查对象

1. `P11.10` 视觉冻结主线
2. `P11.11` 天异设计语言与外来 UI 转化主线
3. 当前正式内容契约链
4. 当前 `home / profile-asset-detail / updating` 的 page-local runtime 与内容接入结构

## 3. 证据来源

1. `规则体系/口径/04-AI-UI-refresh专项治理现行口径.md`
2. `留档/计划记录/2026-04-24-P11.11-天异设计语言提炼与外来UI转化治理计划.md`
3. `src/contracts/content-api.contract.ts`
4. `src/services/content/content.service.ts`
5. `src/adapters/content/*`
6. `src/pages/updating/runtime/*`

## 4. 已确认事实

1. 现有正式内容域已经有统一 `contract -> port -> implementation -> service` 主链。
2. `home / activity / profile / profile-asset-detail` 已经补进了 adapter 归一化层。
3. `updating` 抽签页已经从 page-local 直写假数据，收成 page-local runtime seam。
4. Gemini UI 二线程真正应该处理的是“外来 UI 翻译”和“page-local 视觉转化”，不是“正式契约改造”。
5. 如果不写清 git 分类，Gemini 二线程极容易把 docs、page-local visual、shared/global visual 和 backend adaptation 混进同一批。
6. 如果不写清字段边界，Gemini 二线程极容易在模板层硬写新字段需求，从而导致后续接新版后端契约时返工。

## 5. 已确认风险

1. `高风险`
   - Gemini 二线程把新的 UI 需求直接写成 DTO 或正式接口字段，导致后续正式契约接轨返工。
2. `高风险`
   - Gemini 二线程把 page-local 改造越界写进 shared/global 面，导致与主线 ownership 冲突。
3. `中风险`
   - Gemini 二线程把 docs-only、brief、视觉证据和真实页面代码混成一个提交，造成 git 边界不清。
4. `中风险`
   - Gemini 二线程直接照搬外来导航壳、卡片语言、英文排版和 AI 风格装饰，导致天异设计语言失真。
5. `中风险`
   - Gemini 二线程发现字段不够时，在页面层硬补假字段，导致 mock 和未来正式后端契约双重返工。

## 6. 审查结论

需要为 Gemini 二线程单独补一份执行口径，至少固定：

1. 它的角色不是后端契约主线
2. 它的默认 ownership 是 page-local
3. 它必须先写 brief、证据和 git 分类，再改代码
4. 它命中字段缺口时，只能停在 view-model / adapter / runtime 边界
5. 它必须按独立 git 分类提交，不得混批

状态：`已转计划`
