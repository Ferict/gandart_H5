## 1. 修复目标

1. 让 GitHub 联调源码仓白名单与实际 CI、脚本依赖目录一致。
2. 修正 GitHub Actions 中 Promptfoo step 对 `secrets` 的条件判断方式。
3. 保证按白名单交付后，仍能保留 CI 所需的质量规则目录或明确拆除对应 CI 检查。

## 2. 当前确认问题

### 2.1 交付白名单与 CI 依赖冲突

当前白名单保留：

1. `.github/**`
2. `scripts/**`
3. `src/**`
4. `tests/**`

当前白名单排除：

1. `promptfoo/**`
2. `semgrep/**`
3. `vale-styles/**`
4. `.continue/**`
5. `.tools/**`

但当前 CI 和脚本仍依赖：

1. `promptfoo/promptfooconfig.yaml`
2. `promptfoo/prompts/rule-smoke.prompt.md`
3. `semgrep/rules/**`
4. `vale-styles/**`
5. `.continue/checks/**`

### 2.2 workflow 直接在 `if:` 中引用 secrets

当前 Promptfoo step 的条件判断直接使用：

1. `secrets.OPENAI_API_KEY`
2. `secrets.PROMPTFOO_PROVIDER`

GitHub Actions 官方文档不支持在 `if:` 条件中直接引用 secrets，应先映射到 `env` 后再判断。

## 3. 修复方案

### 3.2 Promptfoo workflow 修复

推荐方案：

1. 在 job 级或 step 级 `env` 中映射 secret 值。
2. 在 `if:` 中只判断 `env.PROMPTFOO_PROVIDER != ''` 和 `env.OPENAI_API_KEY != ''`。
3. 保持脚本内现有环境变量校验，防止本地误跑。

备选方案：

1. 移除 step `if:` 中的密钥判断。
2. 改由 `scripts/quality/run-promptfoo-smoke.mjs` 在缺少 provider 或 key 时输出 skip 并以 0 退出。

默认采用推荐方案；除非验证 GitHub Actions 表达式不支持目标写法，才转备选方案。
