## 4. 自动执行顺序

### P11.6.1 先做现有代码 API 实源覆盖

目标：

1. 全量复核当前应接后端的现行代码链路是否都通过单一 content contract 和统一 `ContentPort` 取数。
2. 把剩余 direct mock bypass 收回到可解释边界。

固定动作：

1. 审查 `contentResourceDetail.service.ts` 和 `profileAssetDetailContent.service.ts` 中对 `createContentResourceSnapshot` 的直接依赖。
2. 将 detail shell fallback 改成以下二选一之一：
   - 仅保留不依赖 mock 数据本体的本地最小壳
   - 或回收到统一 content 接入链的正式边界
3. 不允许继续新增直接导入 `content.mock` 快照函数的上层 service。
4. 如改动了验证入口、handoff 或测试，必须同步回写。

验收标准：

1. 当前“应接后端”的现行链路不再从上层 service 直接读取 mock 实现体。
2. `mock` 只作为统一 provider 的一个实现体存在，而不是页面/业务 service 的第二入口。

提交边界：

1. 作为独立代码批次提交。
2. 本批不允许删除文件；若后续出现清理对象，也只能在 `P11.6.2` 中转入 `垃圾桶/`。

### P11.6.2 再做当前路径送桶清理

目标：

1. 把当前路径里已冻结、已历史化或已确定为生成物的对象清出当前路径。
2. 让现行顶层目录语义更接近“当前真身”。

固定动作：

1. 只处理已在本审查中命中的冻结/历史/生成物对象。
2. 不得直接删除任何文件或目录；所有清理动作只能通过“转入 `垃圾桶/`”完成。
3. 必须为送桶目录或文件补齐：
   - 原路径
   - 新垃圾桶路径
   - 送桶原因
4. 不把 `提交清单/`、`新藏品/`、`src/pages/*` 兼容入口混入本批。

验收标准：

1. 当前路径不再并排保留本批命中的冻结壳和生成物目录。
2. 送桶记录完整，且没有任何删除动作。

提交边界：

1. 作为独立清理批次提交。

### P11.6.3 为 P11.7 预留干净入口

目标：

1. 在 `P11.6` 结束时，把 `P11.7` 需要的前置条件准备好。

固定动作：

1. 回填 `P11` 主计划和总计划，注明：
   - API 实源覆盖是否完成
   - 送桶清理是否完成
   - 下一步进入 `P11.7`
2. 不在 `P11.6` 中顺手做 `P11.7` 的最终复核动作。
3. 所有清理动作继续遵守“没有删除权限，只能转入 `垃圾桶/`”。

只有同时满足以下条件，才允许进入 `P11.7`：

1. 当前应接后端的现行代码链路已回到单一真值源与统一 provider 接口。
2. `P11.6` 命中的冻结壳/生成物已完成送桶。
3. 当前工作区处于可解释、可提交的干净状态。
4. 已重新复读规则，并按 [2026-04-15-P11.7-收口复核与自动执行边界整理详细计划.md](/H:/工作/天异/uniapp+vue新架构/留档/计划记录/2026-04-15-P11.7-收口复核与自动执行边界整理详细计划.md) 开工。
