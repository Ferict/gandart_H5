## 1. 适用范围

本补充约束覆盖以下范围：

1. 当前正在执行的 `P11.10.5` 代码批剩余收口、复查、回填与重开场景。
2. 后续 `P11.10.6A`、`P11.10.6B`、`P11.10.7`、`P11.10.8` 及其子批次。
3. 所有以“零视觉差结构迁移”为名义进行的 adapter、runtime、service、view-model、tests 改造批。

本补充计划不替代总规划，也不替代 `P11.10.5` 正文；它只负责把两个新增硬约束写死，避免后续再次丢颗粒度。

## 2. 子代理默认启用约束

从本补充计划生效起，代码批默认遵守以下规则：

1. 主控在任何代码批开始前，都必须先做一次“是否可拆子代理”的显式判断。
2. 只要当前批次存在 2 个及以上互不重叠的写入集合，就默认应拆成子代理 ownership，而不是继续整批串行。
3. 可拆分的典型集合包括：
   - `src/adapters/**`
   - page-local `runtime / composables / helpers`
   - 与单一业务面绑定的 `tests/**`
   - 不共享写入面的 `services/**`
4. 以下文件或目录仍默认由主控独占，除非单轮明确授权给唯一一个 worker：
   - `src/pages/home/index.vue`
   - `src/pages/home/components/shared/*`
   - `src/components/*`
   - `src/main.ts`
   - `规则体系/*`
   - `规则体系/口径/*`
   - `留档/*`
5. 如果某一轮理论上可拆子代理，但实际没有启用子代理，必须在对应交付记录写明原因；“忘记开子代理”不再属于可接受原因。
6. 若执行中出现写入集合重叠、共享文件锁冲突、提交边界失控，则允许从并行回收为主控串行，但必须在交付记录写明回收原因。

## 3. 对 P11.10.5 的即时修正口径

针对当前 `P11.10.5`，补充要求固定如下：

1. `P11.10.5` 已经发生的主控本地实施不作回滚理由，但后续若重开子批、补测试、补 detail/home 试跑前置收口，必须优先按 ownership 拆子代理。
2. 当前批的交付记录需要明确写出：
   - 本批哪些部分已由子代理承担
   - 哪些部分仍由主控独占
   - 若某段代码未开子代理，原因是什么
3. 从 `P11.10.6A` 开始，默认必须先做子代理分工表，再进入代码。

## 4. 新版后端接口契约兼容约束

后续将存在一份新的后端接口契约，用于改造当前前端并接入已有后端。为避免本期治理与后续接轨冲突，当前批次必须遵守以下规则：

1. 本期新增的 adapter、normalized model、runtime/service 边界，默认都要按“可替换 contract mapper”设计。
2. 本期允许依赖当前 formal contract 作为输入真值源，但不允许把 current contract 的字段组织方式重新固化为 UI 真值。
3. `models / pages / templates / shared components` 只能消费前端稳定模型，不应直接理解“当前 contract 的字段名长相”。
4. 后续若切换到新版后端接口契约，优先替换的位置固定为：
   - adapter 输入侧
   - contract -> normalized model mapper
   - 必要的 runtime/service 输入边界
5. 后续若为了接已有后端必须改动 UI-facing model，必须单开正式计划；本补充计划默认认为“新版契约接轨”不应再次打穿 UI 边界。
6. 当前批若发现某个内部模型已经被当前 formal contract 细节绑死，应在本期优先收口为前端稳定模型，而不是把问题留给后续新版契约接轨批。

## 5. 停止条件

命中以下任一情况，当前批必须停下，不得继续顺手推进：

1. 为兼容未来新版后端契约，需要直接修改正式 contract 真值源。
2. 为启用子代理而必须让多个 worker 同时写同一共享文件。
3. 子代理拆分后导致提交边界不再清楚。
4. 新版契约兼容需求开始反向推动当前批修改 UI 可见输出。

## 6. 本补充计划完成标志

满足以下条件，视为本补充计划已被正确吸收到主线执行中：

1. 当前批与后续代码批都已显式做“是否拆子代理”的判断。
2. 对未启用子代理的代码面，交付记录中有明确原因。
3. 当前新增 adapter / model / runtime 设计不会明显阻塞后续“新版后端接口契约接轨批”。
4. 后续继续执行时，不再把“子代理默认启用”和“后续新版契约兼容”遗漏为口头约束。
