## 1. 总目标

这条主线的终点不是“重写 UI”，而是：

1. 让当前现有 UI 继续消费稳定 view-model
2. 把旧正式契约输入侧替换成新版后端契约
3. 把差异吸收到 adapter / runtime / service 层
4. 避免让页面模板、组件 props 和视觉结构去理解新版 DTO
5. 让 mock、http、provider 切换有清晰过渡，不产生长期混态

当前项目级后端契约入口固定为：

1. [规则体系/后端契约/README.md](/H:/工作/天异/uniapp+vue新架构/规则体系/后端契约/README.md)

当前执行口径固定为：

1. `P11.12` 以该目录作为新版后端契约签收、冻结和差异矩阵入口。
2. `exported-unused`、`unresolved`、`inferred` 一律视为待确认层，不得直接跳过签收写进稳定 UI 消费链。

4. 没有正式签收的新字段臆测接入

## 4. 总体原则

### 4.1 边界原则

1. 新版后端契约只替换输入侧，不直接替换 UI 层。
2. 页面继续只吃稳定 view-model。
3. DTO 差异、字段重命名、payload 结构变化统一吸收在 adapter / runtime-service 层。
4. 只要视图还能不变，就不动模板和视觉层。

### 4.2 假数据原则

1. 假数据统一放在集中 mock 域或 page-local runtime seam。
2. 页面不再长留本地散装假数据。
3. mock 与 http 必须尽量共享同一 port 或同一 service seam。
4. 新版契约接入时，先替换 mock/http 输入侧，再决定是否回收旧 mock 数据。

### 4.3 交接原则

1. `P11.11` 发现字段缺口，不直接改契约，交给 `P11.12`。
2. `P11.12` 决定字段映射、adapter 落点、service 切换和 provider 策略。
3. UI 模板层不承担契约解释职责。

## 5. 分阶段总链路

### P11.12.1 新版后端契约签收与冻结批

目标：

1. 接收新版后端契约原件
2. 明确它的正式落点
3. 冻结“本轮以哪一份新契约为准”

必须完成：

1. 标出正式签收时间
2. 标出来源人或来源文件
3. 标出是否完全替代旧 `content-api.contract.ts`，还是只替代其中一部分资源
4. 标出未签收字段和未签收接口
5. 以 `规则体系/后端契约/README.md` 作为正式签收入口
6. 把 `confirmed / exported-unused / unresolved / inferred` 四类状态统一冻结到差异矩阵与执行口径中

停止条件：

1. 新版契约本身仍在剧烈变化
2. 还无法确认正式真值源落点

### P11.12.2 旧契约 vs 新契约差异矩阵批

目标：

把“旧 contract / 旧 mock / 旧 UI 消费”与“新版后端契约”做成一张完整 delta 矩阵。

矩阵至少要覆盖：

1. endpoint 是否仍存在
2. path / method 是否变化
3. request query / body 是否变化
4. envelope 是否变化
5. resourceType / sceneId / actionType 是否变化
6. DTO 字段增删改
7. UI 已消费字段还能否直接保留
8. 哪些差异只需 adapter 吸收
9. 哪些差异会打到 runtime/service
10. 哪些差异会打到页面行为

完成标准：

1. 后续迁移不再靠执行人边写边猜
2. 每一条差异都有责任归属层

### P11.12.3 目标分层与迁移位设计批

目标：

在开工前先把“新版契约接进来以后，哪一层承担什么”写死。

固定迁移责任：

1. `contract layer`
   - 只承接正式契约真值源
2. `port layer`
   - 只承接统一调用能力
3. `implementation layer`
   - 处理 mock / http 两种来源
4. `adapter layer`
   - 吸收 DTO 差异、名称差异、payload 差异
5. `runtime/service layer`
   - 组装 UI 需要的稳定 view-model
6. `page assembly layer`
   - 只接 view-model，不接 DTO

完成标准：

1. 每个差异都能落到唯一责任层
2. 后续执行批不再反复争论“这里该在 adapter 还是 service 处理”

### P11.12.4 provider 与 mock 过渡策略批

目标：

把旧 mock / 新 mock / 新 http 的过渡路径写清楚。

必须明确：

1. 哪些面继续允许旧 mock
2. 哪些面先切新 mock
3. 哪些面允许直接切新 http
4. provider 切换是全局一次切，还是按批次逐步切
5. 回滚时回到哪一档

完成标准：

1. 不出现“一半旧 mock、一半新 http、页面还看不出来源”的长期混态

### P11.12.5 正式内容域分批迁移批

按 UI 面固定拆成 5 条迁移线：

1. `L1 home/home`
   - 重点：scene home、market list、market resource、notice bar 关联
2. `L2 home/activity`
   - 重点：scene activity、notice list、notice read action、notice resource
3. `L3 home/profile`
   - 重点：scene profile、profile_asset list、profile_asset resource
4. `L4 profile-asset-detail`
   - 重点：detail resource 和详情卡片 view-model
5. `L5 updating`
   - 重点：新版后端契约是否纳入抽签页；若纳入则决定是并入正式内容域还是保留 page-local seam

规则：

1. 每条迁移线单独开批
2. 每条迁移线单独验证
3. 每条迁移线单独提交
4. 不允许多条迁移线长期混在一个工作树里

### P11.12.6 页面级切换与回归批

目标：

在 adapter / runtime / service 已完成迁移后，把页面实际消费切到新版链路。

规则：

1. 先切 runtime/service
2. 再切页面装配
3. 模板和视觉默认不动
4. 若因契约差异必须改模板，只能在单独批次里说明

### P11.12.7 旧链清理与冻结批

目标：

在新版契约稳定后，冻结旧链，避免双真值长期存在。

必须明确：

1. 哪些旧 adapter/mappers 废弃
2. 哪些旧 mock 数据冻结
3. 哪些旧 contract 注释或说明应回写为历史状态
4. 哪些 page-local seam 保留，哪些并入正式主链

### P11.12.8 主线收口批

目标：

完成全部状态回填、索引回填、git 索引回填和最终验证。

## 8. Git 分类

`P11.12` 主线的 git 分类固定为：

1. `docs-only`
   - 总规划、差异矩阵、现状审查、批次计划、交付
   - 提交前缀：`docs(p11.12.x): ...`
2. `contract-freeze`
   - 正式真值源签收与注释回写
   - 提交前缀：`docs(...)` 或 `refactor(...)`
3. `adapter-migration`
   - adapter / mapper / normalized model 改造
   - 提交前缀：`refactor(p11.12.x): ...`
4. `runtime-service-cutover`
   - service / runtime 输入侧切换
   - 提交前缀：`refactor(p11.12.x): ...`
5. `ui-consumer-cutover`
   - 页面装配改线，但视觉保持不变
   - 提交前缀：`feat(p11.12.x): ...`
6. `provider-switch`
   - mock/http/provider 切换
   - 提交前缀：`chore(p11.12.x): ...`
7. `tests-only`
   - 契约 smoke、adapter tests、页面接线 smoke
   - 提交前缀：`test(p11.12.x): ...`
8. `cleanup-freeze`
   - 旧链冻结、说明回写、遗留桥接清理
   - 提交前缀：`chore(...)` 或 `refactor(...)`

固定禁止项：

1. `adapter-migration` 混视觉批
2. `provider-switch` 混 shared/global UI 改造
3. `tests-only` 混业务代码
4. 多条迁移线长期堆在一个工作树里

## 10. 停止条件

命中以下任一情况必须停：

1. 新版后端契约未签收
2. 新版契约与旧 UI 消费模型差异过大，已经超出当前批 write-set
3. 需要直接改页面视觉才能继续
4. 需要直接改共享壳或全局组件才能继续
5. 需要同时改多条迁移线且 git 边界失控
6. 需要改正式契约真值，但当前批并未纳入 contract-freeze
