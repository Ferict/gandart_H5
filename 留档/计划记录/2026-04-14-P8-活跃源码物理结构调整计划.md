## 1. 目标

`P8` 只处理活跃源码的物理结构调整，不继续扩写规则，不处理旧历史档案，也不承接职责头补齐主线。

本阶段目标固定为：

1. 让 `src/` 内部结构更接近人工维护视角，而不是继续沿技术目录平铺堆积。
2. 在不引入新的 `src/` 顶层目录前提下，优先整理页面私有组件、页面私有 runtime 和页面私有结果区壳层。
3. 保持 `pages / services / models / mocks / static` 作为稳定锚点，只在锚点内部做物理分组和落点校正。
4. 为后续更高层的 feature-first 重排预留空间，但本批不直接引入 `src/features/`。

## 2. 固定边界

本批只允许：

1. 在现有稳定锚点内部建立子目录分组。
2. 调整页面私有组件、页面私有 runtime 和 page-local helper 的物理落点。
3. 回写目录 `README.md`、留档索引和受影响 import。

本批明确不做：

1. 新建新的 `src/` 顶层技术目录。
2. 直接引入 `src/features/`。
3. 改动后端接口契约、provider 选择、缓存语义或刷新协议。
4. 重做 `services / contracts / ports / implementations / adapters` 顶层边界。
5. 把 `新藏品/`、`src/static/`、`src/mocks/` 的素材策略强行并到同一落点。

## 3. 默认决策

以下默认决策在本批内直接生效：

1. `src/pages/home/components/` 按 `activity / home / profile / shared` 做物理分组。
2. `src/pages/home/index.vue` 继续保留为首页路由壳，不进入更深层目录。
3. 仍只被首页消费的共享壳层继续保留在 `src/pages/home/components/shared/`。
4. 已被二级页或跨页面组件复用的交互叶子组件，应回流到 `src/components/`。
5. `src/pages/profile-asset-detail/` 只做页内物理整理，不拆成新的跨页共享层。
6. 目录移动必须同时回写 `README` 和留档，不能只改代码落点。

## 4. 执行顺序

1. `P8.1` 活跃源码物理结构调整基线审查
2. `P8.2` `src/pages/home/components` 首轮物理分组
3. `P8.3` 首页页面私有结果区与视觉组件物理归拢
4. `P8.4` `src/pages/profile-asset-detail` 物理整理
5. `P8.5` 共享层与页面私有层回流校正
6. `P8.6` 收口复核与提交边界整理
