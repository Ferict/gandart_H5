## 6. 第 3 批：provider 与 implementation 方案

目标：建立 `backend-http`，不破坏 `mock` 和 `current-http`。

固定三档 provider：

1. `mock`
   - 继续服务前端开发和无后端环境运行。
2. `current-http`
   - 继续服务当前 5 个 `/api/content/*` 抽象接口联调；现行环境值仍是 `VITE_CONTENT_PROVIDER=http`。
3. `backend-http`
   - 新增，用于按 190 个后端 wrapper 接轨。

实现策略：

1. 保留 `ContentPort`。
2. 新增 backend implementation 适配旧 H5 wrapper。
3. backend implementation 内部可以调用多个 wrapper 组合成一个 `ContentSceneDto`、`ContentListDto` 或 `ContentResourceDto`。
4. wrapper 的原始 DTO 先进入 backend normalized model，不直接进入页面。
5. `backend-http` 初期只覆盖当前已搭页面需要的 wrapper，不一次接满 190 个。

完成标准：

1. provider seam 清楚。
2. 当前页面无需知道后端 wrapper 名。
3. 未覆盖 wrapper 有清单，不伪装成已完成。

执行状态：

1. 已实现 `src/implementations/content.backend-http.ts`。
2. 已在 `src/main.ts` 接入 `VITE_CONTENT_PROVIDER=backend-http`，优先读取 `VITE_CONTENT_BACKEND_API_BASE_URL`，未配置时回退 `VITE_CONTENT_API_BASE_URL`。
3. 已覆盖当前主页面首批运行入口：`home/activity/profile` scene、`market_item/notice/profile_asset` list、`notice/profile_asset` resource、`notice-read` action。

## 12. 固定停止条件

命中以下情况停止，不继续自动推进：

1. 外部最新版契约与项目内契约未重同步。
2. 某字段只能从 `exported-unused`、`unresolved` 或 `inferred` 推断出来。
3. 需要决定 UI 是否展示新内容。
4. 需要改视觉才能继续。
5. 需要改当前 5 个 content public API 形态。
6. 需要新增长期目录。
7. 二线同时写入主线 ownership 文件。
8. Git 边界无法区分 docs-only、backend-adaptation、page-local visual 或 tests-only。
