## 0. 先行裁决

这件事有一个必须反驳的点：

1. 如果目标是“一次性把页面全部改成直接调用后端 wrapper”，那是错误做法。
2. 如果目标是“把当前前端实际消费的接口全部纳入统一 provider、adapter、runtime、service 接驳链，再分批切到后端”，这是正确目标。

本计划按第 2 种理解执行。

## 1. 总目标

最终目标固定为：

1. 全仓当前实际接口消费点都有清单。
2. 当前散装假数据统一进入可替换的 mock provider 或 page-local seam，不再让页面直接持有后端形态假数据。
3. 后端契约冻结基线通过 implementation 和 adapter 接入，不直接下放给页面和组件。
4. 当前 UI 继续消费稳定 view-model。
5. 可以在 `mock / current-http / backend-http` 之间按明确策略切换。
6. 后续真实后端接入时，主要替换 provider、base URL、鉴权和少量 adapter 映射，不再重写 UI。

## 2. 最高约束

1. 不改视觉效果。
2. 不改交互语义。
3. 不让页面模板直接 import 后端 contract。
4. 不让组件 props 携带 HTTP 状态码、envelope、requestId、etag、provider、分页协议或重试预算。
5. 不把 190 个后端 wrapper 当成 190 个 UI 入口。
6. 不把 `exported-unused`、`unresolved`、`inferred` 当稳定后端承诺。
7. 不执行任何删除操作；如需退场，另开计划转移到 `垃圾桶/`。

## 3. 目标架构

固定链路：

```text
后端契约冻结基线
  -> backend implementation
  -> backend DTO adapter
  -> normalized content model
  -> runtime / service model
  -> UI view model
  -> component props
```

当前前端抽象链路继续保留：

```text
ContentPort
  -> content.service.ts
  -> home / activity / profile / detail runtime
  -> 页面 view-model
```

职责分配：

1. `规则体系/后端契约/` 只作为项目级后端契约入口和差异确认层。
2. `src/contracts/content-api.contract.ts` 暂时继续作为当前前端内容域实现契约。
3. `src/ports/content.port.ts` 暂时继续作为 UI 稳定调用边界。
4. `src/implementations/` 承担 mock、当前 content http、新 backend http 的来源选择。
5. `src/adapters/content/` 承担 DTO 到前端稳定模型的转译。
6. `src/services/content/` 承担统一内容服务 seam。
7. 页面 runtime 只承担页面装配、缓存、分页、刷新和 view-model 派生。

## 4. 当前已完成基线

1. `P11.12.1` 已冻结项目级后端契约内部适配基线。
2. `P11.12.2` 已确认 5 个当前 content 聚合接口与 190 个后端 wrapper 不是同构关系。
3. `P11.12.3A0` 已迁入外部专项最新版正式契约，并补入接口用途、字段用途和旧前端消费证据。
4. 当前下一步不是直接写代码，而是先做全仓现用接口盘点和目标分层落位。

## 5. 不做什么

本主线不做：

1. 真实视觉刷新。
2. 页面样式优化。
3. 全局组件重设计。
4. 把当前 UI 没有消费证据的成交量、交易量、全平台持有人数写成展示需求。
5. 未确认后端字段的 UI 展示扩展。
6. 把二线外来 UI 产物当接口真值源。
7. 把旧 H5 后端 wrapper 全量暴露给当前页面。
8. 把抽象 content API 直接作废。

### P11.12.3B 目标分层与迁移位设计批

目标：

1. 明确后端 wrapper 接到哪一层。
2. 明确当前 content port 是否继续保留。
3. 明确 page-local seam 哪些保留，哪些并入正式 content 链。

固定裁决：

4. provider seam 由主控维护，不交给二线。

输出：

2. 每条迁移线的写入集合。
3. 每条迁移线的锁定集合。

完成标准：

1. 后续代码批不再争论“这里该放 adapter 还是 service”。
2. 每个差异都有唯一责任层。

### P11.12.4 Provider 与集中 mock 过渡策略批

目标：

1. 把现有假数据统一接入同一条接口链。
2. 明确 `mock / current-http / backend-http` 三档如何切换。
3. 避免页面继续散装持有假数据。

固定 provider 档位：

1. `mock`
   - 使用集中 mock 数据，服务本地开发和页面回归。
2. `current-http`
   - 使用当前 `/api/content/*` 抽象接口，服务现行 content contract 联调。
3. `backend-http`
   - 使用项目级后端契约 wrapper，服务后续真实后端接驳。

固定 mock 规则：

1. mock 数据可以继续存在，但必须经过同一 port/service shape。
2. mock 不能绕过 adapter 给 UI 喂 DTO。
3. page-local mock 只能在未并入正式 content 链前临时保留，并必须登记。

输出：

1. provider 切换策略。
2. mock 数据归属表。
3. page-local seam 保留/并入裁决表。

完成标准：

1. 后续切真实后端时，不需要重写页面。
2. 能清楚说明某个页面当前走的是哪一档 provider。

### P11.12.5 Backend Implementation 基础设施批

目标：

1. 新增或改造真实后端请求实现层。
2. 把后端 envelope、错误、分页、鉴权占位和 base URL 处理集中。
3. 给 adapter 提供稳定输入。

计划写入面：

1. `src/implementations/content.backend.ts`
2. 必要时新增短命名子目录：`src/implementations/content.backend/`
3. `src/implementations/README.md`
4. `src/ports/content.port.ts`
5. `tests/unit/content/`

固定实现原则：

1. 不把后端 wrapper 方法导出给页面。
2. 不把后端返回原样传给 UI。
3. 所有业务错误先归一化为内部错误结果，再由 service 决定页面状态。
4. 缺真实样例的响应字段必须标为待确认，不许默认成功。

完成标准：

1. backend implementation 可被测试注入。
2. mock/current-http/backend-http 三档不会互相污染。
