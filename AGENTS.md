4. 不得随意新增新的长期顶层目录。
5. 在物理目录治理开始前，先以规则定义为准，不提前搬目录。

在 `src/` 真正重构前，先执行以下过渡规则：

1. 不新增新的 `src/` 顶层技术目录。
2. 页面专属逻辑优先放在对应页面或未来业务域附近。

## 13. 自动化与 `codex exec` 非交互执行边界

当用户要求“自动化”“自动化修复”“自动模式”“直接做完”“继续做到规划尽头”，或进入 `codex exec`、云端任务等非交互执行模式时，统一视为同一种执行口径。

定义：

1. 自动化等同于 `codex exec` 非交互执行模式。
2. 自动化不是“只做计划后停下”，而是先确认或补齐计划，再直接执行到已规划的尽头。
3. “已规划的尽头”指当前计划写明的验收标准、完成条件、固定停止点或失败退出条件，不包括顺手扩展新阶段、新目录、新业务目标。

详细边界和停止条件以 [规则体系/10-自动化执行边界专题细则.md](/H:/工作/天异/uniapp+vue新架构/规则体系/10-自动化执行边界专题细则.md) 为准。

额外执行以下硬规则：

1. 开工前必须已有正式计划，或在现有计划中补明本轮的执行目标、停止边界、失败退出条件和完成后汇报要求。
2. 自动化只允许处理当前计划明确纳入的范围，不得在执行中顺手扩题。
3. 自动化执行中不得在每个小步骤完成后机械停下询问；除非触发停止边界，应连续完成审查、实现、验证、留档回写和 Git 提交。
4. 任一批次满足以下任一条件时必须停止，而不是继续自动扩展：
   - 已达到本批验收标准
   - 触碰到新的顶层阶段
   - 需要新的目录许可
   - 发现与现行规则冲突
   - 发现提交边界已经不清楚
5. 自动化执行完成后，必须回写：
   - 执行结果
   - 实际停止位置
   - 下一步建议
   - 未解决阻塞项
6. 若本批是可独立验证的 docs-only 或代码批次，完成对应检查后应尽快形成 Git 提交；若暂不提交，必须在留档中写清原因。
